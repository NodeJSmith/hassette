## Objectives & Success Criteria

- `LogEntry` has a `seq: int` field that increments monotonically per `LogCaptureHandler` instance
- `seq` appears in both REST (`/logs/recent`) and WS (`log` message) payloads
- `frontend/ws-schema.json` includes `seq` in `LogEntryResponse`
- Frontend `LogEntry` type includes `seq: number`
- Unit test confirms `seq` increments monotonically across multiple `emit()` calls

## Subtasks

1. In `src/hassette/logging_.py`: add `self._seq: int = 0` to `LogCaptureHandler.__init__`
2. In `src/hassette/logging_.py`: in `emit()`, increment `self._seq += 1` and pass to `LogEntry` constructor
3. In `src/hassette/logging_.py`: add `seq: int` field to the `LogEntry` dataclass (after `timestamp`)
4. Update `LogEntry.to_dict()` to include `seq`
5. Update `frontend/ws-schema.json`: add `seq` as required integer field to `LogEntryResponse`
6. Update `frontend/src/api/endpoints.ts`: add `seq: number` to `LogEntry` type
7. Update `frontend/src/api/ws-types.ts`: add `seq` to WS log payload type if separately defined
8. Regenerate `frontend/openapi.json` if the REST `/logs/recent` endpoint schema changed
9. Write unit test: `tests/unit/test_logging.py::test_log_capture_handler_seq_increments` — create handler, emit 5 records, assert seq values are 1,2,3,4,5
10. Write unit test: `tests/unit/test_logging.py::test_log_entry_to_dict_includes_seq` — create LogEntry with seq=42, assert `to_dict()` contains `"seq": 42`
11. Run `uv run nox -s dev`

## Review Guidance

- Verify `seq` starts at 1 (not 0) — seq=0 could be confused with "no seq"
- Verify `seq` is per-handler-instance, not global — each `LogCaptureHandler` has its own counter
- Verify `to_dict()` includes seq — this is the serialization path for both REST and WS
- Verify frontend types match the backend schema
