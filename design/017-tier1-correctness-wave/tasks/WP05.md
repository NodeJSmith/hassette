## Objectives & Success Criteria

- `LogTable` uses `seq` watermark to deduplicate REST+WS log entries — no duplicates at the REST/WS boundary
- Ring buffer is cleared on WS reconnect — no stale entries accumulate across connections
- After reconnect, log display recovers cleanly (REST refetch + fresh WS stream)
- Frontend test confirms dedup behavior

## Subtasks

1. In `frontend/src/state/create-app-state.ts`: expose `clear()` method on `LogStore` by calling through to `RingBuffer.clear()` and resetting the version signal
2. In `frontend/src/hooks/use-websocket.ts`: in the reconnect path (inside the `connected` handler when `hasConnectedRef.current` is already true), call `state.logs.clear()` BEFORE re-subscribing
3. In `frontend/src/components/shared/log-table.tsx`: after REST fetch resolves, compute `watermark = Math.max(...initialEntries.map(e => e.seq), 0)`. Store in a ref.
4. In `frontend/src/components/shared/log-table.tsx`: when combining entries, filter wsEntries to `entry.seq > watermark` before concatenation: `const allEntries = [...initialEntries.value, ...wsEntries.filter(e => e.seq > watermark)]`
5. Write frontend test: `frontend/src/components/shared/log-table.test.tsx::test_dedup_filters_ws_entries_below_watermark` — set up initial entries with seq 1-5, push WS entries with seq 3-8, assert only seq 6-8 appear from WS side
6. Write frontend test: `frontend/src/state/create-app-state.test.ts::test_log_store_clear_resets_buffer` — push entries, call clear(), assert toArray() is empty and version incremented
7. Write frontend test: `frontend/src/hooks/use-websocket.test.ts::test_clears_log_store_on_reconnect` — simulate reconnect, assert logs.clear() was called
8. Run frontend tests: `cd frontend && npm test`
9. Run `uv run nox -s dev`

## Review Guidance

- Verify watermark uses `seq` (not `timestamp`) — this is the core design decision from the challenge
- Verify `clear()` is called BEFORE re-subscribe on reconnect, not after
- Verify the watermark ref is reset on component remount (fresh REST fetch = fresh watermark)
- Verify `Math.max(...[])` edge case is handled (empty initial entries → watermark = 0)

## Avoid

- Do NOT use timestamp for dedup — use `seq` (the core design decision from the challenge)
- Do NOT call `clear()` AFTER re-subscribe — clear first, then subscribe, then REST refetch populates fresh data
