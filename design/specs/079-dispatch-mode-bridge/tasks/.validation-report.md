## Warnings

1. **T01 Verify wording is ambiguous on current-code assertion.** The Prompt correctly instructs asserting `mock_warn.assert_called_once_with()` (no arguments) against the current code, then notes T03 will update the assertion. The Verify item says "asserting the watchdog fires at the patched threshold," which could be read as asserting the spy receives 0.05 — that is not true of the current code. An executor reading only the Verify item (not the Prompt) would write the wrong assertion. The Verify criterion should say "asserting the watchdog fires (spy called once, no threshold argument against the current code)." This is a within-task inconsistency, not a design-doc contradiction; it does not block approval.

2. **T02 `depends_on: []` — no explicit ordering against T01.** The design doc's Implementation Sequence says "1. Pin (RED-first)" then "2. Add shared code," implying T02 follows T01. T02 is purely additive (no call-site changes) and cannot break T01's pins, so running T02 concurrently with T01 is safe in practice. However, the explicit sequence in the design doc (step 2 after step 1) is not reflected in the task dependency graph. This is a low-risk omission.

3. **T01 Verify covers the bus release-drain test but AC#3 in the design doc mentions only the stall-watch test.** T01 adds the drain test as a second new test, which the design's Test Strategy section calls for but does not formalize as a separate AC. The Verify item bundles both into AC#3. This is a harmless expansion of the AC's scope rather than a violation; the drain coverage is net positive.
