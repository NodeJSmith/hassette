## 2. Gaps (block PASS)

### Gap 1 — `get_job_summary()` duration aggregates are not excluded for `'skipped'` rows (confirmed by reading the live SQL)

The design doc's own "Blast Radius" section states: *"Telemetry pipeline: every status-aggregation query must account for `'skipped'` — missing one silently miscounts."* It then fixes exactly one query (`get_app_health_aggregates()` in `summary_queries.py`) and adds a blanket statement: *"The same exclusion applies to any other AVG/MIN/MAX duration aggregation that would be distorted by zero-duration skip records."*

Reading `src/hassette/core/telemetry/registration_queries.py::get_job_summary()` (lines 188-191) shows it computes, over **all** job executions with no status filter:

```sql
COALESCE(SUM(e.duration_ms), 0.0) AS total_duration_ms,
COALESCE(AVG(e.duration_ms), 0.0) AS avg_duration_ms,
MIN(e.duration_ms) AS min_duration_ms,
MAX(e.duration_ms) AS max_duration_ms,
```

These three fields (`avg_duration_ms`, `min_duration_ms`, `max_duration_ms`) are public fields on `JobSummary` (`src/hassette/schemas/telemetry_models.py`) — `min_duration_ms`'s own docstring reads *"0.0 means executed in under 1ms"*, which becomes actively misleading once a skip (duration_ms=0.0, never executed) can produce that same value. Once any job accumulates a skip, `min_duration_ms` will drop to 0.0 and `avg_duration_ms` will be diluted, on the same job-detail page T05 is adding the `Skipped` stat cell to.

**T04's prompt (item 2, `registration_queries.py`) only asks for the `skipped` bucket and the two predicate columns in `get_job_summary()` — it does not ask for the `status != 'skipped'` exclusion on `avg_duration_ms`/`min_duration_ms`/`max_duration_ms` in that same query.** The design doc itself only shows the fix applied to `summary_queries.py`, so this gap originates in the design doc and is carried through unchanged into T04. No task addresses it.

**Fix:** T04 needs an additional line item instructing the same `AND e.status != 'skipped'` exclusion be applied to `avg_duration_ms`, `min_duration_ms`, and `max_duration_ms` inside `get_job_summary()`.

### Gap 2 — Two design-doc-designated test files are not assigned to any task

Design doc's "Test Strategy > Existing Tests to Adapt" lists five files. Three are covered by task Target Files (`test_scheduled_job_lifecycle.py` → T01/T02, `tests/integration/test_scheduler.py` → T03, `test_model_types.py` → T01). Two are not referenced by any task's Target Files, Prompt, or Verify section, in any of T01–T05:

- `tests/unit/scheduler/test_scheduled_job_mark_registered.py` — "verify predicate doesn't interfere with registration flow"
- `tests/unit/core/test_scheduler_service_reschedule.py` — "verify rescheduling is unaffected by predicate presence"

Both files exist in the repo today. Neither is fatal to the feature working (both construct `ScheduledJob` via keyword arguments with defaults, so the new `predicate`/`_predicate_wants_job` fields won't break them mechanically), but the design doc explicitly called for them to be checked/adapted, and as written, an executor following the task files exactly would never open either file — so if `dispatch_and_log()`'s new predicate-check insertion point (T03) subtly changes reschedule timing or removal-callback behavior, `test_scheduler_service_reschedule.py` would be the test to catch it, and nothing routes anyone there.

**Fix:** Add both files to T03's Target Files (they exercise `dispatch_and_log()` behavior, which T03 modifies) or explicitly call them out in T03's Verify/Focus section.

## 3. Warnings (do not block PASS)

1. **T01's Verify checklist mislabels an item as FR#16.** T01's `implements` front-matter lists FR#16, and its Verify section has a bullet "`FR#16: ExecutionStatus enum includes SKIPPED (async validation is in T02)`". This describes adding the `SKIPPED` enum member, not FR#16 ("Predicates are synchronous only — async callables raise `TypeError` at registration time"). The enum addition more properly supports FR#5/FR#12. Not a functional problem — T02 fully implements and correctly verifies the actual FR#16 — but the traceability label is wrong and could confuse a reviewer checking "is FR#16 done" against T01 alone.

2. **T03 references a helper function that doesn't exist.** T03's prompt says to "update `_job_insert_params()` (or equivalent parameter-building helper)" in `repository.py`. No such helper exists — `register_job()` builds its params dict inline (lines 389-406), unlike `register_listener()` which does use a `_listener_insert_params()` helper. The parenthetical hedge covers this, but it's worth the implementer noticing there's no helper to "update" — they need to add keys directly to the inline dict literal.

3. **Design doc's `read: src/hassette/web/routes/scheduler.py` verification step (Changed Files list) isn't assigned to any task.** It's a low-risk "confirm routes use `JobSummary` directly as `response_model`" check, not a code change, so omission is unlikely to cause a defect — but no task's Target Files or Prompt mentions it.
