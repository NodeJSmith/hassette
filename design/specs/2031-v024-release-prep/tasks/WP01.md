## Objectives & Success Criteria

- The `## Unreleased` section of `CHANGELOG.md` is compressed from ~200 lines of PR-by-PR entries into themed groups with flat bullet lists
- Breaking Changes section appears first with 4 items and migration notes
- Only user-facing changes are included; internal implementation details are omitted
- The section header remains `## Unreleased` (version stamp deferred to Phase 2)
- Cross-referencing `git log --oneline v0.23.0..HEAD` against the compressed changelog confirms no user-facing commit is missing

## Subtasks

1. Read `CHANGELOG.md` lines 8-202 (the full Unreleased section) to inventory all entries
2. Classify each entry as user-facing or internal-only using the design doc's guiding principle and explicit omit/include lists
3. Write the **Breaking Changes** section with 4 items:
   - `TriggerProtocol` split (#452)
   - `/api/healthz` removed (#448)
   - Scheduler job name uniqueness + `if_exists` (#297)
   - `once=True` + debounce/throttle `ValueError` (#430)
4. Write themed groups (flat bullet lists, no Added/Changed/Fixed subheadings):
   - **Web UI** — Preact SPA rebuild, design tokens, session scope toggle, source location display, accessibility, self-hosted fonts
   - **Database & Telemetry** — persistent storage, retention/size config, degraded indicator
   - **Bus & Scheduler** — rate limiter redesign, `once=True` + rate limiting ValueError, debounce/throttle validation, `if_exists` parameter
   - **Logging** — per-service `*_log_level` config fields
   - **Configuration** — new config fields (`total_shutdown_timeout_seconds`, `db_path`, `db_retention_days`, `db_max_size_mb`)
   - **Bug Fixes** — user-visible fixes only (startup races, WebSocket reconnect, stale telemetry, connection bar flash, URL quote parsing)
5. Verify completeness: run `git log --oneline v0.23.0..HEAD` and confirm every user-facing commit is represented

## Test Strategy

N/A — no testable code changes. Verification is manual review of changelog content against git log.

## Review Guidance

- Every entry must pass the "would a user care?" test — reject any internal-only items
- Breaking Changes must each have a migration action (what to change)
- No duplicate `### Added`/`### Changed`/`### Fixed` headings — flat lists only
- PR references (#NNN) must be preserved for traceability
- The omit list from the design doc must be respected — no domain model layer, OpenAPI codegen, lifecycle propagation, etc.
