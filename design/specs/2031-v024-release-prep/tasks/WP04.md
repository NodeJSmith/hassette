## Subtasks

1. **sessions.md** — Read `frontend/src/pages/sessions.tsx` and `src/hassette/core/session_manager.py` for content. Write user-facing documentation covering:
   - The `/sessions` route and what it shows (session list with status badges, formatted timestamps)
   - Session scope toggle ("This Session" / "All Time") in the status bar with localStorage persistence
   - How sessions relate to Hassette restarts (new session per startup, orphan cleanup)
   - Follow the existing web-ui page structure (look at `apps.md` or `logs.md` as a template)

2. **database-telemetry.md** — Read `src/hassette/core/database_service.py` and `src/hassette/config/config.py` for content. Write user-facing documentation covering:
   - What telemetry is collected: handler invocations, job executions, sessions
   - Configuration: `db_path`, `db_retention_days`, `db_max_size_mb` with TOML examples
   - What the degraded indicator means in the UI
   - How to monitor via `/api/telemetry/status` endpoint (returns 503 when degraded)
   - Distinguish `/api/health` (overall system) from `/api/telemetry/status` (telemetry subsystem only)
   - Do NOT cover: Alembic migration internals, TelemetryRepository class, CommandExecutor architecture

3. **log-level-tuning.md** — Read `src/hassette/config/config.py` for all `*_log_level` fields. Write user-facing documentation covering:
   - Table of all per-service `*_log_level` config fields with their TOML key names and what service they control
   - How to set them in `hassette.toml` with examples
   - Fallback behavior: service-level → global `log_level`
   - Count the actual fields from the source (verify the "14" claim from the design doc)
   - Do NOT cover: the three-mode convention (dedicated/cross-bound/app-owned)

4. Run `uv run mkdocs build --strict` to verify all pages build correctly
