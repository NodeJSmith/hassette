## Objectives & Success Criteria

Add the per-app config endpoint, app source endpoint, WS completion events via bus topics, the `timed_out` field fix, and `ConfigResponse` path fields. After this WP: `GET /apps/{key}/config` returns app config for any app state, `GET /apps/{key}/source` returns source with path traversal protection, WS broadcasts `invocation_completed` and `execution_completed` events, `ListenerWithSummary` includes `timed_out`, and `ConfigResponse` includes `app_dir`/`data_dir`/`config_dir`. All schemas regenerated.

## Subtasks

1. Add `timed_out: int = 0` to `ListenerWithSummary` in `src/hassette/web/models.py`. Update `to_listener_with_summary()` in `src/hassette/web/mappers.py` to copy `timed_out` from `ListenerSummary`.
2. Add `app_dir`, `data_dir`, `config_dir` as `str` fields to `ConfigResponse` in `src/hassette/web/models.py`. Update the config route to populate them from the `HassetteConfig` instance.
3. Create `AppConfigResponse` model in `src/hassette/web/models.py`: `app_key: str`, `filename: str`, `class_name: str`, `enabled: bool`, `app_config: dict[str, Any] | list[dict[str, Any]]`.
4. Add `GET /apps/{app_key}/config` route in `src/hassette/web/routes/apps.py`. Look up `AppManifest` from the registry. Return `AppConfigResponse` with raw `manifest.app_config`. Return 404 if app_key not found.
5. Create `AppSourceResponse` model: `app_key: str`, `filename: str`, `content: str`, `line_count: int`.
6. Add `GET /apps/{app_key}/source` route. Use `manifest.full_path` to read the file. Validate `manifest.full_path.resolve()` starts with `manifest.app_dir.resolve()` — return 403 on traversal. Return 404 if file not found. Return `AppSourceResponse`.
7. Add two new `Topic` values to `src/hassette/types/enums.py`: `HASSETTE_EVENT_INVOCATION_COMPLETED` and `HASSETTE_EVENT_EXECUTION_COMPLETED`.
8. In `CommandExecutor._drain_and_persist()` (line ~604): after `_persist_batch()` succeeds, emit `Topic.HASSETTE_EVENT_INVOCATION_COMPLETED` for each handler invocation record in the batch (carrying `listener_id`). Emit `Topic.HASSETTE_EVENT_EXECUTION_COMPLETED` for each job execution record (carrying `job_id`).
9. In `CommandExecutor._handle_fk_violation()`: emit the same topics after `persist_batch_with_fk_fallback()` succeeds.
10. Create `InvocationCompletedData` and `ExecutionCompletedData` Pydantic models in `src/hassette/web/models.py` with fields: `listener_id`/`job_id`, `app_key`, `instance_index`, `status`, `duration_ms`, `error_type: str | None`.
11. Create `InvocationCompletedWsMessage` and `ExecutionCompletedWsMessage` models. Add both to the `WsServerMessage` discriminated union.
12. In `RuntimeQueryService`: subscribe to the two new topics. On receipt, resolve `app_key` and `instance_index` from the listener/job registry. Build the enriched WS payload and broadcast. Use per-drain batching (accumulate completions received within a single event-loop tick, then broadcast one summary message).
13. Run schema regeneration: `uv run python scripts/export_schemas.py --types`.
14. Manually update `frontend/src/api/ws-types.ts` with the two new message interfaces and add them to the `WsServerMessage` union type.

## Test Strategy

- Test the config endpoint: `tests/integration/test_api_app_config.py` — test with a known app_key (returns config), unknown app_key (404), multi-instance app (returns list config).
- Test the source endpoint: `tests/integration/test_api_app_source.py` — test with a valid app (returns source), missing file (404), path traversal attempt (403).
- Test `timed_out` appears in listener response: update existing listener telemetry tests to assert the field is present.
- Test WS completion events: verify that after a handler invocation is persisted, a WS client receives an `invocation_completed` message with correct `app_key`.
- Run `uv run nox -s dev -- -n 2`.

## Review Guidance

- The source endpoint MUST validate `manifest.full_path.resolve().is_relative_to(manifest.app_dir.resolve())` — not `config.app_dir`. This was a challenge finding (Finding 2).
- `CommandExecutor` must NOT import or reference `RuntimeQueryService` directly — it only emits bus topics. The coupling is via `Topic` enum values only.
- The `WsServerMessage` union must include both new message types — missing this causes schema drift that CI catches late.
- `ws-types.ts` is hand-maintained — verify the new interfaces match the Pydantic models exactly.
