## Reference

Fields and endpoints required by WP12, WP13, and WP14 that don't yet exist in the backend.

## Objectives

Add backend fields and endpoints needed by the frontend remediation WPs. Mostly additive fields on existing response models, plus two small new query methods.

## Subtasks

### 1. Hassette version in SystemStatus

Add `version: str` to `SystemStatus` (and `ConnectedPayload` for WS). Compute from `importlib.metadata.version("hassette")`. Surfaces in sidebar (WP13) and overview metadata (WP12).

### 2. Run rate in DashboardKpis

Add `runs_per_hour: float | None` to `DashboardKpisResponse`. Compute as `(total_invocations + total_executions) / window_hours`. When `window_hours < 1/60` (less than 1 minute of uptime), return `null` — the frontend will show the raw count instead of a misleading per-hour rate. `window_hours` is derived from the `since` parameter vs current time.

### 3. Source location on error entries

Add `source_location: str | None` to both `HandlerErrorEntry` and `JobErrorEntry` in the dashboard error response. Populate via a JOIN in `get_recent_errors()` against the `listeners` / `scheduled_jobs` table's `source_location` column. Surfaces in WP12's error table LOCATION column.

### 4. Invocation counts on manifests

Add `recent_invocations_1h: int` to `AppManifestResponse`. Populate from a fixed 1-hour window query on the existing telemetry data, keyed by `app_key`. Do NOT use the time-scoped dashboard endpoint — the sidebar is globally visible and must not couple to `timePreset`. Surfaces in sidebar (WP13).

### 5. Boot issues on SystemStatus

Add `boot_issues: list[BootIssue]` to `SystemStatus` where `BootIssue` has fields: `severity: str` ("err" | "warn"), `label: str`, `detail: str`. Populate from the existing boot-time validation results — hassette already logs config validation errors and blocked app reasons at startup. Collect these into a list on `RuntimeQueryService` during `on_initialize`. Surfaces in WP12's FrameworkErrorChip.

### 6. Activity feed endpoint

Add `GET /api/dashboard/activity` returning the most recent N (default 20) cross-app invocations and executions, merged and sorted by timestamp descending. Response model `DashboardActivityEntry` with fields: `status: str`, `timestamp: float`, `app_key: str`, `handler_name: str`, `duration_ms: float | None`, `error_type: str | None`, `kind: str` ("handler" | "job"). Query from `handler_invocations` UNION `job_executions` with `since` parameter support. Surfaces in WP12's recent activity feed.

### 7. Sparkline time buckets in DashboardKpis

Add `activity_buckets: list[ActivityBucket]` to `DashboardKpisResponse` where `ActivityBucket` has fields: `ok: int`, `err: int`. Divide the time window into 12 equal buckets, count ok and err invocations+executions per bucket. Surfaces in WP12's activity card sparkline.

### 8. Schema regeneration

After all changes: run `uv run python scripts/export_schemas.py` then `cd frontend && npx openapi-typescript openapi.json -o src/api/generated-types.ts`. Verify frontend builds.

## Test Strategy

- Unit test: `SystemStatus` includes `version` field with correct value
- Unit test: `DashboardKpisResponse` includes `runs_per_hour` — null for short windows, correct value otherwise
- Unit test: `BootIssue` list populated from validation errors
- Integration test: `/api/system/status` returns version and boot_issues
- Integration test: dashboard KPIs endpoint returns `runs_per_hour` and `activity_buckets`
- Integration test: `/api/dashboard/activity` returns merged invocations+executions sorted by timestamp
- Integration test: `AppManifestResponse` includes `recent_invocations_1h`

## Notes

Subtasks 1–5 are additive fields on existing response models — no database migrations needed. Subtask 6 is a new endpoint with a new query method in `TelemetryQueryService`. Subtask 7 adds bucketed counting to the existing KPIs query. Frontend schema regeneration is required after all changes.
