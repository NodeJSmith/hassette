## Objectives & Success Criteria

Add the Code tab (syntax-highlighted source), Config tab (per-app configuration with redacted-by-default values), and multi-instance support (parent overview with instance grid, instance switcher in detail header). After this WP: the Code tab shows the app's Python source with line numbers and handler annotations, the Config tab shows raw TOML config values with a reveal toggle for potentially sensitive data, multi-instance app parent pages show an instance grid, and the instance detail header includes a sibling switcher.

## Subtasks

1. Add Shiki as a dependency: `cd frontend && npm install shiki`. Configure with `github-light` and `github-dark` themes for Python syntax highlighting.
2. Create `frontend/src/components/app-detail/code-tab.tsx`: fetch app source via `getAppSource(appKey)`. Render with Shiki highlighting, line numbers in a gutter. If the response includes handler line numbers (from listener `source_location`), annotate those lines with the handler name in the gutter. Handle loading state and error state (file not found → "Source file not found at expected path").
3. Create `frontend/src/components/app-detail/config-tab.tsx`: fetch app config via `getAppConfig(appKey)`. Render app metadata (filename, class_name, enabled) as a structured header. Render `app_config` values as a key-value table. Values are redacted by default (shown as `••••••`) with a "Reveal" toggle button per-value or globally. For multi-instance apps with a list config, render per-instance config blocks.
4. Update `frontend/src/pages/app-detail.tsx`: add Code and Config tabs to the tab strip (alongside Handlers and Jobs from WP07). Wire tab switching.
5. Create multi-instance parent view: when `app_key` has instances (detected from manifest `instance_count > 1` or `instances` array), render the parent overview instead of single-instance detail. Show: display name with `×{count} instances` badge, instances grid (auto-fill, minmax 280px) with per-instance cards showing instance name, status pill, run count, last run time, uptime, inline error preview, sparkline. Cards link to instance detail.
6. Create instance switcher component: horizontal tab strip in the instance detail header showing sibling instances with status dots. Clicking a sibling navigates to that instance without returning to the parent.
7. Add breadcrumb navigation: when viewing an instance, show `← {parent_display_name}` breadcrumb above the title to navigate back to the parent overview.
8. Add component CSS for code tab (line numbers, gutter annotations, Shiki overrides), config tab (key-value table, redact/reveal toggle), instance grid, instance cards, instance switcher, breadcrumb.

## Test Strategy

- `components/app-detail/code-tab.test.tsx`: renders source code with line numbers, handles loading state, handles error state (file not found), handler annotations appear at correct lines.
- `components/app-detail/config-tab.test.tsx`: renders config metadata, values are redacted by default, reveal toggle shows values, handles multi-instance list config.
- `pages/app-detail.test.tsx` (extend): multi-instance app renders parent overview with instance grid, single-instance app renders detail directly, instance switcher navigates between siblings, breadcrumb navigates to parent.
- Run `npm run test`.

## Review Guidance

- Shiki must be configured to use the correct theme based on `[data-theme]` attribute — `github-light` for light mode, `github-dark` for dark mode. Dynamic theme switching must re-highlight.
- Config values MUST be redacted by default — the design doc explicitly notes that `AppManifest.app_config` may contain secrets. The reveal toggle is a security requirement.
- The instance grid must use CSS grid `auto-fill, minmax(280px, 1fr)` — not a fixed column count.
- The parent/instance routing must handle: `app_key` only → parent if multi-instance, detail if single-instance. `app_key + index` → instance detail.
