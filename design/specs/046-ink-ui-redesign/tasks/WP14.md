## Reference

**Source of truth:** `unified-hifi/handlers-tab.jsx` lines 50–59 (stats strip), lines 84–250 (detail pane).
**Screenshot:** See `mockup-app-detail.png` for the visual target.

## Objectives

Fix the app detail health strip to show the mockup's metrics, add the stats strip above the handler list, and enrich the handler detail pane.

## Subtasks

### 0. MasterList row enrichment

**Mockup ref:** `handlers-tab.jsx:99–213`

The master list handler rows need three additions not in the current implementation:

1. **Kind glyph column** (mockup line 101): Insert a glyph between the status dot and name. Map: `{ event: "⚡", state: "◇", schedule: "⏱", service: "☎" }`. This is in ADDITION to the existing kind chip — the glyph is a visual anchor in the narrow list column.

2. **Subline switching** (mockup lines 132–145): Below the handler name, show either `last_error_message` in `--err` color (when the handler is failing) OR `human_description` in dim (otherwise). Currently the row always shows `human_description`.

3. **Job next-run line** (mockup lines 139–143): For schedule-kind handlers, show an additional line: "next {relative_time}" in mono dim, or "cancelled" if the job is cancelled.

Grid layout: `14px 14px 1fr auto` (status dot, kind glyph, name+subline, count).

### 1. Health strip metrics

**Mockup ref:** `handlers-tab.jsx:53–59`

Change the 4-column health strip (currently: ERROR RATE, HANDLER AVG, JOB AVG, LAST ACTIVITY) to match the mockup's 5-column stats strip:

| Column | Label | Value | Tone |
|--------|-------|-------|------|
| 1 | HANDLERS | handler + job count | — |
| 2 | INVOCATIONS · 1H | total invocations + executions | — |
| 3 | SUCCESS RATE | computed from failed+timed_out vs total | warn if any errors |
| 4 | FAILED | total failed count | err if > 0 |
| 5 | TIMED OUT | total timed_out count | warn if > 0 |

Move this strip from the current position (above the tab strip) to inside the Handlers tab (above the master/detail layout), matching mockup line 53. **Remove the old health strip from `app-detail.tsx`** — do not leave it, or the page will show two strips. Code and Config tabs intentionally have no strip.

### 2. Handler detail pane: complete rendering spec

**Mockup ref:** `handlers-tab.jsx:260–400`

The detail pane header has these sections in order. Build ALL of them:

**For event/state handlers:**
1. Kind badge + handler name + status pill (e.g., `● state change  on_door_opened  ■ failing`)
2. `human_description` subtitle
3. `FIRES WHEN` line: use `predicate_description` field (NOT `human_description` — that's the subtitle). If `predicate_description` is null, omit this line.
4. Modifier chip row (mockup lines 331–340): debounce, throttle, once, priority, immediate — show only chips for non-null/non-false modifiers. These already exist in `ModifierChips` component — verify it renders in the detail pane.
5. `METHOD HANDLER` label + code block showing `handler_method` in mono
6. Source file location: `{filename}:{line}` — parse line number from `source_location` field (format: `filename.py:LINE`)
7. **Errored handler banner** (mockup lines 357–370): if the handler has errors, show a `--err` tinted card with "LAST ERROR — {error_type}" heading and error message. Render above the invocations table.
8. Stats row and invocations table (subtasks 3–4 below)

**For scheduled jobs:**
1. Kind badge + job name + status pill
2. `human_description` subtitle
3. Schedule chips row (mockup lines 288–328): trigger_type, trigger_label, trigger_detail, jitter, group. These already exist in `ScheduleChips` component — verify it renders.
4. **Next-run strip**: "next {relative_time}" or "fire at {time}" in mono
5. Stats row and executions table (subtasks 3–4 below)

### 3. Handler detail: stats row

**Mockup ref:** `handlers-tab.jsx:200–210`

Below the handler detail content, add a stats row:
```
CALLS · 1H    LAST        FAILED    TIMED OUT    P50    P95
{count}       {relative}  {count}   {count}      {ms}   {ms}
```

P50/P95 require backend percentile computation. If not available yet, show "—" as placeholder.

### 4. "view in code →" button

**Mockup ref:** `handlers-tab.jsx:210`

Add a button/link at the bottom of the handler detail that switches to the Code tab and scrolls to the handler's source line. Use the existing `source_location` field to determine the line number.

### 5. Code tab: header enrichment

**Mockup ref:** `unified-hifi/code-tab.jsx` header section

Add to the code tab header:
- Line count: `{lineCount} lines`
- "read-only" label
- "copy path" button that copies the filename to clipboard
