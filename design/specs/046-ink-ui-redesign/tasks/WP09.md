## Objectives & Success Criteria

Rebuild the logs page and build the config page. After this WP: the logs page renders a filterable, sortable table with Ink styling, real-time log streaming works via WS, level filtering controls both display and WS subscription, and the config page shows system configuration including the new path fields (`app_dir`, `data_dir`, `config_dir`).

## Subtasks

1. Rewrite `frontend/src/components/shared/log-table.tsx`: rebuild with Ink styling. Columns: timestamp (mono, dim), level badge (StatusShape-based: ok for INFO, warn for WARNING, err for ERROR/CRITICAL, mute for DEBUG), source location (module:function:line in mono), app (tag chip), message (flex, expandable), age (mono, dim). Sticky header. Level filtering controls. App filter dropdown. Search input. Sort by any column. Expandable rows for long messages. Row background tint for WARNING (`var(--warn-bg)`) and ERROR/CRITICAL (`var(--err-bg)`).
2. Rewrite `frontend/src/pages/logs.tsx`: full-page log table with all filter controls visible. Wire WS log subscription level to the filter control (changing filter sends `subscribe` message to update server-side filtering).
3. Create a config page in `pages/` that fetches system config via the existing config endpoint. Render as a structured table with key-value rows. Group related fields (connection settings, paths, timeouts, etc.). The new `app_dir`, `data_dir`, `config_dir` fields should appear in a "Paths" group.
4. Update `frontend/src/app.tsx`: wire the `/config` route to the new config page (replace placeholder from WP05).
5. Add component CSS for log table (sticky header, level badges, expandable rows, severity tinting, sort indicators), config page (key-value table, field groups).

## Test Strategy

- `components/shared/log-table.test.tsx`: renders log rows with all columns, level badge maps correctly, expanding a row shows full message, level filter hides/shows rows, app filter scopes display, search filters by message content, sort changes column order.
- `pages/logs.test.tsx`: renders full-page log table, filter controls are visible.
- `pages/config.test.tsx`: renders config key-value pairs, path fields (`app_dir`, etc.) are present, handles loading/error states.
- Run `npm run test`.

## Review Guidance

- Log table must use StatusShape-based level badges, not the old text badges — this is a design system requirement.
- The log-level filter must both filter the displayed rows AND update the WS subscription (via `updateLogSubscription`) — not just client-side filtering.
- Config page must not expose any fields outside the `ConfigResponse` allowlist — the endpoint handles this, but verify the frontend doesn't add mock fields.
- The log table is the most complex shared component — it was 392 lines before. The rebuild should be cleaner (FR#577 calls for splitting into composable hooks and view layer) but verify it's not oversimplified at the cost of functionality.
