## Objectives & Success Criteria

Rebuild the app detail page for single-instance and per-instance views: header with metadata and action buttons, health strip, and the unified Handlers tab containing both event handlers and scheduled jobs in a single master/detail layout. After this WP: clicking an app in the sidebar shows the detail page with all data surfaced (human_description, predicate_description, modifier chips, trigger schedule chips, timed_out split from failed, last_error_message inline), the master/detail layout works on desktop (300px list + flex detail) and collapses to drill-down on mobile, invocation/execution history tables render with expandable rows, and real-time WS events update the tables without manual refresh. App detail tabs are: Handlers | Code | Logs | Config.

## Subtasks

1. Rewrite `frontend/src/pages/app-detail.tsx`: header section with app metadata (app_key in mono, class name, file path, uptime). Lifecycle status pill (using StatusShape). Action buttons: Start (if stopped), Reload, Stop (with confirm dialog). Auto-loaded badge. Error banner (if crashed/failed) with last error type + message. Tab strip: Handlers (default), Code, Logs, Config (Code and Config tabs are placeholders until WP08).
2. Create `frontend/src/components/app-detail/handlers-tab.tsx`: master/detail layout containing a unified list of event handlers AND scheduled jobs. Desktop: 300px left master list + flexible right detail pane. Mobile (<768px via ResizeObserver): drill-down (tap item → detail only, back button to list). Fetch both `getAppListeners(appKey)` and `getAppJobs(appKey)`, merge into a single list sorted by kind then name.
3. Rewrite `frontend/src/components/app-detail/handler-list.tsx`: unified scrollable list of handlers and jobs. Each row shows: StatusShape for status, kind chip (tiny monospace: `state`, `event`, `cron`, `svc`), name (handler_method or job_name), `human_description` as subtitle, invocation/execution count, last-error preview if errored. Stats strip above the list: total count, invocations+executions/1h, success rate, failed count, timed-out count (distinct from failed, per FR#45).
4. Rewrite `frontend/src/components/app-detail/handler-row.tsx`: unified row component for both handlers and jobs. Row click selects the item in the detail pane. Kind chip visually distinguishes the type.
5. Create a kind-aware detail pane component. When a state/event handler is selected, show: full registration signature, modifier chips (debounce with ms value, throttle with value, once, priority with level, immediate), and invocation history table. Predicate description chip is deferred (#692 — backend currently returns raw `repr()` strings unsuitable for display). When a scheduled job is selected, show: trigger schedule chips (cron expression, human-readable `trigger_label` + `trigger_detail`, next run, fire_at, jitter, group), and execution history table.
6. Rewrite `frontend/src/components/app-detail/handler-invocations.tsx`: invocation history table for event handlers. Columns: status dot, timestamp, trigger description, duration, error note. Expandable rows showing event payload (JSON) and full traceback. "View triggering event" link (disabled text until Events inspector ships).
7. Rewrite `frontend/src/components/app-detail/job-executions.tsx`: execution history table for scheduled jobs. Columns: status, timestamp, duration, error note. Expandable for traceback.
8. Rewrite `frontend/src/components/app-detail/health-strip.tsx`: 4-column grid showing error rate, handler avg duration, job avg duration, last activity. Use `error_rate_class` from server for color coding.
9. Wire real-time updates: when `invocationCompleted` signal fires with a `listener_id` matching the currently viewed handler, trigger a targeted refetch of that handler's invocations. Same for `executionCompleted` + job. Dashboard-level KPIs/health also refetch (debounced). Cap the invocation/execution history list to the most recent N entries (e.g., 50) with a "Load more" button — real-time appends must not grow the list unbounded (see Edge Case #5 in the design doc).
10. Handle `focus` query param on the app detail route: when the URL contains `?focus={handler_method}`, auto-select that handler in the master list and scroll it into view on mount. This supports deep-linking from the command palette (WP10) and future cross-linking from the events inspector. Use `wouter`'s `useSearch()` or `URLSearchParams` to read the param. Clear it from the URL after applying (via `replaceState`) to avoid stale focus on subsequent navigations.
11. Delete `frontend/src/components/app-detail/job-list.tsx` and `job-row.tsx` (replaced by the unified list). Delete their tests.
12. Add component CSS to `global.css` for master/detail layout, unified handler rows, kind chips, invocation/execution rows, modifier/schedule chips, stats strip, health strip.

## Review Guidance

- Event handlers and scheduled jobs MUST appear in a single unified list — NOT separate tabs. This matches the design prototype.
- `human_description` (`handler_summary` field) must be rendered under every item name. Note: current summary strings are verbose (e.g., "Fires on hass.event.state_changed.zone.home entity zone.home and state Increased()") — render as a truncated subtitle with expand-on-hover, not a chip. Improving the summary format is tracked in #692.
- `timed_out` must be a SEPARATE count from `failed` — not collapsed. This was a challenge finding (Finding 5).
- Modifier chips (for handlers) and schedule chips (for jobs) must show VALUES (e.g., "debounce 500ms", "every day at 22:00") not just labels. `registration_source` chip is omitted — decorator registration is not yet implemented, so the value is always "programmatic" and the chip would be visual noise.
- The detail pane must be kind-aware: handler-specific fields (modifiers, invocation history) for state/event items, job-specific fields (cron, next run, jitter, execution history) for schedule items. Predicate description chip is deferred (#692).
- The master/detail layout must use ResizeObserver (not media queries) for the mobile collapse.
- Real-time refresh must be targeted (refetch the specific item's invocations/executions, not all items) to avoid UI flicker.
