## Objectives & Success Criteria

Rebuild the app detail page for single-instance and per-instance views: header with metadata and action buttons, health strip, and the two primary tabs (Handlers and Jobs) using the master/detail layout. After this WP: clicking an app in the sidebar shows the detail page with all handler data surfaced (human_description, predicate_description, modifier chips, timed_out split from failed, last_error_message inline), the master/detail layout works on desktop (300px list + flex detail) and collapses to drill-down on mobile, invocation/execution history tables render with expandable rows, and real-time WS events update the tables without manual refresh.

## Subtasks

1. Rewrite `frontend/src/pages/app-detail.tsx`: header section with app metadata (app_key in mono, class name, file path, uptime). Lifecycle status pill (using StatusShape). Action buttons: Start (if stopped), Reload, Stop (with confirm dialog). Auto-loaded badge. Error banner (if crashed/failed) with last error type + message.
2. Create `frontend/src/components/app-detail/handlers-tab.tsx`: master/detail layout. Desktop: 300px left master list + flexible right detail pane. Mobile (<768px via ResizeObserver): drill-down (tap handler → detail only, back button to list).
3. Rewrite `frontend/src/components/app-detail/handler-list.tsx`: scrollable list of handlers. Each row shows: StatusShape for status, kind glyph (⚡ state, ◇ event, ⏱ schedule, ☎ service), handler name, `human_description` as subtitle, invocation count, last-error preview if errored. Stats strip above the list: handler count, invocations/1h, success rate, failed count, timed-out count (distinct from failed, per FR#45).
4. Rewrite `frontend/src/components/app-detail/handler-row.tsx`: remove expand-on-click pattern (now handled by master/detail selection). Row click selects the handler in the detail pane.
5. Rewrite `frontend/src/components/app-detail/handler-invocations.tsx`: invocation history table in the detail pane. Columns: status dot, timestamp, trigger description, duration, error note. Expandable rows showing event payload (JSON) and full traceback. "View triggering event" link (disabled until Events inspector ships — renders as text, not a link). Modifier chips above the table: debounce (with ms value), throttle (with value), once, priority (with level), immediate, registration_source (decorator/programmatic). Predicate description chip.
6. Rewrite `frontend/src/components/app-detail/job-list.tsx`: same pattern as handler-list. Each row shows: StatusShape, job name, trigger type label, next run time, execution count. Stats strip: job count, executions/1h, success rate, failed count, timed-out count.
7. Rewrite `frontend/src/components/app-detail/job-row.tsx`: row click selects job in detail pane.
8. Rewrite `frontend/src/components/app-detail/job-executions.tsx`: execution history in detail pane. Columns: status, timestamp, duration, error note. Expandable for traceback. Trigger schedule info: cron expression, human-readable label (`trigger_label` + `trigger_detail`), next run, fire_at, jitter, group.
9. Rewrite `frontend/src/components/app-detail/health-strip.tsx`: 4-column grid showing error rate, handler avg duration, job avg duration, last activity. Use `error_rate_class` from server for color coding.
10. Wire real-time updates: when `invocationCompleted` signal fires with a `listener_id` matching the currently viewed handler, trigger a targeted refetch of that handler's invocations. Same for `executionCompleted` + job. Dashboard-level KPIs/health also refetch (debounced).
11. Add component CSS to `global.css` for master/detail layout, handler rows, invocation rows, modifier chips, stats strip, health strip.

## Review Guidance

- `human_description` must be rendered under every handler name — it's the most impactful data surfacing change.
- `timed_out` must be a SEPARATE column/count from `failed` — not collapsed. This was a challenge finding (Finding 5).
- Modifier chips must show the VALUE (e.g., "debounce 500ms") not just the label ("debounce").
- The master/detail layout must use ResizeObserver (not media queries) for the mobile collapse — matching the prototype's pattern.
- Real-time refresh must be targeted (refetch the specific handler's invocations, not all handlers) to avoid UI flicker.
