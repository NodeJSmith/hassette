# Context: CSS Modules Migration

## Problem & Motivation
All frontend component styles live in a single 4600-line `global.css` file. When components change, their old styles remain because no one can determine which selectors are still used. New styles are written for things that already exist because the monolith is undiscoverable. Duplicated and near-duplicate styles accumulate. A recent quality pass manually removed ~100 lines of orphaned CSS and 3 duplicate utilities — symptoms of a structural problem. The migration extracts component-specific styles into co-located `.module.css` files, leaving `global.css` as a slim (~400 line) foundation of shared tokens, resets, layout primitives, and cross-cutting utilities.

## Visual Artifacts
None.

## Key Decisions
1. **CSS Modules via Vite's native support** — `*.module.css` files are auto-scoped. No configuration change for the core feature. Chosen for zero-runtime-cost scoping and ecosystem maturity.
2. **`vite-css-modules` plugin for per-class type safety** — generates `.d.ts` per module so `styles.typo` is a compile error. Installed in T01, not deferred.
3. **`clsx` for conditional class composition** — 330 bytes, zero deps. Replaces error-prone template literal interpolation for mixed global+module+conditional classes.
4. **`ht-` prefix removed inside modules** — module scoping makes namespace prefixes redundant. Global classes keep `ht-` for namespacing.
5. **3-file threshold for "stays global"** — classes used across 3+ component files (`ht-btn`, `ht-card`, `ht-table`, `ht-badge`, `ht-chip`, `ht-text-*`, `ht-pill`) stay in `global.css`. 2-file classes adjudicated per the design doc's table.
6. **State modifiers use `:global()`** — `.is-active`, `.is-blocked`, `.is-expanded` are applied as string literals, so they need `:global()` in module files to avoid being scoped.
7. **ARIA attributes replace class-based state assertions** — unit tests use `aria-current`, `aria-expanded` instead of `className.toContain("is-active")`.
8. **Atomic migration rule** — each batch includes its e2e test updates in the same commit.
9. **Media query blocks decomposed upfront in T02** — the mixed 900px/768px/480px blocks are split into per-component labeled sections before any component migration.
10. **Playwright screenshot baselines** — `expect(page).to_have_screenshot()` at 4 viewports x 2 themes for key pages, generated on Ubuntu.

## Constraints & Anti-Patterns
- **Preact uses `class=` not `className=`** — all code must use `class=` with template literals or `clsx`.
- **No bulk migration** — global.css and module files must coexist at all times during migration.
- **Shared classes must NOT move to modules** — `ht-btn`, `ht-card`, `ht-table`, `ht-badge`, `ht-chip`, `ht-text-*`, `ht-pill` stay global.
- **No new runtime dependencies beyond `clsx`** — no CSS-in-JS, no Tailwind.
- **No empty modules** — components using only global classes (e.g., `icons.tsx`, `error-boundary.tsx`) do not get a `.module.css` file.
- **Cross-component contextual rules use `:global()`** — e.g., `:global(.ht-drawer) .sidebar { display: flex }`.
- **Dark theme overrides for component classes** — use `:global([data-theme="dark"]) .localClass { ... }`.
- **Generated `.d.ts` files are gitignored** — regenerated by `vite build` and dev server.
- **Orphaned classes to remove first**: `ht-error-feed`, `ht-error-entry`, `ht-group-filter-bar`, `ht-group-chip`.
