# Design: Startup Smoke Tests Against Real Home Assistant

**Date:** 2026-03-13
**Status:** approved
**Research brief:** [design/research/2026-03-13-startup-smoke-tests/research.md](../../research/2026-03-13-startup-smoke-tests/research.md)

---

## Problem

Two consecutive PRs (#329, #330) passed the full test suite — unit, integration, coverage — but crashed on first homelab deployment due to startup race conditions:

1. `register_listener()`/`register_job()` accessed `database_service.db` before migrations finished
2. Handlers fired via WebSocket before `_create_session()` completed, raising `RuntimeError("No active session")`

Both bugs live in the window between WebSocket connection and full service readiness — a window that `HassetteHarness` cannot exercise because it unconditionally mocks `WebsocketService` and never calls `Hassette.run_forever()`. No test currently exercises the 16-service startup sequence with a real WebSocket handshake.

---

## Non-Goals

- Does not replace `HassetteHarness` for integration testing of individual components
- Does not add HA API compatibility testing (verifying HA service calls, state mutations, etc.)
- Does not test the Hassette Docker container or `docker_start.sh` — that's covered by `test_docker_integration.py`
- Does not add the fake HA WebSocket server (Option B) — that is a potential future complement for edge-case testing
- Does not gate every PR — the smoke test CI workflow runs only when core startup files change

---

## Architecture

### Overview

A new `tests/smoke/` tier runs Hassette **in-process** (via pytest-asyncio) pointed at a real `homeassistant/home-assistant:stable` Docker container using the `demo:` integration. The container is started before the test session via Docker Compose and torn down afterward.

The smoke tests exercise the exact path that crashed: `Hassette.run_forever()` → WebSocket handshake → all 16 services initialize → `_create_session()` → "Hassette is running." Five targeted assertions confirm clean startup.

### File Structure

```
tests/
  smoke/
    __init__.py
    conftest.py              # ha_container fixture + startup_context helper
    docker-compose.yml       # HA service definition
    test_startup.py          # the 5 smoke tests
  fixtures/
    ha-config/               # pre-seeded HA config (committed to repo)
      configuration.yaml     # demo: integration only
      .storage/

        onboarding           # {"done": ["user", "core_config", "analytics", "integration"]}

        core.config_entries  # empty entries list

```

### Pre-Seeded HA Config

HA stores auth in `config/.storage/auth` as versioned JSON containing bcrypt-hashed tokens. This file is generated **once** by `generate_ha_fixtures.py` and committed to the repo. The token string (`hassette-smoke-test-token`) is hardcoded in `tests/smoke/conftest.py` — it grants access only to an ephemeral test container with no real data.

**`tests/fixtures/ha-config/configuration.yaml`:**
```yaml
homeassistant:
  name: Hassette Smoke Test
  unit_system: metric
  time_zone: America/New_York

demo:
```

**`tests/fixtures/ha-config/.storage/onboarding`:**
```json
{
  "version": 4,
  "data": {
    "done": ["user", "core_config", "analytics", "integration"]
  }
}
```

**`tests/fixtures/ha-config/.storage/auth`:** Generated by `generate_ha_fixtures.py`. Contains one user (`admin`), one `homeassistant` auth provider credential, and one long-lived access token entry with the bcrypt hash of `hassette-smoke-test-token`. The exact schema follows HA's internal `auth` storage version 7 format.

**`tests/smoke/generate_ha_fixtures.py`** (one-time script, not a test):

**Important:** The HA auth storage format (`version: 7`) and the bcrypt hash for long-lived tokens should be validated when the pinned HA Docker image is updated. The `generate_ha_fixtures.py` script is the source of truth — re-run it after pinning a new HA version.

Notes:
- Port `18123` avoids collision with a running homelab HA on `8123`
- Fixed image version (`2025.3`) rather than `stable` tag — prevents fixture drift from HA updates
- Health check uses the pre-seeded token to verify both HA readiness AND that auth storage loaded correctly

### pytest Fixture: `ha_container`

**`tests/smoke/conftest.py`:**

```python
"""Smoke test fixtures."""
import asyncio
import subprocess
import time
from collections.abc import AsyncIterator
from contextlib import asynccontextmanager
from pathlib import Path
import pytest
import httpx
from hassette import Hassette
from hassette.config.config import HassetteConfig

COMPOSE_FILE = Path(__file__).parent / "docker-compose.yml"
HA_URL = "http://localhost:18123"
HA_TOKEN = "hassette-smoke-test-token"
STARTUP_TIMEOUT = 60  # seconds


@pytest.fixture(scope="session")
def ha_container():
    """Start the HA Docker container for the test session and tear it down after."""
    subprocess.run(
        ["docker", "compose", "-f", str(COMPOSE_FILE), "up", "-d", "homeassistant"],
        check=True,
    )
    # Wait for HA to be ready (matches docker-compose healthcheck)
    deadline = time.monotonic() + STARTUP_TIMEOUT
    while time.monotonic() < deadline:
        try:
            r = httpx.get(f"{HA_URL}/api/", headers={"Authorization": f"Bearer {HA_TOKEN}"}, timeout=3)
            if r.status_code == 200:
                break
        except Exception:
            pass
        time.sleep(2)
    else:
        subprocess.run(["docker", "compose", "-f", str(COMPOSE_FILE), "down"], check=False)
        pytest.fail(f"HA did not become ready within {STARTUP_TIMEOUT}s")
    yield HA_URL
    subprocess.run(["docker", "compose", "-f", str(COMPOSE_FILE), "down"], check=False)


@asynccontextmanager
async def startup_context(config: HassetteConfig, timeout: int = 30) -> AsyncIterator[Hassette]:
    """Run Hassette.run_forever() in a background task until ready, then yield for assertions."""
    hassette = Hassette(config)
    task = asyncio.create_task(hassette.run_forever())
    try:
        deadline = asyncio.get_event_loop().time() + timeout
        while hassette._session_id is None:
            if asyncio.get_event_loop().time() > deadline:
                task.cancel()
                raise TimeoutError(f"Hassette did not reach running state within {timeout}s")
            await asyncio.sleep(0.1)
        yield hassette
    finally:
        hassette.shutdown_event.set()
        await task


def make_smoke_config(ha_url: str, tmp_path: Path) -> HassetteConfig:
    """Build a minimal HassetteConfig pointing at the smoke test HA."""
    return HassetteConfig(
        base_url=ha_url,
        token=HA_TOKEN,
        data_dir=tmp_path,
        app_dir=tmp_path / "apps",
    )
```

Key design decisions:
- `ha_container` is `scope="session"` — HA starts once per `pytest -m smoke` run, not once per test
- `startup_context` polls `hassette._session_id` as the readiness gate — this is the exact point after all 16 services are ready and `_create_session()` has run
- `make_smoke_config` passes a `tmp_path`-based `data_dir` so each test gets a fresh SQLite DB
- `httpx` is already in dev dependencies

### The 5 Smoke Tests

**`tests/smoke/test_startup.py`:**

```python
"""Startup smoke tests — verify Hassette.run_forever() completes cleanly against real HA."""
import asyncio
import logging
import pytest
from tests.smoke.conftest import startup_context, make_smoke_config

pytestmark = pytest.mark.smoke


@pytest.mark.asyncio
async def test_startup_completes(ha_container, tmp_path):
    """Hassette reaches running state without uncaught exceptions."""
    config = make_smoke_config(ha_container, tmp_path)
    async with startup_context(config) as hassette:
        assert hassette._session_id is not None and hassette._session_id > 0


@pytest.mark.asyncio
async def test_session_created_in_db(ha_container, tmp_path):
    """A session row is written to SQLite after successful startup."""
    config = make_smoke_config(ha_container, tmp_path)
    async with startup_context(config) as hassette:
        async with hassette.database_service.db.execute(
            "SELECT id, status FROM sessions WHERE id = ?", (hassette.session_id,)
        ) as cursor:
            row = await cursor.fetchone()
        assert row is not None
        assert row[1] == "running"


@pytest.mark.asyncio
async def test_demo_entities_visible(ha_container, tmp_path):
    """Demo entities from the HA demo: integration are accessible via StateProxy."""
    config = make_smoke_config(ha_container, tmp_path)
    async with startup_context(config) as hassette:
        states = await hassette.api.get_states()
        entity_ids = {s["entity_id"] for s in states}
        assert "light.kitchen_lights" in entity_ids
        assert "binary_sensor.movement_backyard" in entity_ids


@pytest.mark.asyncio
async def test_bus_handler_fires_on_state_change(ha_container, tmp_path):
    """A registered bus handler receives a state_changed event from HA."""
    config = make_smoke_config(ha_container, tmp_path)
    received: list[object] = []

    async def capture_event(event):
        received.append(event)

    async with startup_context(config) as hassette:
        hassette.bus.on_state_change("light.kitchen_lights", handler=capture_event)
        # Trigger a state change via the HA API
        await hassette.api.call_service("light", "toggle", {"entity_id": "light.kitchen_lights"})
        # Give the event loop a moment to dispatch
        await asyncio.sleep(1.0)

    assert len(received) >= 1, "Expected at least one state_changed event"


@pytest.mark.asyncio
async def test_no_sentinel_records_dropped(ha_container, tmp_path, caplog):
    """No listener_id=0 or session_id=0 sentinel records are dropped during startup.

    This is the definitive regression test for the startup race conditions fixed in #330.
    Sentinel records indicate that a handler fired before its DB registration or before
    the session was created — both startup races. A clean startup produces zero sentinels.
    """
    config = make_smoke_config(ha_container, tmp_path)
    with caplog.at_level(logging.WARNING, logger="hassette.core.command_executor"):
        async with startup_context(config):
            await asyncio.sleep(2.0)  # let any startup-window events settle

    dropped = [r for r in caplog.records if "Dropping" in r.message and "invocation record" in r.message]
    assert dropped == [], f"Startup sentinel records were dropped: {[r.message for r in dropped]}"
```

### Nox Session

Add to `noxfile.py`:

```python
@nox.session(python=["3.13"])
def smoke(session: "Session"):
    """Startup smoke tests against a real HA Docker container.

    Requires Docker and the HA container to be running:
        docker compose -f tests/smoke/docker-compose.yml up -d
    Or run via CI which starts the container automatically.
    """
    session.run(
        "uv",
        "run",
        "--active",
        "--reinstall-package",
        "hassette",
        "pytest",
        "-m",
        "smoke",
        "-v",
        "-n",
        "0",       # No xdist — one shared ha_container fixture
        "--tb=short",
        external=True,
    )
```

Notes:
- Python 3.13 only (matches e2e session — these are slow, external-facing tests)
- `-n 0` disables xdist parallelism — the `ha_container` session fixture is not xdist-safe
- No `-W error` — Docker startup and HA may emit deprecation warnings irrelevant to Hassette

### pytest Marker Registration

Add `smoke` to `pyproject.toml` `[tool.pytest.ini_options]`:

```toml
markers = [
    "...",
    "smoke: marks startup smoke tests requiring a running HA Docker container (run with '-m smoke')",
]
```

Update `addopts` to also skip smoke by default:

```toml
addopts = "-m 'not docker and not e2e and not smoke'"
```

### CI Workflow

**`.github/workflows/smoke-tests.yml`:**

```yaml
name: Smoke Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
    paths:
      - "src/hassette/core/core.py"
      - "src/hassette/core/command_executor.py"
      - "src/hassette/core/websocket_service.py"
      - "src/hassette/core/database_service.py"
      - "src/hassette/core/bus_service.py"
      - "tests/smoke/**"
      - "tests/fixtures/ha-config/**"
      - ".github/workflows/smoke-tests.yml"
  workflow_dispatch:

jobs:
  smoke:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install uv
        uses: astral-sh/setup-uv@v5
        with:
          enable-cache: true
          python-version: "3.13"

      - name: Start HA container
        run: docker compose -f tests/smoke/docker-compose.yml up -d homeassistant

      - name: Wait for HA to be ready
        run: |
          for i in $(seq 1 30); do
            curl -sf http://localhost:18123/api/ \
              -H "Authorization: Bearer hassette-smoke-test-token" && break
            echo "Attempt $i/30 — waiting..."
            sleep 3
          done || (docker compose -f tests/smoke/docker-compose.yml logs homeassistant && exit 1)

      - name: Run smoke tests
        run: uv run nox -s smoke

      - name: Tear down HA container
        if: always()
        run: docker compose -f tests/smoke/docker-compose.yml down
```

Notes:
- `workflow_dispatch` allows manual runs anytime
- Path filter ensures it only runs when startup-relevant files change
- HA logs are printed on startup failure for easier debugging
- `if: always()` on teardown ensures cleanup even on test failure

### Dependencies

Add to `pyproject.toml` `[project.optional-dependencies]` (or `[dependency-groups]`) dev section:
- `bcrypt>=4.0` (for `generate_ha_fixtures.py`) — only needed to regenerate fixtures, but cheap to keep
- No other new runtime dependencies; `httpx` is already present

## Alternatives Considered

**Option B: Fake HA WebSocket server** — A Python `aiohttp` server implementing the HA handshake. Rejected because it would be a mock that might diverge from real HA behavior — the exact root cause of the problem we're solving. The startup race bugs were caused by mocks being too accommodating. A fake server that always returns the right messages in the right order would not catch future protocol-level issues.

**Full Docker container approach** — Run both Hassette and HA as Docker containers, assert on container logs. Rejected for this tier: harder to assert on internals (`session_id`, DB row, caplog), slower iteration cycle, and the Docker image tests are already covered by `test_docker_integration.py`.

**First-boot provisioning script** — Call HA's onboarding API on first start to create user and token dynamically. Rejected in favor of pre-seeded fixtures: simpler, no onboarding API variability, deterministic across CI runs. The pre-seeded approach is what the `hassette-examples` pattern already relies on.

## Open Questions

All resolved before writing this doc:

- ~~Token provisioning~~ → Pre-seeded `.storage/auth` committed to repo; `generate_ha_fixtures.py` for regeneration
- ~~Hassette execution in tests~~ → In-process via `startup_context` async context manager
- ~~HA onboarding~~ → Pre-seeded `.storage/onboarding` with `"done"` list suppresses wizard

Remaining unknowns (low risk, can be resolved during implementation):

- [ ] **HA auth storage format validation**: The `.storage/auth` schema is from HA 2025.3. Validate that the bcrypt hash format for long-lived tokens matches what HA 2025.3 expects. Run `generate_ha_fixtures.py` against the pinned image and verify the health check passes.
- [ ] **`startup_context` readiness gate**: Polling `_session_id` directly accesses a private attribute. If we want to avoid that, an alternative is watching for the "Hassette is running." log message via `caplog`. The private attribute approach is simpler and more direct.
- [ ] **CI runner timing**: HA takes 10–20 seconds to boot on GitHub Actions free runners. The 30-second wait loop (10 attempts × 3s) should be sufficient, but may need tuning.
