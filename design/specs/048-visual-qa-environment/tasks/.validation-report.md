## Step 4: Findings

### Finding #1 — MEDIUM: AC#1 verify items do not collectively satisfy the full criterion

**Criterion (design.md):**
> AC#1: Running the single command from a clean worktree (including worktrees at non-standard paths) results in **all three services running** and **a dashboard visible at the reported frontend URL** (validates FR#1, FR#3, FR#6, FR#8)

**What the tasks verify:**
- T02 verify item for AC#1: "scripts/docker/ha-demo.yml uses ${HA_PORT} and ${HA_CONFIG_PATH} env var substitution, healthcheck uses the pre-seeded JWT token, and container name includes the port for uniqueness." — This verifies file contents, not runtime behavior.
- T04 verify item for AC#1: "After the script prints DEMO_READY=true, all three URLs respond (HA returns 200 with auth header, hassette /api/health returns 200, Vite returns HTML)."
- T05 verify item for AC#1: "mise run demo produces the same result as uv run nox -s demo."

**Gap:** No verify item checks that a **dashboard is visible** at the reported frontend URL. T04 checks that Vite returns HTML, which is necessary but not sufficient to verify that the dashboard (the actual hassette UI with its navigation and app data) renders correctly. T05's verify item ("produces the same result") is too vague — it doesn't define what "same result" means in terms of verifiable output.

The non-standard worktree path sub-criterion ("including worktrees at non-standard paths") is addressed by FR#8/T04 via the `Path(__file__).resolve().parent.parent` approach, but no verify item for AC#1 explicitly tests the non-standard path case.

**Recommendation:** Add to T04's verify item for AC#1: a check that the frontend URL returns HTML containing dashboard-recognizable content (e.g., the page title or a known UI element). Add a verify item noting that the test must be run from a non-standard path (e.g., a worktree at a non-default location) to confirm FR#8 integration.

### Finding #2 — HIGH: AC#3 verify item omits "event activity" sub-criterion

**Criterion (design.md):**
> AC#3: The dashboard shows active automation app instances with **listeners, scheduled jobs, and event activity** (validates FR#4)

**What T06 verifies:**
> "The test queries GET /api/apps and verifies the response includes active app instances with **listeners and scheduled jobs**"

**Gaps:**

**2a. "Event activity" is not verified.** The design explicitly names three categories: listeners, scheduled jobs, and event activity. T06's verify item lists only two of these. No task contains a verify item that checks event activity (e.g., via the logbook, event history, or invocation records visible in the UI or API).

**2b. AC#3 tests the wrong artifact.** AC#3's criterion text says "The dashboard shows..." — this is a UI criterion. T06 tests it via the REST API (`GET /api/apps`), not by verifying the dashboard renders the data. If the API returns data but the frontend fails to display it, AC#3 would pass in CI while being visually broken. For a visual QA environment spec, this is a meaningful gap.

The design doc's Test Strategy section acknowledges: "Manual verification: Start the demo, confirm dashboard shows live data." If this is the intended mechanism for AC#3, the task should say so explicitly and the verify item should reflect it (e.g., "Manual: dashboard shows running apps, listener counts, scheduled jobs, and recent event activity after demo starts").

**Recommendation:**
- Add an explicit verify item for event activity — either via `GET /api/invocations` (or equivalent endpoint) or as a manual verification note.
- Clarify whether AC#3 is validated programmatically (API) or manually (dashboard). If manually, mark it clearly in the T06 verify item or move it to a separate "Manual Verification" section in T06's Verify block.

### Finding #3 — LOW: T02's claim of AC#1 inflates apparent coverage without adding verifiable behavior

**Observation:** T02 lists `AC#1` in its `implements` field, but its verify item for AC#1 checks only static file contents (compose file structure, env var names, JWT token value). AC#1 is a runtime behavioral criterion — "all three services running and a dashboard visible." T02's contribution is a *precondition* for AC#1, not a verification of AC#1 itself.

This is not a coverage gap (T04 properly verifies AC#1 runtime behavior) but it creates a misleading traceability signal: a reader scanning the matrix might assume AC#1 has two independent verification points when T02's verify item does not actually validate the criterion.

**Recommendation:** Remove `AC#1` from T02's `implements` field (or relabel T02's verify item to "Precondition for AC#1: compose file structure is correct") to keep traceability semantically accurate.
