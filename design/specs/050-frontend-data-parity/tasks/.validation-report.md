## 2. Coverage Gaps

### GAP-1: FR#12 and AC#8 not claimed in T06 frontmatter

**Severity: MEDIUM**

T06 is the task that implements the frontend diagnostics page — the component that actually renders the services panel required by FR#12 and AC#8. The backend data is prepared by T03, but the design doc explicitly splits responsibility:

- FR#12: "A system diagnostics page displays each internal service with its current status, readiness phase, and role"
- AC#8: "The system diagnostics page shows all internal services with status, role, and readiness phase, updating in real time via WebSocket (FR#12)"

T06's `implements` frontmatter only lists: `["FR#14", "FR#15", "FR#16", "AC#10", "AC#11", "AC#12"]`. Neither FR#12 nor AC#8 appear. The task body does implement the services panel (see the "Services panel" section), but the formal claim is missing.

T03's `implements` includes FR#12, FR#13, AC#8, AC#9 — but T03 is the backend task (new endpoint + `ServiceInfoResponse` extension). The frontend rendering in T06 is what satisfies the user-facing portion of FR#12 and AC#8.

**Impact:** No functionality gap — the work is described in T06's prompt. The traceability gap means there is no explicit verify checklist item in T06 for services panel rendering against FR#12 criteria.

### GAP-2: FR#13 and AC#9 not claimed in T06 frontmatter

**Severity: MEDIUM**

Same pattern as GAP-1. FR#13 states: "Services in a cooling/retry state display the retry time as a relative timestamp that refreshes on WebSocket updates." AC#9 states: "A service in cooling state shows a relative retry timestamp (e.g., 'retry in 3m') that refreshes on WebSocket updates."

T06's task body describes this exactly ("Services in cooling state (status contains 'cooling' or 'exhausted_cooling'): show relative retry timestamp using `formatRelativeTime(retry_at)` — this refreshes when WS updates the signal"), but neither FR#13 nor AC#9 appear in T06's `implements` frontmatter.

T03 claims both FR#13 and AC#9, but T03 only delivers the data (`retry_at` field on `ServiceInfoResponse`). The UI refresh behavior is entirely T06's responsibility.

**Impact:** Same as GAP-1 — no functionality gap, but the verify checklist in T06 has no explicit check for the cooling state behavior (FR#13 / AC#9).

### GAP-3: FR#17 not claimed in T04 frontmatter

**Severity: LOW**

FR#17 states: "Handler error banners on the detail pane include an expandable traceback section (currently shows type and message only)."

T02 claims FR#17 (`implements: ["FR#4", "FR#17", "AC#3", "AC#13"]`) and delivers the backend data (`last_error_traceback` field). T04 implements the UI rendering (expandable traceback disclosure pattern for handlers). However, T04's `implements` frontmatter is `["FR#2", "FR#5", "FR#6", "FR#7", "FR#8", "AC#4", "AC#5"]` — FR#17 is absent. T04 does have AC#13 in its verify checklist, but AC#13 is a derived outcome; FR#17 is the primary requirement.

T02 and T04 together satisfy FR#17, but only T02 claims it. The UI half (T04) should also reference FR#17.

**Impact:** No functionality gap. The traceback disclosure pattern is described in T04's prompt. Low-severity traceability issue only.

## 3. Contradictions

### CONTRA-1: T03 claims FR#12, AC#8 but describes only backend data

**Severity: LOW**

T03's `implements` frontmatter includes `["FR#10", "FR#12", "FR#13", "AC#7", "AC#8", "AC#9"]`. FR#12 and AC#8 describe the diagnostics *page* rendering:

- FR#12: "A system diagnostics page displays each internal service with its current status, readiness phase, and role"
- AC#8: "The system diagnostics page shows all internal services with status, role, and readiness phase, **updating in real time via WebSocket**"

T03 does not create the diagnostics page — T06 does. T03 only delivers the backend data (`ServiceInfoResponse` extension and `GET /api/health` response fields). Claiming the full FR#12 and AC#8 in T03 overstates what T03 actually delivers and contributes to the gap in T06 (see GAP-1, GAP-2).

A clean split would have T03 claim the data layer ("ServiceInfoResponse extended with role/ready_phase/retry_at") and T06 claim FR#12, AC#8, FR#13, AC#9 for the page rendering.

### CONTRA-2: T03 verify checklist mixes backend and frontend concerns

**Severity: LOW**

T03's verify list states:
- `FR#12: The global jobs response includes each service with status, readiness phase, and role (via extended ServiceInfoResponse)`
- `AC#8: GET /api/health response includes services with role, ready_phase, and retry_at fields`

The verify items reframe FR#12 and AC#8 narrowly as backend API fields — which is the only part T03 can actually verify. This is a reasonable defense of the claim but creates a situation where FR#12's actual intent (a rendered diagnostics page) is never formally checked by any verify item in any task.

## 4. Vague or Underspecified Criteria

### VAGUE-1: T04 traceback toggle behavior underspecified

T04 says "Use a disclosure pattern: collapsed by default, click to expand." The design doc says "expandable traceback section" but does not specify the toggle interaction precisely. T04 does not specify:
- Whether the toggle label is "Show traceback" / "Hide traceback" or a chevron icon
- Whether the design context component inventory defines a disclosure pattern

This is minor — T04 references existing handler error banner line 218 as a template. However, if no such disclosure pattern currently exists for handlers (the design doc implies it does not: "currently shows type and message only"), T04 may be creating the pattern from scratch with no explicit visual spec.

**Recommendation:** Implementer should check whether a disclosure primitive exists in `global.css` before creating a new one.

### VAGUE-2: T06 "stale indicator" for WS disconnection

T06 says "Handle WS disconnection: show stale indicator on the panel (check `wsConnected` signal from app state)." No visual spec for what "stale indicator" looks like. The design doc does not define this either (it describes the behavior: "shows last known state with a stale indicator"). T06 also does not include a verify item for this edge case.

This aligns with an edge case in the design doc ("WebSocket disconnected while on diagnostics page") but the test coverage is not explicitly specified for this case.

**Recommendation:** Add a test for the WS-disconnected stale indicator to T06's test list or note this is tested in E2E only.

### VAGUE-3: T05 "overdue" job display not explicitly tested

The design doc edge case states: "A job cancelled between reads may appear active with a stale `next_run`. Accepted — millisecond window. The frontend must handle `next_run < now` on a non-cancelled job by showing 'overdue'."

T05 mentions this in the prompt ("Handle `next_run < now` on non-cancelled jobs by showing 'overdue'"), and cites the design doc Edge Cases section. However, no component test verifies the "overdue" rendering path. Only E2E tests are listed, which are higher-cost to run and less suited for testing conditional display states.

**Recommendation:** Add a unit test case: job with `next_run` in the past and `cancelled: false` renders "overdue" text.

## 7. Warnings

### WARN-1: Documentation update task missing

**Severity: LOW**

The design doc's Documentation Updates section states:
> "Add the new pages to the sidebar nav documentation in `design/context.md` Component Inventory section"

No task file lists this in its `implements` or prompt. T05 and T06 both add pages to the sidebar but neither one mentions updating `design/context.md`. Per the `design-completeness.md` rule: "These are not follow-ups — they ship in the same PR."

If the implementer does not notice this, the design/context.md inventory will be stale after the PR lands.

**Recommendation:** Add a step to T05 or T06 (or a brief standalone note) to update `design/context.md` Component Inventory with the new "handlers" and "diagnostics" pages.

### WARN-2: Schema regeneration step repeated in T01 and T02 but not reconciled for parallel execution

**Severity: LOW**

Both T01 and T02 end with:
> "After model changes, regenerate schemas: `uv run python scripts/export_schemas.py` then `cd frontend && npx openapi-typescript openapi.json -o src/api/generated-types.ts`"

T01 and T02 are parallel (no dependency between them). If both run concurrently (e.g., in an orchestration context), they will both try to regenerate schemas at the same time, potentially with only a partial set of model changes applied. T03 also triggers schema regeneration after its own model changes.

In practice, if tasks are executed sequentially (T01 → T02 → T03), the final regeneration at T03 completion captures all model changes. If T01 and T02 run in parallel, the intermediate regenerations are redundant and potentially misleading.

**Recommendation:** Clarify in context.md or the individual task files that the schema regeneration step at the end of T01/T02 is for local verification — the canonical regeneration happens after T03 (which finalizes all model changes). This avoids confusion in automated orchestration.

### WARN-3: ServiceStatusPanel visibility behavior change in T05 is a behavior regression risk

**Severity: LOW**

T05 Focus section states:
> "The `ServiceStatusPanel` where the dropped events line goes is at `service-status-panel.tsx` — it currently only renders when there are degraded services. The dropped events line should render ALWAYS (even when the panel would otherwise be empty/hidden)."

This is a behavior change to an existing component: the panel will now be visible even when all services are healthy. The design doc supports this (FR#18: "always displays a dropped telemetry events count"). However, T05's test list does not include a test for the panel rendering when no services are degraded (only the dropped events line rendering test is listed).

**Recommendation:** Add a test: "ServiceStatusPanel renders even when no services are degraded" to T05's test list.
