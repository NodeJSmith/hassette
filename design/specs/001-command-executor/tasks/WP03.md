   ```python
   async def _dispatch(self, topic: str, event: "Event[Any]", listener: "Listener") -> None:
       cmd = InvokeHandler(listener=listener, event=event, topic=topic, listener_id=listener.db_id or 0)
       await self._executor.execute(cmd)
       if listener.once:
           self.remove_listener(listener)
   ```

   - `add_listener()` remains synchronous â€” `Bus.on()` and all user app `on_initialize()` callers stay unchanged
   - Note: `listener.db_id` may be `None` for the first invocation if it fires before the registration task completes â€” the `or 0` sentinel in `InvokeHandler` handles this gracefully

## Review Guidance

- `_dispatch()` must delegate ALL exception handling to the executor â€” no try/except in `_dispatch()` itself (FAIL if any exception handling remains)
- `listener.once` removal must stay in `_dispatch()` â€” the executor does NOT own listener lifecycle (per prereq-03 decision)
- `listener.db_id` must be set BEFORE the listener enters the dispatch path â€” if `db_id` is `None` when `_dispatch()` fires, the DB FK is lost
- `get_all_listener_metrics()` returning `[]` is intentional for this WP â€” it is NOT a bug, it's a planned stub (confirm reviewer sees the `TODO(#267)` comment)
- `add_listener()` MUST remain synchronous â€” spawning registration as a background task is correct; making it async would cascade to `Bus.on()`, `DataSyncService`, and all ~20+ `on_initialize()` callers (FAIL if `add_listener` signature changes to async)
