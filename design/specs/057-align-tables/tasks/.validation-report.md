## Step 4: Problems Found

### Coverage Gaps

**GAP-1 (CRITICAL): FR#1 and AC#1 are only fully verified for apps/handlers via T03/T04/T05, but the log table search move is not verified.**

FR#1 requires "all three table pages" to have a search input above the table. T03 verifies the `TableCard` component exposes the search slot, and T04/T05 verify apps and handlers pages wire it up. However, T06 — the log table migration — does NOT include a Verify item confirming that the log table's search input has been moved above the table (from its current footer position to `TableCard`'s search slot). The task Prompt (item 3) and Focus mention it, but the Verify section only claims FR#3 and AC#3. The log table is the page that currently has search in the *wrong* location (footer), making it the highest-risk for regressions — yet it has no Verify item covering FR#1 / AC#1.

**GAP-2 (CRITICAL): FR#2 and AC#2 are not verified for the apps and log table pages — only handlers.**

FR#2 requires "all three table pages" to display a record count in the footer. AC#2 says the count is visible and updates. Only T05 (handlers) claims FR#2 and AC#2. T04 (apps) implements `TableFooter` with a count (Prompt item 3 describes `count={pluralize(filtered.length, "app")}`), but FR#2 and AC#2 are absent from T04's `implements` list and `Verify` section. T06 (logs) similarly wires up a count to `TableFooter` (Prompt item 3), but FR#2 and AC#2 are also absent from T06's `implements` and `Verify`. The "all three pages" requirement is only half-traced.

**GAP-3 (MODERATE): FR#4 and AC#4 have no Verify items confirming the apps and handlers pages actually render funnel icons.**

FR#4 says "all three table pages" use inline column filter popovers. T02 verifies that `SortHeader` renders a funnel icon when `filterContent` is provided — but this is a component-level test. T04 (apps) and T05 (handlers) both wire up `filterContent` in their Prompts, but neither task includes a Verify item that confirms a funnel icon actually appears on the page in the browser-level integration test. The apps/handlers Verify items focus on filter behavior (popover opens, selection filters table, dot appears) which implicitly requires the icon — but "at least one column header on each page has a clickable funnel icon" (AC#4) is not explicitly traced to T04 or T05. Consider adding AC#4 to T04 and T05's `implements` lists.

**GAP-4 (MODERATE): AC#7 mismatch — T01's Verify item for AC#7 covers focus management, not Escape/click-outside.**

The design doc's AC#7 states: "All filter popovers close on Escape or click-outside." T01's Verify item for AC#7 reads: "ColumnFilterPopover moves focus to first focusable child on open and restores focus to trigger on close." That is the correct Verify text for the focus management WCAG fix — not for AC#7. The actual Escape/click-outside close behavior is correctly covered by the FR#6 Verify item ("ColumnFilterPopover closes on Escape key press and click outside"), but T01 has misassigned this Verify text to AC#7 when it should describe AC#7's content. The gap: AC#7 is claimed as `implements` in T01 but the corresponding Verify item describes focus management (a sub-requirement), not the close-on-Escape/click-outside behavior that AC#7 defines. This is ambiguous — functionally the test will likely cover both, but the traceability label is wrong.

**GAP-5 (MODERATE): AC#8 Verify text does not match AC#8's definition.**

AC#8 in the design doc: "Active column filters show a dot indicator on the funnel icon." T01's Verify item for AC#8: "TableFooter displays a record count and auto-generates mobile filter panel from columnFilters map." This is the correct verification for what `TableFooter` does — but it does not verify AC#8 (dot indicator on funnel icon). The dot indicator behavior is actually tested in T02 ("Test hasActiveFilter shows the dot indicator" per T02 Prompt item 4), but AC#8 is not in T02's `implements` list, and T02 has no Verify item for AC#8. The dot indicator is tested implicitly in T05 (AC#6 Verify: "Active tier or app filters show a dot indicator on their respective funnel icons") — but AC#8 is not in T05's `implements` list either.

**Summary:** AC#8's Verify item in T01 describes `TableFooter` behavior (should be a separate AC or none), while the actual dot-indicator requirement goes unverified in any Verify list. T02 tests it in prose but doesn't claim it. This is a labeling gap with functional coverage risk.

**GAP-6 (MINOR): Dead code removal is not traced to any task.**

Architecture §12 lists several files to remove after migration: `tier-toolbar.tsx` (covered by T05), `column-filter.tsx` + `.module.css` (covered by T01 implicitly — moved to shared), `log-table-footer.tsx` + `.module.css` (covered by T06 item 8), `HeaderCell` (covered by T06 item 2), `FilterPills` (covered by T04 item 2), `ht-table--compact` redundant rule (covered by T01 item 5), and `ht-table-toolbar` styles (covered by T01 item 5). None of these dead code removals have an explicit `implements` claim or Verify item. Since the design doc's Key Constraints explicitly list them, there is no FR/AC to trace to — but there is also no verification that all listed dead code is actually removed. Consider adding a Verify checklist item to the relevant tasks enumerating deleted files.

### Contradictions

**CONTRADICTION-1 (MODERATE): T06 implements FR#3 twice — once in T04 and once in T06.**

FR#3 ("all three tables use same header background") is claimed by both T04 and T06. This is not a contradiction — it's correct multi-task coverage of a cross-cutting requirement. However, T05 (handlers) does NOT claim FR#3 in its `implements` list despite Architecture §10 and §7 both applying to the handlers table. The handlers table will also gain `var(--bg-sunken)` through `ht-table` adoption, but this is unverified. Add FR#3 / AC#3 to T05's `implements` and Verify sections, or add a note in T05 acknowledging it's inherited from the `ht-table` update in T01 (which does verify the CSS change in tables.css, though there's no Verify item for the `ht-table thead tr` background rule specifically).

**CONTRADICTION-2 (LOW): T04's AC#3 Verify text describes filter behavior, not background color.**

T04's Verify item for AC#3 reads: "Clicking the funnel on STATUS opens a popover; selecting 'failed' filters to only failed apps; the funnel shows an active dot." This describes AC#5's behavior (STATUS column filter works), not AC#3 (thead background is `var(--bg-sunken)`). The heading says "AC#3" but the content verifies the wrong acceptance criterion. AC#3 verification is separately captured in the FR#3 Verify item ("The apps table `<thead>` row has `var(--bg-sunken)` background"), so coverage is present — but the AC#3 label on the wrong Verify item is a labeling error that could confuse the implementer.

### Warnings (non-blocking)

**WARN-1:** T03 says "Do NOT update page callers yet" and notes apps/handlers will "temporarily break." No task explicitly coordinates landing T03+T04+T05 together. This is an implementation coordination concern, not a requirements gap, but worth noting for the developer.

**WARN-2:** T06's Verify section is the shortest of all tasks (only 2 items for FR#3 and AC#3) despite T06 being the most complex task (9 Prompt items, significant refactoring of log-table internals). Specifically absent from T06's Verify: search-moved-above-table (FR#1/AC#1 gap per GAP-1), filter panel in mobile footer auto-generated (FR#4/AC#4), record count in footer (FR#2/AC#2 gap per GAP-2), and zero-results empty state if applicable. T06 carries substantial risk with minimal verification coverage.

**WARN-3:** Architecture §13 specifies adding tests for `ColumnFilterPopover` focus management specifically. T01's Verify item for AC#7 describes this, but the specific Tab-trap test is only mentioned in T01's Prompt — not in a Verify checkbox. Tab trap behavior should be a binary Verify item.
