## Objectives & Success Criteria

A startup WARNING is emitted from `Hassette.on_initialize()` when either `scheduler_job_timeout_seconds` or `event_handler_timeout_seconds` is set to `None`. User-facing documentation is updated to describe the `timeout` and `timeout_disabled` parameters, both config fields, the `TimeoutError` swallowing limitation, and sync handler timeout behavior.

## Subtasks

1. Add startup WARNING check to `Hassette.on_initialize()` (or appropriate lifecycle hook after log infrastructure is running): emit one WARNING per disabled field when `scheduler_job_timeout_seconds is None` or `event_handler_timeout_seconds is None`
2. Update scheduler documentation (`docs/`) to describe the `timeout` and `timeout_disabled` parameters on all scheduling methods
3. Update bus documentation (`docs/`) to describe the `timeout` and `timeout_disabled` parameters on listener registration methods
4. Document `scheduler_job_timeout_seconds` and `event_handler_timeout_seconds` config fields in the configuration reference
5. Add a "Timeouts" section to the configuration reference documenting: default values (600s), per-item override semantics (`None` = global default, `float` = explicit, `timeout_disabled=True` = disable), `None` global opt-out, and the `TimeoutError` swallowing limitation

## Test Strategy

- **`tests/unit/core/test_hassette_timeout_warning.py`** (new):
  - `test_startup_warning_when_job_timeout_none` — WARNING logged for `scheduler_job_timeout_seconds=None`
  - `test_startup_warning_when_handler_timeout_none` — WARNING logged for `event_handler_timeout_seconds=None`
  - `test_no_warning_when_timeouts_configured` — no WARNING when both have values
  - `test_warning_fires_once_per_startup` — not repeated on subsequent lifecycle events

## Review Guidance

- Verify WARNING is emitted from `on_initialize()` (not from `HassetteConfig` model construction — it would fire before log infrastructure and spam on hot-reload)
- Verify WARNING message matches design: "execution timeout enforcement is disabled globally — framework components are unprotected"
- Verify documentation covers the `TimeoutError` swallowing limitation and sync handler behavior
- FAIL if WARNING is emitted from `model_post_init` or any config-level hook
