## Objectives & Success Criteria

The frontend renders `'timed_out'` execution status with an amber/warning badge, distinct from success (green) and error/cancelled (red). A centralized `executionStatusVariant()` helper in `status.ts` replaces the hardcoded binary ternaries in both `job-executions.tsx` and `handler-invocations.tsx`. The `APP_STATUS_MAP` includes `'timed_out'`.

## Subtasks

1. Add `executionStatusVariant(status: string): StatusVariant` function to `frontend/src/utils/status.ts` mapping: `"success"` → `"success"`, `"timed_out"` → `"warning"`, everything else → `"danger"`
2. Add `"timed_out"` entry to `APP_STATUS_MAP` in `frontend/src/utils/status.ts`
3. Replace the binary ternary in `frontend/src/components/app-detail/job-executions.tsx` (find by `status === "success" ? "success" : "danger"` pattern) with `executionStatusVariant(ex.status)`
4. Replace the binary ternary in `frontend/src/components/app-detail/handler-invocations.tsx` (find by same pattern) with `executionStatusVariant(inv.status)`

## Test Strategy

- N/A — frontend utility is trivial; visual verification covers correctness

## Review Guidance

- Verify `executionStatusVariant` handles unknown status values (fallback to `"danger"`)
- Verify both component files use the centralized helper (no remaining inline ternaries for execution status)
- Verify `"timed_out"` maps to `"warning"` (amber), not `"danger"` (red)
- FAIL if either component still has an inline `status === "success" ? ...` ternary
