## Objectives & Success Criteria

`ScheduledJob` gains `timeout: float | None` and `timeout_disabled: bool` fields. `matches()` includes `timeout`, `timeout_disabled`, and the existing `jitter` field (bug fix). All `Scheduler` public methods accept `timeout=` and `timeout_disabled=`. `SchedulerService` resolves the effective timeout and passes it through `ExecuteJob`. The `add_job()` error message distinguishes config-mismatch from name-collision. Config gains `scheduler_job_timeout_seconds: float | None = 600.0`.

## Subtasks

1. Add `timeout: float | None = None` and `timeout_disabled: bool = False` fields to `ScheduledJob` in `src/hassette/scheduler/classes.py`. Validate: `if isinstance(timeout, (int, float)) and timeout <= 0: raise ValueError("timeout must be a positive number")`
2. Add `timeout`, `timeout_disabled`, and `jitter` to the hand-written `matches()` method in `ScheduledJob`
3. Update `add_job()` error message in `src/hassette/scheduler/scheduler.py` to distinguish config-mismatch from name-collision: "A job named '...' already exists but its configuration has changed (changed fields: ...)"
4. Add `timeout: float | None = None` and `timeout_disabled: bool = False` parameters to `schedule()` and all convenience methods (`run_in()`, `run_once()`, `run_every()`, `run_daily()`, `run_minutely()`, `run_hourly()`, `run_cron()`) in `src/hassette/scheduler/scheduler.py`. Thread through to `ScheduledJob` construction in `schedule()`
5. Add `scheduler_job_timeout_seconds: float | None = Field(default=600.0)` to `HassetteConfig` in `src/hassette/config/config.py`
6. In `SchedulerService.run_job()`, resolve effective timeout: `timeout_disabled=True` → `None`; `timeout` is float → use it; `timeout` is `None` → config default. Pass as `effective_timeout` on `ExecuteJob` construction

## Test Strategy

- **`tests/unit/scheduler/test_scheduled_job_timeout.py`** (new):
  - `test_scheduled_job_timeout_field_default` — defaults to `None`
  - `test_scheduled_job_timeout_disabled_default` — defaults to `False`
  - `test_scheduled_job_timeout_validation_rejects_zero` — `timeout=0` raises `ValueError`
  - `test_scheduled_job_timeout_validation_rejects_negative` — `timeout=-1` raises `ValueError`
  - `test_matches_returns_false_when_timeout_differs` — timeout change detected
  - `test_matches_returns_false_when_jitter_differs` — jitter bug fix verified
  - `test_matches_returns_false_when_timeout_disabled_differs`
  - `test_matches_returns_true_when_all_fields_match` — including timeout/jitter
- **`tests/unit/scheduler/test_scheduler_timeout_threading.py`** (new):
  - `test_schedule_passes_timeout_to_job` — `scheduler.schedule(fn, trigger, timeout=5.0)` produces job with `timeout=5.0`
  - `test_run_in_passes_timeout` — convenience method threads through
  - `test_run_every_passes_timeout` — convenience method threads through
  - `test_run_daily_passes_timeout_disabled` — `timeout_disabled=True` threads through
- **`tests/unit/core/test_scheduler_service_timeout.py`** (new — uses real CommandExecutor):
  - `test_run_job_resolves_effective_timeout_from_job` — job.timeout=5 → effective_timeout=5
  - `test_run_job_resolves_effective_timeout_from_config` — job.timeout=None → uses config default
  - `test_run_job_resolves_timeout_disabled` — job.timeout_disabled=True → effective_timeout=None
  - `test_job_rescheduled_after_timeout` — job that times out is still rescheduled for its next occurrence (FR7, AC1)

## Review Guidance

- Verify `timeout` and `jitter` are both in `matches()` — the jitter exclusion was an existing bug
- Verify `add_job()` error message distinguishes config-mismatch from name-collision
- Verify effective timeout resolution order: `timeout_disabled` checked first, then `timeout`, then config
- Verify `effective_timeout` is passed to `ExecuteJob` construction (not `None` — the field is required)
- FAIL if `timeout_disabled` is missing from `matches()`
