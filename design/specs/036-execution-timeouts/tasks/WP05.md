## Objectives & Success Criteria

A new Alembic migration (005) adds `'timed_out'` to the SQLite CHECK constraints on `handler_invocations.status` and `job_executions.status`. All 8 aggregation query sites in `TelemetryQueryService` include `'timed_out'` as a distinct bucket. `ListenerSummary` and `JobSummary` Pydantic models gain a `timed_out: int` field. `HandlerInvocationRecord` and `JobExecutionRecord` status docstrings list `'timed_out'` as a valid value. Health metrics helpers account for `timed_out` in calculations.

## Subtasks

1. Create `src/hassette/migrations/versions/005_add_timed_out_status.py` following the table-recreation pattern from migration 004. Recreate both `handler_invocations` and `job_executions` tables with updated CHECK: `status IN ('success', 'error', 'cancelled', 'timed_out')`. Include `upgrade()` and `downgrade()` functions
2. Update status docstrings in `JobExecutionRecord` (`src/hassette/scheduler/classes.py:317`) and `HandlerInvocationRecord` (`src/hassette/bus/invocation_record.py:27`) to list `'timed_out'` as a valid value
3. Add `timed_out: int = 0` field to `ListenerSummary` and `JobSummary` in `src/hassette/core/telemetry_models.py`
4. Update all SQL CASE expressions in `src/hassette/core/telemetry_query_service.py` to count `'timed_out'` as a separate bucket (not merged into `error`). This includes `get_listener_summary`, `get_job_summary`, `get_all_app_summaries`, `get_global_summary`, `get_error_counts`, and `get_recent_errors`
5. Update `src/hassette/web/telemetry_helpers.py` to include `timed_out` count in health metrics calculations (error rate, health bar classification)
