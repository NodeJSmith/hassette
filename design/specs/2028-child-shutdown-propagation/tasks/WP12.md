## Objectives & Success Criteria

Add docstring to `Service.initialize()` documenting the STARTING-on-return behavior and the deadlock constraint. Add a test asserting Service status is STARTING after `initialize()` returns. The `_initializing` defense-in-depth reset with conditional warning (DEBUG vs WARNING based on `shutdown_event`) is already included in WP09's `_finalize_shutdown()` refactor.

After this WP:
- `Service.initialize()` has a clear docstring explaining the status asymmetry
- A test asserts `service.status == ResourceStatus.STARTING` after `await service.initialize()`
- The deadlock pattern (`child.on_initialize` calling `parent.wait_ready()` on a Service parent) is documented

## Subtasks

1. Add docstring to `Service.initialize()` in `src/hassette/resources/base.py` explaining: (a) status is STARTING on return, (b) handle_running() is called by _serve_wrapper(), (c) children MUST NOT call self.parent.wait_ready() during on_initialize
2. Add test `test_service_status_is_starting_after_initialize` in `tests/unit/resources/test_lifecycle_propagation.py`: create a StubService, call `await service.initialize()`, assert `service.status == ResourceStatus.STARTING`

## Test Strategy

Single test in `tests/unit/resources/test_lifecycle_propagation.py`:
- `test_service_status_is_starting_after_initialize`: uses existing `StubService` fixture, asserts status after `initialize()` returns

## Review Guidance

- Docstring MUST warn about the deadlock pattern — this is the primary documentation for a non-obvious constraint
- The test MUST use `ResourceStatus.STARTING`, not `ResourceStatus.RUNNING` — this is the whole point of the asymmetry documentation
- Verify that the docstring references `_serve_wrapper()` as the location where `handle_running()` is called
