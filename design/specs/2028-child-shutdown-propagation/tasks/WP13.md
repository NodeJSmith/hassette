## Objectives & Success Criteria

Add integration tests that exercise the full Hassette shutdown path with timeout enforcement, close_streams ordering, and the total timeout safety net. Verify the pre-existing flaky test is not worsened.

After this WP:
- Integration test verifies close_streams() ordering (children STOPPED before streams close)
- Integration test verifies total timeout caps wall-clock shutdown time
- Integration test verifies _force_terminal() recursion on timeout
- Existing `test_bus_driven_failed_events_trigger_shutdown_after_max_attempts` passes 10x with random ordering

## Subtasks

1. Add `test_hassette_shutdown_close_streams_ordering` in `tests/integration/test_lifecycle_propagation.py`: instrument `close_streams()` and children's `handle_stop()` to verify ordering (children's STOPPED events emitted before streams close)
2. Add `test_hassette_total_timeout_caps_shutdown` in `tests/integration/test_lifecycle_propagation.py`: create a hassette with a child that hangs in `on_shutdown`, verify total shutdown completes within `total_shutdown_timeout_seconds + 1s` tolerance
3. Add `test_force_terminal_on_total_timeout` in `tests/integration/test_lifecycle_propagation.py`: verify all children and grandchildren are in STOPPED state after total timeout fires
4. Run `test_bus_driven_failed_events_trigger_shutdown_after_max_attempts` 10x with random ordering to verify no regression: `uv run pytest tests/integration/test_service_watcher.py::test_bus_driven_failed_events_trigger_shutdown_after_max_attempts --count=10 -v`

## Test Strategy

Integration tests in `tests/integration/test_lifecycle_propagation.py`. These use the `HassetteHarness` fixture for real-component wiring. The flaky test check uses `pytest-repeat` with the existing test.

## Review Guidance

- Close_streams ordering test MUST verify temporal ordering (children STOPPED before close_streams), not just that both were called
- Total timeout test MUST use a real `asyncio.timeout` with a short config value (e.g., 2s) to avoid slow tests
- The flaky test 10x run is informational — if it fails, it's a pre-existing issue, not a regression from this PR. Document the result either way.
