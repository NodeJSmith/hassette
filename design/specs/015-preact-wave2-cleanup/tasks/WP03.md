## Objectives & Success Criteria

- API errors from start/stop/reload actions are displayed to the user (not silently swallowed)
- Error message clears when a new action is initiated
- All buttons disabled while any action is in flight
- Existing loading state and try/finally logic preserved

## Subtasks

1. In `frontend/src/components/apps/action-buttons.tsx`, add an `error` signal (`useRef(signal<string | null>(null)).current`).
2. Add a `catch` block to the existing `exec()` function (line 33-40): set `error.value` to the error message. Clear `error.value = null` at the start of `exec()` before `loading.value = true`.
3. Render the error message below the button group: `{error.value && <p class="ht-text-danger ht-text-sm">{error.value}</p>}`.
4. Ensure all buttons check `loading.value` for disabled state (already the case at lines 52, 63, 74).

## Test Strategy

- No unit tests — this is a UI-only change to a component that calls external APIs. The behavior is verified visually and via E2E tests.
- Existing E2E tests that exercise the app detail page must continue passing.
- Optional: unit test asserting `error.value` is set when `exec()` rejects and cleared when a new action starts.

## Review Guidance

- Verify `catch` block correctly extracts error message (handle both `Error` instances and string errors)
- Verify error clears on new action start
- Verify all buttons disabled during any action (not just the pressed one)
- BLOCKER if: errors are still silently swallowed, or loading state is broken

## Visual Verification

| Page | Setup | Verify |
|------|-------|--------|
| App Detail — action buttons | Navigate to an app detail page. The start/stop/reload buttons should be visible. | Buttons render correctly with no error message visible by default. |

## Pitfalls to Avoid

- Do not rebuild the existing loading/disable logic. Only add error display and catch block.
