## Objectives & Success Criteria

New nullable TEXT columns (`execution_id`, `trigger_context_id`, `trigger_origin`) exist in the DB. `persist_batch()` and `persist_batch_with_fk_fallback()` use shared parameter-building functions (no duplicated column strings). Query service returns the new fields. Pydantic telemetry models include the new fields. Integration tests verify the full persist-query roundtrip including the FK fallback path.

## Subtasks

1. Create migration `src/hassette/migrations/versions/007_add_execution_id_columns.py`:
   - `upgrade()`: `ALTER TABLE handler_invocations ADD COLUMN execution_id TEXT`, `ALTER TABLE handler_invocations ADD COLUMN trigger_context_id TEXT`, `ALTER TABLE handler_invocations ADD COLUMN trigger_origin TEXT`, `ALTER TABLE job_executions ADD COLUMN execution_id TEXT`.
   - `downgrade()`: `raise NotImplementedError("migration is not reversible — execution_id data would be lost")`.

3. Add `execution_id: str | None = None` to `JobExecution` in `src/hassette/core/telemetry_models.py`.

## Test Strategy

- `test_persist_and_query_handler_invocation_with_execution_id` — create an in-memory DB with all migrations applied. Assert all three fields match.
- `test_persist_and_query_job_execution_with_execution_id` — same pattern for `JobExecutionRecord` with `execution_id="def-789"`. Assert `execution_id` matches and trigger fields are `None`.
- `test_fk_fallback_preserves_execution_id` — insert a `HandlerInvocationRecord` with a non-existent `listener_id` FK. The FK fallback should null the FK but preserve `execution_id`, `trigger_context_id`, and `trigger_origin`. Query back and assert the three fields are intact.
- `test_null_execution_id_persists_and_queries` — persist a record with `execution_id=None`. Query back and assert `execution_id` is `None` (not empty string).
- `test_shared_params_match_persist_batch_columns` — assert that `_inv_insert_params()` returns the same keys as the column list used in `persist_batch()`'s INSERT statement (structural test to prevent future drift).

- Verify migration `down_revision = "006"` matches the actual last migration.
- The FK fallback test is critical — this was the highest-risk structural issue from the challenge review.
- Check that `_insert_row_with_fk_fallback()` does NOT null the new fields — only the FK field (`listener_id` or `job_id`) should be nulled on violation.
