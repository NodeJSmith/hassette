## 3. Findings

### Finding 1 — CONTRADICTION: T05's pre-commit wiring instruction misattributes the pattern to `check_internal_patches.py` (BLOCKING)

T05's Prompt: *"Wire it into `.pre-commit-config.yaml` as a local hook (same pattern as the existing `check_internal_patches` hook). Use the `pre-push` stage (matching the existing check hooks)."*

This is factually wrong. `tools/check_internal_patches.py` is **not** wired into `.pre-commit-config.yaml` — verified by grepping the file for `check_internal_patches`/`internal_patches` (no hits). It runs as a bare CI step instead: `.github/workflows/lint.yml:113` — `- name: Check internal MUT patches in core tests` → `run: uv run python tools/check_internal_patches.py`. Two sibling design docs in this same repo call out exactly this mistake: `design/specs/005-test-infra-dedup/design.md:122` ("Do not use `check_internal_patches.py` as the exemplar — it is CI-only, not pre-commit") and `:153` ("`check_internal_patches.py` is a CI-only lint … not a pre-commit hook — do not use it as the wiring exemplar").

T05 correctly cites `check_internal_patches.py`'s AST/`tokenize` **detection** approach (that part is accurate and worth reusing), but incorrectly attributes a pre-commit **wiring** pattern to it. The real pre-commit exemplars for a new `check_*.py` script are `check_module_boundaries.py` / `check_test_factories.py` (both use `language: system`, `pass_filenames: false`, `stages: [pre-commit, pre-push]`).

**Fix:** Correct T05 to cite `check_module_boundaries.py`/`check_test_factories.py` as the wiring exemplar for `.pre-commit-config.yaml`, or move the new check to `.github/workflows/lint.yml` if CI-only enforcement (matching `check_internal_patches.py`'s actual precedent) is preferred.

### Finding 2 — GAP: T05 is missing a dependency on T04 (BLOCKING)

T05's `depends_on: ["T02", "T03"]` omits `T04`. `check_coordinator_internal.py` scans `test_core.py`, `test_fatal_shutdown.py`, and `test_resource_deps.py` for any private-attribute access on `hassette_instance` (or `sm`) lacking a `# coordinator-internal` annotation. `hassette_instance._init_waves` (in `test_resource_deps.py`, 3 tests) and `hassette_instance._init_order` (in `test_core.py`'s `test_init_order_contains_all_children`) fit this exact shape — private attribute on the `hassette_instance` receiver — and FR#8's three annotation-worthy categories (fatal-shutdown-reason writes, loop state, SessionManager internals) explicitly do **not** include them. FR#6/AC#2 (owned entirely by T04) eliminate these accesses rather than annotate them.

If T05 executes (or its final verification — `uv run nox -s dev`, `prek -a`, and the new lint script itself) before T04 has migrated those two files, the lint will false-positive on the still-present, unannotated `_init_waves`/`_init_order` reads, since nothing else in the plan adds a `# coordinator-internal` comment to them and nothing removes them except T04. Nothing in the current dependency graph prevents T05 from running before or concurrently with T04 — both T04 and (transitively, via T02/T03) T05 only require T01.

*(This directly disagrees with the prior report's Finding 5, which concluded "no ordering hazard" on the reasoning that T04 "removes private access entirely rather than annotating it." That reasoning is backwards: it is precisely because T04 removes rather than annotates that the access is a lint violation for as long as T04 hasn't run yet.)*

**Fix:** Add `T04` to T05's `depends_on`.

### Finding 3 — GAP: T02 and T04 both modify `test_core.py` with no ordering edge between them (BLOCKING)

T02 (`depends_on: ["T01"]`) rewrites nearly all of `tests/integration/test_core.py` — every private-attr migration across its 22 tests, including the constructor test. T04 (`depends_on: ["T01"]`) separately modifies the same file to refactor `test_init_order_contains_all_children` and add a `topological_sort` import. Neither task depends on the other; both depend only on T01. Nothing prevents these from being dispatched as parallel workstreams against the same file, which is a real merge-conflict / lost-edit risk per this repo's own decomposition-discipline rule on shared mutable state (parallel writers to one file is not parallelism, it's a merge conflict).

**Fix:** Add `T02` to T04's `depends_on` (or the reverse) to force serialization on `test_core.py`.

### Finding 4 — GAP: T05 omits a unit test for the new lint script (should-fix, test co-location convention)

T05's Target Files are `create: tools/check_coordinator_internal.py`, `modify: .pre-commit-config.yaml`, `read: tools/check_internal_patches.py` — no test file. Every comparable `tools/check_*.py` script wired into pre-commit/CI in this repo has a co-located characterization test in `tests/unit/tools/`: `test_check_lazy_imports.py`, `test_check_llm_cruft.py`, `test_check_module_boundaries.py`, `test_check_spec_tokens.py`, `test_check_test_factories.py`, and — notably — `test_check_internal_patches.py`, the exact script T05 cites as its pattern exemplar. Shipping new lint tooling with no test breaks this repo's own Test Co-location convention.

**Fix:** Add `create: tests/unit/tools/test_check_coordinator_internal.py` to T05's Target Files and Prompt.

### Finding 5 — Minor: T05's own AC#3 Verify item implies a lint scope it doesn't have

T05 Verify lists: *"AC#3: Fixture teardown in conftest.py passes the lint (cleanup helper consolidates private access, fixture itself has none inline)."* T05's Prompt scopes `check_coordinator_internal.py` to exactly three files — `test_core.py`, `test_fatal_shutdown.py`, `test_resource_deps.py` — and never mentions `conftest.py`. The lint cannot "pass" on a file it never scans. AC#3 is genuinely satisfied structurally (T01 relocates the private access into a helper so the fixture body itself has none inline), but the Verify wording conflates "out of scope" with "passes," which could make an implementer wonder whether `conftest.py` should be added to `IN_SCOPE_FILES`.

**Fix:** Reword to something like "AC#3: fixture body contains no inline private-attribute access (verified by inspection, not by the lint — `conftest.py` is out of the lint's scope)."

### Finding 6 — Minor: T02's FR#4 constructor-test rewrite has an inconsistent/incomplete type list

`Hassette.wire_services()` registers 22 distinct child types via `add_child()` (confirmed by grep: `SyncExecutorService, EventStreamService, DatabaseService, LoggingService, CommandExecutor, BusService, SchedulerService, SessionManager, ServiceWatcher, WebsocketService, FileWatcherService, WebUiWatcherService, AppHandler, ApiResource, StateProxy, RuntimeQueryService, TelemetryQueryService, WebApiService, Bus, Scheduler, StateManager, Api`). T02's specified expected-type list has 20 entries: it adds `StateManager` (never checked by the original test at all) but still omits `StateProxy` and `SyncExecutorService` (also never checked by the original test). The `StateManager` addition is unexplained and asymmetric relative to the other two still-uncovered types.

Non-blocking — this doesn't regress existing coverage, since the original test also missed `StateProxy`/`SyncExecutorService` — but worth a one-line decision (add all three for full children coverage, or state why `StateManager` alone is added).
