## Warnings

1. **FR#11 is split across two tasks.** T01 claims `AC#10` (the FR#11 acceptance criterion) and handles the checker output strings and module docstrings. T09 claims `FR#11` directly and handles the test rename (`test_real_src_files_pass` → scope-agnostic name) and `CLAUDE.md` wording. This split is coherent — both pieces are covered, the Verify items in each task are non-overlapping, and neither task contradicts the other. No action required, but reviewers should confirm both T01 and T09 are treated as jointly necessary for FR#11 sign-off.

2. **T07 Verify section omits a pytest run assertion.** T04–T06 and T08 each include an explicit `uv run pytest ...` verification command in their Verify section prose; T07's second Verify bullet (`AC#7`) states only the checker-import zero-hit check and does not name a pytest command. The Focus section notes some integration tests are Docker-backed and may be impractical locally, so this is a deliberate omission, not an oversight — but the asymmetry could cause a reviewer to miss that the suite needs to be run (even partially). Not a coverage gap; no identifier is missing.

3. **T06 does not formally depend on T03.** T03 resolves lazy imports in `test_sync_executor_service.py`, `test_scheduler_coroutine_conversion.py`, and `test_forgotten_await_completeness.py`. T06 rewrites spec tokens in those same files. T06's `depends_on` lists only `T02`, not `T03`. The Focus note says "T03 also precedes T06 in ID order," implying the orchestrator will sequence them correctly, but this is not enforced in the dependency graph. If T06 runs before T03 completes, both tasks modify import lines and docstrings in the same files simultaneously. Low risk given the orchestrator runs one task at a time, but making `T03` an explicit dependency of `T06` would eliminate the ambiguity.
