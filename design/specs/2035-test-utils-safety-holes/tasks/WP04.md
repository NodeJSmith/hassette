## Objectives & Success Criteria

After this WP:

- A new `DrainError` exception class exists in `src/hassette/test_utils/exceptions.py` that aggregates handler task exceptions surfaced during drain.
- `AppTestHarness._drain_task_bucket` is replaced with an iterative implementation that:
  - Loops between waiting for bus dispatch idle and waiting for app task_bucket pending tasks.
  - Uses `await_dispatch_idle` for the final stability check (inheriting the 5ms anyio window).
  - Raises a rich `TimeoutError` via `_raise_drain_timeout` when the deadline elapses, including pending task names and a debounce hint if applicable.
  - Wraps `await_dispatch_idle` in `try/except TimeoutError` so its raw timeout is translated into the rich diagnostic.
  - Has a top-of-loop deadline guard that prevents infinite spin on perpetually-spawning handlers.
  - Collects non-cancellation exceptions from completed handler tasks and raises `DrainError` before returning.
- The drain uses the public accessors from WP03 (`app.task_bucket.pending_tasks()`, `bus_service.is_dispatch_idle`, `bus_service.dispatch_pending_count`) — no direct private-attribute access.
- New tests exercise each of the iterative drain's correctness guarantees: depth-2 chain, depth-3 chain, perpetual-spawn timeout, handler exception surfacing via `DrainError`, debounce timeout diagnostic, bus-dispatch timeout wrapping.
- `DrainError` is importable from `hassette.test_utils` (added to `__all__` in `hassette/test_utils/__init__.py`).
- `uv run pytest tests/unit/test_drain_iterative.py tests/integration/test_app_test_harness.py tests/integration/test_app_harness_simulation.py -v -n 2` passes.

## Subtasks

1. Create `src/hassette/test_utils/exceptions.py` containing the `DrainError` class:
   ```python
   class DrainError(Exception):
       """Raised when AppTestHarness drain surfaces handler task exceptions.

       Aggregates all non-cancellation exceptions from completed tasks during drain
       so test failures report the real cause instead of silently masking handler
       crashes with misleading assertion failures.
       """

       task_exceptions: list[tuple[str, BaseException]]

       def __init__(self, task_exceptions: list[tuple[str, BaseException]]) -> None:
           self.task_exceptions = task_exceptions
           count = len(task_exceptions)
           first_name, first_exc = task_exceptions[0]
           parts = [
               f"{count} handler task exception{'s' if count != 1 else ''} during drain.",
               f"First: {first_name}: {type(first_exc).__name__}: {first_exc}",
           ]
           if count > 1:
               parts.append(f"({count - 1} more — see .task_exceptions)")
           super().__init__(" ".join(parts))
   ```
2. Export `DrainError` from `src/hassette/test_utils/__init__.py` — add to the Tier 1 imports and to `__all__` alphabetically.
3. In `src/hassette/test_utils/app_harness.py`, add a module-level import for `DrainError` from `hassette.test_utils.exceptions`.
4. Replace the `_drain_task_bucket` method body with the iterative implementation from the design doc. Use `app.task_bucket.pending_tasks()`, `bus_service.is_dispatch_idle`, and `bus_service.dispatch_pending_count` — not the private fields.
5. Add the `_raise_drain_timeout` helper method on `AppTestHarness` (also from the design doc). This method builds a diagnostic `TimeoutError` with pending task names and the debounce hint.
6. Ensure the drain body:
   - Has a top-of-loop `if asyncio.get_running_loop().time() >= deadline: self._raise_drain_timeout(...)` guard.
   - Wraps each `await bus_service.await_dispatch_idle(timeout=...)` call in `try/except TimeoutError: self._raise_drain_timeout(...)` to translate the raw BusService timeout.
   - Calls `await_dispatch_idle` again as the Step-3 stability check (do not use `asyncio.sleep(0)`).
   - Collects exceptions: after `asyncio.wait(pending, ...)`, iterate `done` tasks, skip cancelled ones, call `.exception()`, append to `collected_exceptions` if non-None.
   - At the exit condition (both sides quiescent), raises `DrainError(collected_exceptions)` if any were collected before returning.

## Test Strategy

### New file: `tests/unit/test_drain_iterative.py`

- `test_drain_waits_for_depth_1_task_chain` — a handler that `await`s completion normally. Drain returns cleanly.
- `test_drain_waits_for_depth_2_task_chain` — a handler that spawns a task via `task_bucket.spawn(...)`; the spawned task calls `api.turn_on(...)`. Drain returns, then `assert_called("turn_on", ...)` succeeds. Without the iterative drain, this test would fail (the drain would return before the spawned task runs).
- `test_drain_waits_for_depth_3_task_chain` — handler spawns task A, which spawns task B, which calls `api.turn_on(...)`. Same assertion.
- `test_drain_times_out_on_perpetually_spawning_handler` — handler that repeatedly spawns short-lived tasks via a generator pattern. Assert `TimeoutError` is raised and the message includes pending task names.
- `test_drain_surfaces_handler_exception_as_drainerror` — handler raises `ValueError("boom")`. After `simulate_state_change(...)`, `DrainError` is raised (not masked as an `AssertionError`). Assert `err.task_exceptions[0][1]` is a `ValueError` with message "boom".
- `test_drain_surfaces_multiple_handler_exceptions` — two handlers, both raise. `DrainError.task_exceptions` has length 2.
- `test_drain_timeout_message_includes_pending_task_names` — perpetually-spawning handler with task names containing "my_handler"; assert the timeout message contains "my_handler".
- `test_drain_timeout_message_has_debounce_hint_when_applicable` — a debounce-wrapped handler with `debounce=5.0` called via `simulate_state_change(..., timeout=0.5)`. Assert `TimeoutError` and message contains "debounce".
- `test_drain_wraps_await_dispatch_idle_timeout` — set up a scenario where `await_dispatch_idle` would time out first (e.g., stuck dispatch queue); assert the drain surfaces it through `_raise_drain_timeout`, not raw `BusService` message.
- `test_drain_uses_public_accessors_not_private_attributes` — introspect the drain method source via `inspect.getsource` and assert it does NOT contain `_dispatch_pending` or `_dispatch_idle_event` or `app.task_bucket._tasks`. (Optional — captures regressions against the design's encapsulation requirement.)
- `test_drain_error_message_single_exception` — assert `str(DrainError([("task", ValueError("x"))]))` contains "1 handler task exception", "task", "ValueError", "x".
- `test_drain_error_message_multiple_exceptions` — assert "2 handler task exceptions" and "1 more — see .task_exceptions".

## Review Guidance

- **Blocker**: Direct access to `bus_service._dispatch_pending`, `bus_service._dispatch_idle_event`, or `app.task_bucket._tasks` in the drain code. Use the public accessors from WP03.
- **Blocker**: Missing top-of-loop deadline check — a handler that perpetually spawns short-lived tasks must hit the timeout and raise, not spin indefinitely.
- **Blocker**: `await_dispatch_idle` timeouts NOT wrapped in `try/except` — the raw `BusService` error message bypasses the rich diagnostic and is a regression.
- **Blocker**: Step 3 using `asyncio.sleep(0)` instead of `await_dispatch_idle` — the 5ms anyio stability window is load-bearing.
- **Blocker**: Handler exceptions not being collected and raised as `DrainError` — silent false-greens were the original motivation for this finding.
- **Blocker**: `DrainError` missing from `hassette.test_utils.__all__`.
- **Warning**: `DrainError` raised even when no exceptions occurred (empty list) — the guard `if collected_exceptions:` must be present.
- **Warning**: `_raise_drain_timeout` not called consistently at every deadline check — one missed call means the raw `TimeoutError` leaks.
- **Warning**: Tests that depended on silent handler failures (e.g., a test that asserted `assert_call_count(...)` expecting handlers to have failed silently) will now fail with `DrainError`; confirm these tests are either correct under the new contract or are testing something outside this feature's scope.
- Ensure that the drain contract remains backwards-compatible for the happy path: tests with handlers that run successfully and spawn no fire-and-forget work should behave identically to before.
