## Warnings

The following items are notable but do not block approval:

1. **T02 claims FR#7 and AC#8 without test dependency on T01** — FR#7/AC#8 require a working `bus.emit` (added in T01) to pass. T02 has `depends_on: ["T01"]`, so ordering is correct, but the `implements` claim in T02 for FR#7 implicitly relies on T01's work. This is fine structurally; just be aware the self-delivery test in T02 cannot pass in isolation without T01.

2. **AC#5 (Pyright) verified only via running Pyright, not a test file** — The design doc explicitly notes "no new test file needed; type checking catches this." T01's Verify section confirms Pyright must be run. This is intentional and acceptable.

3. **T01 test coverage for sync emit** — T01's Prompt instructs "Test that `bus.sync.emit("test.topic", data)` delivers from a sync context (use `task_bucket.run_sync` in the test or a sync fixture)." The implementation detail (`task_bucket.run_sync` in the test) is an unusual pattern for sync facade testing. The implementer should check existing sync facade tests for the correct fixture pattern before writing this test.

4. **context.md "Visual Artifacts" section is "None"** — There are genuinely no visual artifacts for this backend/API change. The empty value is correct, not a gap.

5. **T03 Verify checklist items are fewer than the task scope** — T03's Prompt covers four documentation changes (apps/index.md, snippet, migration page, utilities.md), but the Verify section only has two checklist items (AC#6 and AC#7). The utilities.md disambiguation note (part of AC#6's spirit) and the snippet Pyright check are covered implicitly by the `mkdocs build --strict` verification step in Prompt item 5. This is acceptable but the implementer should confirm Pyright runs on the snippet during `mkdocs build` or run it separately.
