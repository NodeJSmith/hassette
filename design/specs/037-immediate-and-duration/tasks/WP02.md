## Objectives & Success Criteria

`DurationTimer` class exists at `src/hassette/bus/duration_timer.py`, follows the RateLimiter pattern, manages a single asyncio.Task, holds a cancellation Subscription reference, supports start/cancel/is_active, and is wired into `Listener.cancel()`. Unit tests pass verifying timer lifecycle and idempotent cancellation.

## Subtasks

1. Create `src/hassette/bus/duration_timer.py` with `DurationTimer` class:
   - Constructor accepts `task_bucket: TaskBucket`, `duration: float`, `predicates` (for re-evaluation), `entity_id: str`, `owner_id`, and a `create_cancel_sub` callback (a dependency-injected factory — the concrete implementation is provided by `BusService._create_cancel_listener()` in WP04 during integration wiring; for unit tests in this WP, use a mock)
   - `start(triggering_event, on_fire)` — cancels existing task, re-creates cancellation subscription via `create_cancel_sub()` callback if consumed/None, spawns new delayed task via `task_bucket.spawn()`
   - `cancel()` — sets `self._cancelled = True` first (idempotency guard), cancels asyncio task, removes cancellation listener from Router directly (synchronous), clears references
   - `is_active` property — True when a timer task is pending
   - `_on_cancel_event(event)` — the cancellation handler: re-evaluates predicates against new event; if predicates no longer match → cancel the timer; if still match → do nothing
   - Must NOT import from `listeners.py` at runtime — use `TYPE_CHECKING` guard
2. Wire `DurationTimer` into `Listener`:
   - Add `_duration_timer: DurationTimer | None` as a non-init field on `Listener` (same pattern as `_rate_limiter`)
   - In `Listener.create()`, construct `DurationTimer` when `duration is not None`
   - In `Listener.cancel()`, call `self._duration_timer.cancel()` if set (alongside `_rate_limiter.cancel()`)
3. Ensure `DurationTimer.cancel()` uses the `_cancelled` flag to be idempotent — multiple calls are safe and produce no errors

## Test Strategy

Unit tests in `tests/unit/bus/test_duration_timer.py` (new file):
- `test_start_spawns_task_and_fires_after_delay` — start() with 0.05s delay fires the on_fire callback
- `test_cancel_prevents_fire` — cancel() before delay elapses prevents on_fire
- `test_cancel_is_idempotent` — calling cancel() twice does not raise
- `test_start_cancels_previous_task` — calling start() twice cancels the first timer
- `test_start_recreates_cancel_subscription` — after cancel() consumed the sub, start() creates a new one
- `test_is_active_reflects_pending_task` — True after start, False after cancel or fire
- `test_on_cancel_event_matching_does_not_cancel` — event that still matches predicates does not cancel timer
- `test_on_cancel_event_non_matching_cancels` — event that fails predicates cancels timer
- `test_cancel_removes_cancellation_listener_synchronously` — verify Router.remove is called directly, not via task_bucket

## Review Guidance

- Verify no runtime imports from `listeners.py` — only `TYPE_CHECKING` guard
- Verify idempotency guard is the FIRST operation in `cancel()` (matches `RateLimiter.cancel()` pattern)
- Verify cancellation listener removal is synchronous (no `task_bucket.spawn()`) to avoid shutdown-ordering issues
- Verify `start()` re-creates the cancellation subscription rather than reusing a potentially consumed one
- Verify the cancellation handler re-evaluates predicates rather than blindly cancelling
