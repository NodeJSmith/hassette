## Step 4: Issues Found

### ISSUE-1 (LOW): BusService method docstrings not assigned to any task

**Design doc requirement** (`## Documentation Updates`):
> **`BusService` method docstrings**: Update all changed methods to reflect sync routing.

No task file's Prompt, Focus, or Verify section mentions updating BusService method docstrings. T02 covers the code restructuring of `add_listener`, `remove_listener`, `_remove_listener_by_id`, `remove_listeners_by_owner`, `get_listeners_by_owner`, and `_create_cancel_listener`, but does not instruct the implementer to update the docstrings on those methods.

**Affected task:** T02 (most natural home, since it restructures those methods)

**Recommended fix:** Add a step to T02's Prompt — "Update the docstring on each restructured method (`add_listener`, `remove_listener`, `_remove_listener_by_id`, `remove_listeners_by_owner`, `get_listeners_by_owner`, `_create_cancel_listener`, `dispatch`) to remove async language and describe the new sync routing / async DB split." Add a Verify checkbox: "BusService method docstrings for all restructured methods reflect sync routing (no references to task spawning for routing)."

---

### ISSUE-2 (LOW): AC#1 appears in T04 body text but not T04 frontmatter `implements`

T04's body references AC#1 ("the race they tested is eliminated by construction and covered by new AC#1 ordering test in T05"), but T04's frontmatter `implements` list is `["AC#9"]` only. This is semantically correct — T04 does not implement AC#1, it merely cross-references it — but it creates a minor traceability ambiguity: a grep of task `implements` fields for AC#1 finds only T05, while a full-text grep finds T04 as well.

This is a documentation clarity issue, not a coverage gap. No fix required unless the traceability tooling does full-text matching instead of frontmatter-only matching.

---

### ISSUE-3 (LOW): AC#6 implemented in both T03 and T05 with no explicit split of responsibility

T03's frontmatter claims `implements: ["FR#6", "AC#6"]` and its Verify checks the `Subscription.registration_task` docstring. T05's body also describes a behavioral test for AC#6 (`registration_task` resolves on DB failure) but T05's frontmatter only claims `["AC#1", "AC#2", "AC#3", "AC#4"]` — omitting AC#6.

The split is actually coherent (T03 = docstring/contract; T05 = behavioral test), but T05's frontmatter underreports its AC#6 contribution. Either:
- Add AC#6 to T05's `implements` list to make the split explicit, or
- Accept that T03's Verify fully covers AC#6 and T05's test is additive.

Not a coverage gap. Low-priority clarification.

### ISSUE-4 (NOTE): T04 and T05 are parallel siblings (both depend only on T03) but T05 writes new tests while T04 deletes old ones in `test_bus.py`

T04 deletes `test_cancel_before_add_task_completes_*` from `tests/integration/test_bus.py` (lines 749-842). T05 creates new files (`tests/unit/bus/test_bus_ordering.py`, `tests/unit/bus/test_bus_contract.py`). The files touched are disjoint — no merge conflict risk. T05's Verify (AC#1 ordering test) is the semantic replacement for what T04 deletes, and the dependency ordering correctly puts both after T03. This is informational — no action needed.
