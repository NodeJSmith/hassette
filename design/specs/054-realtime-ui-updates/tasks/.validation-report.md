## Warnings (non-blocking)

### W1 — T04 depends on T01 but T03 does not; ordering risk for `row_id` fixture updates

T04 instructs updating `overview-tab.test.tsx` fixtures to include `row_id` (step in T04 Focus). T01 also instructs updating those same fixtures (T01 Prompt section, "Frontend test fixture updates"). Both tasks reference the same files (`overview-tab.test.tsx`, `frontend/src/test/handlers.ts`). If T01 and T03 are executed concurrently (both have `depends_on: []`), T03's edits to `overview-tab.tsx` may conflict with T04's fixture updates at the same lines. The `depends_on` graph correctly requires T04 to run after T01, but T03 is independent — confirm implementers do not execute T03 concurrently with T01 on the same `overview-tab.tsx` lines.

### W2 — AC#7 quantitative bound stated as "at most 3" in AC but "at most 3-4" in T02 Verify

The design doc states AC#7 as "at most 3 API refetch calls" for 50 firings in 5 seconds. T02's Verify item says "at most 3-4 refetch calls." With `delayMs=500` and `maxWaitMs=1500`, the theoretical maximum is `ceil(5000 / 1500) ≈ 4`. The design doc's "3" is likely a rounded expectation. The implementer should confirm the exact bound from the math and update either the AC text or the Verify item so they agree. This is a documentation inconsistency, not a functional gap.

### W3 — Architecture section numbering in design.md is non-sequential

The design doc has sections numbered "1, 2, 3, 5, 4" (sections 4 and 5 are out of order). This does not affect task coverage but could cause confusion when cross-referencing. No action required for implementation; flagged for awareness.
