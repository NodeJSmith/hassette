## Objectives & Success Criteria

Extend `FailedMessageError` (`src/hassette/exceptions.py`) so it stores HA's error `code` and `original_data` as instance attributes, enabling programmatic error handling via `except FailedMessageError as e: if e.code == "...": ...`. Update the WebSocket service to forward the `code` field from HA's error envelope. This WP is a prerequisite for WP04's `_ws_helper_call` wrapper — without it, the wrapper cannot preserve structured error data through its chain.

Concrete outcomes:

- `FailedMessageError.__init__(self, msg, *, code, original_data)` exists with both kwargs defaulting to `None`.
- `self.code` and `self.original_data` are instance attributes (readable post-construction).
- `FailedMessageError.from_error_response` signature accepts `code: str | None = None` alongside existing `error` and `original_data` kwargs.
- The `from_error_response` method body calls `return cls(msg, code=code, original_data=original_data)` — NOT `return cls(msg)`. This is the single most likely implementation mistake; the unit test below is the gate.
- The typo in `from_error_response` is fixed: `"WebSocket message for failed with response"` → `"WebSocket message failed with response"`.
- `src/hassette/core/websocket_service.py` at the `_respond_if_necessary` error path forwards HA's error envelope `code` field: `FailedMessageError.from_error_response(error=..., code=message.get("error", {}).get("code"), original_data=message)`. If WP01 documented HA using a different error envelope structure, use the documented field name instead.
- Existing positional-only callers of `FailedMessageError(msg)` continue to work (backward compatibility).
- `tests/unit/test_exceptions.py` exists (new file) with tests covering: backward compat, kwargs construction, `from_error_response` forwarding, and the end-to-end path from `websocket_service.py` to a caught exception with populated attributes.
- `uv run pytest tests/unit/test_exceptions.py` passes.
- `uv run pyright src/hassette/exceptions.py src/hassette/core/websocket_service.py` passes.
- Existing tests that exercise `FailedMessageError` continue to pass (check `tests/integration/test_api.py` and any test mocking WebSocket errors).

## Subtasks

1. Read `src/hassette/exceptions.py` lines 52–62 (current `FailedMessageError` implementation) to confirm the class has no `__init__` and the classmethod message has the typo.
2. Add `def __init__(self, msg: str, *, code: str | None = None, original_data: dict | None = None) -> None:` to `FailedMessageError`. Body: `super().__init__(msg); self.code = code; self.original_data = original_data`.
3. Update the `from_error_response` classmethod signature to accept `code: str | None = None` alongside the existing `error` and `original_data` parameters (order: `error`, `code`, `original_data`).
4. Fix the message typo: change `"WebSocket message for failed with response"` to `"WebSocket message failed with response"` in the `from_error_response` body.
5. **Critical**: update the `from_error_response` body's return statement from `return cls(msg)` to `return cls(msg, code=code, original_data=original_data)`. Without this change, the extension is useless and callers will see `e.code is None` forever.
6. Read `src/hassette/core/websocket_service.py:293` (or wherever `FailedMessageError.from_error_response` is called) and update the call site to forward `code=message.get("error", {}).get("code")`. If WP01's HA source verification documented a different error envelope shape, use that instead. Add a brief comment referencing WP01's `## HA WebSocket Commands` section.
7. Create `tests/unit/test_exceptions.py` (new file) with a module docstring and imports for `FailedMessageError` and `pytest`.
8. Add test `test_failed_message_error_backward_compat_positional_only()`: `e = FailedMessageError("boom"); assert str(e) == "boom"; assert e.code is None; assert e.original_data is None`. Confirms backward compatibility for callers using positional-only construction.
9. Add test `test_failed_message_error_stores_kwargs()`: `e = FailedMessageError("boom", code="name_in_use", original_data={"source": "test"}); assert e.code == "name_in_use"; assert e.original_data == {"source": "test"}`.
10. Add test `test_failed_message_error_from_error_response_forwards_all_fields()`: `e = FailedMessageError.from_error_response(error="already exists", code="name_in_use", original_data={"type": "input_boolean/create"}); assert e.code == "name_in_use"; assert e.original_data == {"type": "input_boolean/create"}; assert "WebSocket message failed" in str(e); assert "for failed" not in str(e)`. The last assertion is the typo-fix gate.
11. Add test `test_failed_message_error_from_error_response_defaults_code_to_none()`: `e = FailedMessageError.from_error_response(error="x"); assert e.code is None`. Ensures the new `code` parameter is optional.
12. Add test `test_failed_message_error_chain_preserves_original()`: simulate a caught `FailedMessageError` being chained through a wrapper — `try: raise FailedMessageError("first", code="a") except FailedMessageError as e: raise FailedMessageError("wrapped", code=e.code, original_data=e.original_data) from e`. Catch the wrapped exception and assert `wrapped.code == "a"` and `wrapped.__cause__.code == "a"`. This is a regression test for WP04's `_ws_helper_call` pattern.
13. Run `uv run pytest tests/unit/test_exceptions.py -n 2` and fix any failures. If the `from_error_response` return statement was not updated in step 5, the forwarding test will fail — this is the implementer gate.
14. Run `uv run pytest tests/integration/test_api.py -n 2` to ensure existing WebSocket-using tests continue to pass — no test mocking `FailedMessageError` with positional-only construction should have broken.
15. Run `uv run pyright src/hassette/exceptions.py src/hassette/core/websocket_service.py tests/unit/test_exceptions.py` and fix any type errors.

## Review Guidance

**Code reviewer must verify:**

- `__init__` uses `*` to make `code` and `original_data` keyword-only (prevents accidental positional misuse).
- `from_error_response` body's return statement is `return cls(msg, code=code, original_data=original_data)` — **not** `return cls(msg)`. This is the single most likely implementation mistake. Look at the actual source, not the test output — a reviewer who only checks `pytest` passed can be fooled by a test that forgot to assert on `e.code`.
- The typo fix ("for failed" → "failed") is present.
- The `websocket_service.py` call site forwards `code` from HA's error envelope.
- No backward compat breaks — grep for all existing `FailedMessageError(` and `FailedMessageError.from_error_response(` call sites and verify each continues to work.
- `test_failed_message_error_from_error_response_forwards_all_fields` is not trivially-passing (i.e., it actually asserts `e.code == "name_in_use"`, not just that the exception was raised).

**Integration reviewer must verify:**

- No new imports of `FailedMessageError` broken in this WP — grep for `from hassette.exceptions import` and verify all expected callers still compile.
- The new `tests/unit/test_exceptions.py` file follows existing test module conventions (module docstring, pytest fixture discipline, naming).

**Blocker-level deviations:**

- `from_error_response` body's return statement not updated → BLOCKER. Every subsequent WP depending on `e.code` will silently malfunction.
- Backward compat broken (`FailedMessageError("msg")` no longer works) → BLOCKER. Affects all existing call sites.
