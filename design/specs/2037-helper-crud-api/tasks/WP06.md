## Objectives & Success Criteria

Add a new test that compares `ApiProtocol`'s declared public async methods against `Api`'s public async methods, ensuring `ApiProtocol` stays in sync with `Api` over time. Without this test, future `Api` method additions can silently skip `ApiProtocol` updates, and the `_: ApiProtocol = cast("ApiProtocol", RecordingApi)` assertion at `recording_api.py:522` provides false confidence (it's explicitly documented as a Pyright no-op).

The test uses the `vars(cls) + inspect.iscoroutinefunction` pattern — **NOT `ApiProtocol.__annotations__`**. Protocol method declarations do not populate `__annotations__`, so a test built on annotation inspection would always pass vacuously (round-2 finding F4).

Concrete outcomes:

- `tests/unit/test_recording_api_protocol_parity.py` exists (new file).
- The test iterates `vars(ApiProtocol)` filtering for public async methods and compares against the same set on `Api`.
- When `Api` has a public async method missing from `ApiProtocol`, the test fails with a helpful message naming the missing methods.
- The test passes in its current state (because WP05 added all 35 new method signatures to `ApiProtocol`).
- The test is not trivially-passing (it would fail if the `ApiProtocol` declarations were removed).
- `uv run pytest tests/unit/test_recording_api_protocol_parity.py -n 2` passes.
- `uv run pyright tests/unit/test_recording_api_protocol_parity.py` passes.

## Subtasks

1. Read `tests/unit/test_recording_api_write_parity.py` lines 65–76 to understand the existing `_public_async_methods` helper pattern (uses `vars(cls)` + `inspect.iscoroutinefunction`).
2. Create `tests/unit/test_recording_api_protocol_parity.py` with a module docstring explaining the purpose: "Parity test: `ApiProtocol` must declare every public async method that `Api` exposes, so the `_: ApiProtocol = cast(...)` assertion at `recording_api.py:522` remains structurally valid and `RecordingApi` callers get accurate Pyright coverage."
3. Import `inspect`, `Api` from `hassette.api.api`, and `ApiProtocol` from `hassette.test_utils.recording_api`.
4. Define a module-level helper `_public_async_methods(cls: type) -> set[str]` that returns the same set as the write-parity test's version:
   ```python
   def _public_async_methods(cls: type) -> set[str]:
       return {
           name for name, member in vars(cls).items()
           if not name.startswith("_") and inspect.iscoroutinefunction(member)
       }
   ```
5. Define `test_api_protocol_matches_api_methods()` that:
   - Collects `api_methods = _public_async_methods(Api)` and `protocol_methods = _public_async_methods(ApiProtocol)`.
   - Computes `missing_from_protocol = api_methods - protocol_methods`.
   - Asserts `not missing_from_protocol`, with an error message listing the missing method names and pointing at `recording_api.py`.
6. Add a sanity-check test `test_protocol_not_vacuous()`: assert `len(_public_async_methods(ApiProtocol)) > 0`. This catches the case where someone accidentally refactors `ApiProtocol` in a way that makes `vars()` return empty — without this, the parity test would pass vacuously.
7. Run `uv run pytest tests/unit/test_recording_api_protocol_parity.py -n 2` — both tests must pass (assuming WP05's `ApiProtocol` updates are in place).
8. **Negative test**: temporarily comment out one of the 35 new method declarations in `ApiProtocol` (e.g., `async def list_input_booleans(self) -> list[InputBooleanRecord]: ...`). Re-run the test and verify it fails with the expected "missing from ApiProtocol" message naming that method. Revert the comment before committing.
9. Run `uv run pyright tests/unit/test_recording_api_protocol_parity.py` and fix any type errors.
10. Verify that the test does NOT use `ApiProtocol.__annotations__` — this would return `{}` for Protocol method declarations and make the test trivially-passing (round-2 finding F4).

## Test Strategy

**New file: `tests/unit/test_recording_api_protocol_parity.py`** with 2 tests:

1. `test_api_protocol_matches_api_methods` — the main parity assertion. Uses `vars(cls)` + `inspect.iscoroutinefunction` (NOT `__annotations__`). Fails if `Api` has a public async method missing from `ApiProtocol`.
2. `test_protocol_not_vacuous` — sanity check ensuring `_public_async_methods(ApiProtocol)` returns a non-empty set. Without this, the main test could silently pass if Protocol method inspection broke in a future refactor.

**Negative verification gate** (one-time, during WP development): manually comment out one `ApiProtocol` entry, run the test, confirm it fails with a helpful message, then revert. Document this in the WP notes as "manual negative test completed on <date>".

Run command: `uv run pytest tests/unit/test_recording_api_protocol_parity.py -n 2`.

## Review Guidance

**Code reviewer must verify:**

- The test uses `vars(cls)` + `inspect.iscoroutinefunction`, **NOT** `ApiProtocol.__annotations__`. The latter returns `{}` for Protocol method declarations and makes the test vacuously pass (round-2 finding F4 is explicit about this).
- The `test_protocol_not_vacuous` guard exists and checks `len(_public_async_methods(ApiProtocol)) > 0`. Without it, a future `ApiProtocol` refactor could silently break the main test.
- The assertion error message in `test_api_protocol_matches_api_methods` names the missing methods (not just "parity failed") and directs the reader to `recording_api.py` for the fix.
- The test does not import anything from `RecordingApi` — only `Api` and `ApiProtocol`. This keeps the test focused on the Protocol drift problem.

**Integration reviewer must verify:**

- The test file follows the naming and structure of `test_recording_api_write_parity.py` (the pre-existing parity test it's modeled on).
- No test ordering coupling — this test runs independently of other recording API tests.
- The `_public_async_methods` helper is defined locally in this file, not imported from `test_recording_api_write_parity.py`. Importing from another test module is a smell; a future refactor could consolidate, but doing it in this PR is scope creep.

**Blocker-level deviations:**

- Test uses `__annotations__` instead of `vars()` → BLOCKER. The test would pass vacuously, providing false protection.
- Missing the `test_protocol_not_vacuous` sanity check → BLOCKER. The main test has no way to detect its own breakage.
- Test coupled to `RecordingApi` or any mutable state → non-blocker WARNING. This test should be pure inspection of two classes.
