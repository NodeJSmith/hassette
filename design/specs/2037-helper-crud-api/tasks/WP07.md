## Objectives & Success Criteria

Write a narrative documentation guide explaining how to use the typed helper CRUD API, including both the simple and race-safe idempotent-bootstrap patterns. The guide lives in `docs/pages/advanced/managing-helpers.md` (NOT `docs/guides/` — that directory doesn't exist; advanced topics live under `docs/pages/advanced/` per the existing repo structure).

The guide is the replacement for the `ensure_helper` method that was dropped in favor of user-written patterns. Without this guide, users discovering the feature would have no canonical source for "how do I self-provision a helper at app startup?" and would likely reinvent the wheel (often with the same race condition `ensure_helper` had).

Concrete outcomes:

- `docs/pages/advanced/managing-helpers.md` exists and follows the conventions of sibling advanced docs (e.g., `state-registry.md`, `dependency-injection.md`, `custom-states.md`).
- The guide has sections covering: overview of helper management, typed models (Record / Create / Update), CRUD method reference summary, the simple 5-line bootstrap pattern, the race-safe variant, counter service-call shortcuts, testing with `harness.seed_helper()`, and a "gotchas" section.
- The simple 5-line bootstrap pattern is presented as the **default recommendation** (round-3 finding F4). The race-safe variant is labeled "Advanced: concurrent provisioning."
- The race-safe variant explicitly uses `if e.code != "name_in_use": raise`, NOT a bare `except FailedMessageError:`. The text explains why.
- At least one worked example per helper domain is included, showing typical use cases from the issue body (vacation mode, counter for tracking cycles, input_select for user-facing mode selection).
- A "Testing" section shows how to use `harness.seed_helper()` and assertions against `harness.api_recorder.calls`.
- Cross-references to the auto-generated API reference for each method (links to the readthedocs URL pattern, if one exists; otherwise just method names in `code` formatting).
- A "Not included" section listing what's deliberately out of scope: subscribe commands, `ensure_helper`-style framework-level wrappers.
- The guide is referenced from `docs/pages/advanced/index.md` (or the appropriate navigation index).
- `uv run mkdocs build` succeeds without warnings (if the docs build is part of CI).

## Subtasks

1. Read `docs/pages/advanced/state-registry.md` and `docs/pages/advanced/dependency-injection.md` to understand the tone, structure, and code-block conventions of sibling advanced docs.
2. Read `docs/pages/advanced/index.md` (if it exists) to see how other advanced topics are listed in the navigation.
3. Create `docs/pages/advanced/managing-helpers.md` with a frontmatter matching other advanced docs (if any), and the following sections:
   - **Overview** — one paragraph on what HA helpers are, why apps might want to self-provision them, and the scope of this feature (32 CRUD methods + 3 counter shortcuts; subscribe commands and `ensure_helper`-style wrappers are not included).
   - **Typed models** — brief explanation of the three model types per domain (`Record` for stored config, `Create*Params` for create payloads, `Update*Params` for partial updates). Note the `extra="allow"` on Record and `extra="ignore"` on Update, and why.
   - **Creating a helper** — code example showing `await api.create_input_boolean(CreateInputBooleanParams(name="vacation_mode", initial=False))`.
   - **Listing / updating / deleting** — one code example each, using `input_boolean` as the running example.
   - **Idempotent bootstrap (the simple pattern)** — introduces the scenario ("my app wants to own an `input_boolean` but doesn't know if it exists yet"), presents the 5-line pattern as the default:
     ```python
     async def _ensure_vacation_mode(self) -> InputBooleanRecord:
         for record in await self.api.list_input_booleans():
             if record.id == "vacation_mode":
                 return record
         return await self.api.create_input_boolean(
             CreateInputBooleanParams(name="vacation_mode", initial=False)
         )
     ```
     Explain that this is safe when only ONE app in the deployment will provision this helper.
   - **Advanced: concurrent provisioning** — introduces the narrow case (multiple apps provision the same helper, launched in parallel via `asyncio.gather`). Shows the race-safe variant:
     ```python
     async def _ensure_vacation_mode(self) -> InputBooleanRecord:
         for record in await self.api.list_input_booleans():
             if record.id == "vacation_mode":
                 return record
         try:
             return await self.api.create_input_boolean(
                 CreateInputBooleanParams(name="vacation_mode", initial=False)
             )
         except FailedMessageError as e:
             if e.code != "name_in_use":
                 raise
             for record in await self.api.list_input_booleans():
                 if record.id == "vacation_mode":
                     return record
             raise
     ```
     **Critical**: explicitly call out that the bare `except FailedMessageError:` form is WRONG — it swallows transport timeouts and permission errors along with races. The `if e.code != "name_in_use": raise` form is the only acceptable variant. Cite `FailedMessageError.code` as the thing that makes this possible, and mention that the `code` field was added specifically to support this pattern.
   - **Counter service-call shortcuts** — brief section explaining `increment_counter` / `decrement_counter` / `reset_counter` operate on live entity state (not stored config). Show a usage example ("after every motion event, increment a cycle counter"). Note `timer.start` / `timer.pause` / `timer.cancel` are NOT wrapped — callers use `api.call_service("timer", "start", target={"entity_id": ...})` directly. Explain the asymmetry (counter actions are high-frequency, timer actions are typically one-off).
   - **Testing** — show the `harness.seed_helper(InputBooleanRecord(id="vacation_mode", name="Vacation Mode"))` API. Include a code example of a test that seeds a helper, exercises an app's `on_initialize`, and asserts the `api_recorder.calls` contains the expected `create_input_boolean` call. Cross-reference the existing testing docs if any.
   - **Gotchas** — a bullet list covering:
     - `CreateInputDatetimeParams` requires `has_date=True` or `has_time=True`.
     - `exclude_unset=True` is how Create/Update fields are serialized — omitting a field and setting a field to `None` produce different behavior.
     - `CounterRecord` (stored config) and `CounterState` (live runtime state) are two different models; reading the current counter value uses `Api.get_state("counter.xxx")`, while changing the configured `initial` uses `update_counter(...)`.
     - Helper creation via WebSocket persists across HA restarts (stored in HA's `.storage/`), unlike entity state.
   - **Not included / out of scope** — brief section noting: `{domain}/subscribe` commands are deferred to a follow-up issue; framework-level `ensure_helper` convenience method was considered and rejected after design review (link to `design/specs/2037-helper-crud-api/design.md` for the full rationale).
4. Add a link entry for this guide to `docs/pages/advanced/index.md` (if that file exists) in the correct alphabetical or topical position.
5. Run `uv run mkdocs build` locally to verify the guide renders without warnings.
6. Read the rendered output (if possible) or re-read the markdown to catch broken links and formatting issues.
7. Verify the code examples in the guide match the actual method signatures in `src/hassette/api/api.py` (after WP04). No stale `ensure_helper` references, no `exclude_none` references, no bare `except FailedMessageError`.

## Review Guidance

**Documentation reviewer must verify:**

- The simple 5-line pattern is the default, with the race-safe variant labeled as "Advanced" or similar (round-3 finding F4 — do NOT lead with the 8-line variant).
- The race-safe variant uses `if e.code != "name_in_use": raise`, NOT a bare `except FailedMessageError:`. A bare except would swallow timeouts and permission errors as if they were races.
- Every code example uses method signatures that actually exist in `Api` (check against `src/hassette/api/api.py` after WP04). A doc that references `ensure_helper(...)` or `api.helpers.create(...)` is broken.
- `exclude_unset` is mentioned (not `exclude_none`) wherever the serialization behavior matters.
- The "Testing" section uses `harness.seed_helper(record)` (no domain parameter) — NOT `seed_helper(domain, record)`. The signature was changed in WP05 per round-2 finding F5.
- The "Not included" section references the design doc for the `ensure_helper` decision — this preserves the reasoning for future maintainers.
- `CounterRecord` vs `CounterState` distinction is explained in "Gotchas" — otherwise users will confuse the two.
- The file path is `docs/pages/advanced/managing-helpers.md`, NOT `docs/guides/managing-helpers.md` (round-3 finding — `docs/guides/` doesn't exist in this repo).

**Integration reviewer must verify:**

- The guide is linked from the navigation index (`docs/pages/advanced/index.md` or the site's navigation YAML).
- `mkdocs build` passes without warnings related to this file.
- No broken internal links.

**Blocker-level deviations:**

- Bare `except FailedMessageError:` in the race-safe variant → BLOCKER. Trains users into the exact anti-pattern round-3 caught.
- Stale references to `ensure_helper` or any method signature that doesn't exist in the real code → BLOCKER. Users following the doc will get `AttributeError`.
- File placed at `docs/guides/managing-helpers.md` → BLOCKER. Directory doesn't exist; build fails.
