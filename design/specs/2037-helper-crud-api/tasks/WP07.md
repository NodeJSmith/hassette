## Objectives & Success Criteria

Write a narrative documentation guide explaining how to use the typed helper CRUD API, including the simple idempotent-bootstrap pattern. The guide lives in `docs/pages/advanced/managing-helpers.md` (NOT `docs/guides/` — that directory doesn't exist; advanced topics live under `docs/pages/advanced/` per the existing repo structure).

The guide is the replacement for the `ensure_helper` method that was dropped in favor of user-written patterns. Without this guide, users discovering the feature would have no canonical source for "how do I self-provision a helper at app startup?" and would likely reinvent the wheel.

Concrete outcomes:

- `docs/pages/advanced/managing-helpers.md` exists and follows the conventions of sibling advanced docs (e.g., `state-registry.md`, `dependency-injection.md`, `custom-states.md`).
- The guide has sections covering: overview of helper management, typed models (Record / Create / Update), CRUD method reference summary, the simple 5-line bootstrap pattern, counter service-call shortcuts, testing with `harness.seed_helper()`, and a "gotchas" section.
- The simple 5-line bootstrap pattern is presented as the sole recommended pattern for self-provisioning helpers.
- The guide does **not** include any "race-safe variant" that catches `name_in_use` and re-fetches. WP01 verified against HA source at tag `2026.4.1` that HA never emits `name_in_use` — `IDManager.generate_id` silently auto-suffixes on collisions. The "gotchas" section explains this behavior and the naming-discipline mitigation (each app uses a helper name unique within the deployment, typically prefixed with the app identifier) instead of documenting a broken catch-and-retry pattern.
- At least one worked example per helper domain is included, showing typical use cases from the issue body (vacation mode, counter for tracking cycles, input_select for user-facing mode selection).
- A "Testing" section shows how to use `harness.seed_helper()` and assertions against `harness.api_recorder.calls`.
- Cross-references to the auto-generated API reference for each method (links to the readthedocs URL pattern, if one exists; otherwise just method names in `code` formatting).
- A "Not included" section listing what's deliberately out of scope: subscribe commands, `ensure_helper`-style framework-level wrappers.
- The guide is referenced from `docs/pages/advanced/index.md` (or the appropriate navigation index).
- `uv run mkdocs build` succeeds without warnings (if the docs build is part of CI).

   - **Typed models** — brief explanation of the three model types per domain (`Record` for stored config, `Create*Params` for create payloads, `Update*Params` for partial updates). Note the `extra="allow"` on Record and `extra="ignore"` on Update, and why.
   - **Creating a helper** — code example showing `await api.create_input_boolean(CreateInputBooleanParams(name="vacation_mode", initial=False))`.
   - **Listing / updating / deleting** — one code example each, using `input_boolean` as the running example.
   - **Idempotent bootstrap (the simple pattern)** — introduces the scenario ("my app wants to own an `input_boolean` but doesn't know if it exists yet"), presents the 5-line pattern:
     ```python
     async def _ensure_vacation_mode(self) -> InputBooleanRecord:
         for record in await self.api.list_input_booleans():
             if record.id == "vacation_mode":
                 return record
         return await self.api.create_input_boolean(
             CreateInputBooleanParams(name="vacation_mode", initial=False)
         )
     ```
     Explain that this is correct when only ONE app in the deployment owns provisioning for this helper (the default, recommended topology). If two apps can race to create the same-named helper, HA will NOT raise an error on the loser — it will silently auto-suffix the id (`vacation_mode`, `vacation_mode_2`) and both creators will succeed, leaving semantically-duplicate records in storage. The correct mitigation is naming discipline (unique helper names per app, typically prefixed with the app identifier), not a catch-and-retry wrapper. Cross-reference the "Gotchas" section for the full explanation and cite the `## HA WebSocket Commands` section of `design/specs/2037-helper-crud-api/design.md` for the source trace.
   - **Counter service-call shortcuts** — brief section explaining `increment_counter` / `decrement_counter` / `reset_counter` operate on live entity state (not stored config). Show a usage example ("after every motion event, increment a cycle counter"). Note `timer.start` / `timer.pause` / `timer.cancel` are NOT wrapped — callers use `api.call_service("timer", "start", target={"entity_id": ...})` directly. Explain the asymmetry (counter actions are high-frequency, timer actions are typically one-off).
   - **Testing** — show the `harness.seed_helper(InputBooleanRecord(id="vacation_mode", name="Vacation Mode"))` API. Include a code example of a test that seeds a helper, exercises an app's `on_initialize`, and asserts the `api_recorder.calls` contains the expected `create_input_boolean` call. Cross-reference the existing testing docs if any.
   - **Gotchas** — a bullet list covering:
     - `CreateInputDatetimeParams` requires `has_date=True` or `has_time=True`.
     - `exclude_unset=True` is how Create/Update fields are serialized — omitting a field and setting a field to `None` produce different behavior.
     - `CounterRecord` (stored config) and `CounterState` (live runtime state) are two different models; reading the current counter value uses `Api.get_state("counter.xxx")`, while changing the configured `initial` uses `update_counter(...)`.
     - Helper creation via WebSocket persists across HA restarts (stored in HA's `.storage/`), unlike entity state.
     - **HA auto-suffixes on name collision.** `create_*` with a name that slugifies to an existing id does NOT raise — HA's `IDManager.generate_id` silently appends `_2`, `_3`, ... until it finds an unused slot. Two concurrent creators of the same-named helper will both succeed, with two semantically-duplicate records in storage. There is no `name_in_use` error code to catch. The correct mitigation is naming discipline: each app should use a helper name unique within the deployment (e.g., prefix with the app identifier, `myapp_vacation_mode`), and only one app should own provisioning for any given helper. Cite `design/specs/2037-helper-crud-api/design.md` → `## HA WebSocket Commands` → "HA does NOT reject duplicate create names" for the verified trace through HA source.
   - **Not included / out of scope** — brief section noting: `{domain}/subscribe` commands are deferred to a follow-up issue; framework-level `ensure_helper` convenience method was considered and rejected after design review (link to `design/specs/2037-helper-crud-api/design.md` for the full rationale).
