## Objectives & Success Criteria

Add 32 typed CRUD methods (8 domains × 4 ops) plus 3 counter service-call shortcut methods to `src/hassette/api/api.py`, wired through the module-level `_ws_helper_call` / `_expect_list` / `_expect_dict` helpers. Regenerate `src/hassette/api/sync.py` via the codegen target so the public sync facade picks up all 35 new methods. Integration tests cover transport shape for every new method.

Concrete outcomes:

- `src/hassette/api/api.py` contains three new module-level helpers: `_expect_list(val, context)`, `_expect_dict(val, context)`, `_ws_helper_call(api, domain, operation, **data)`. All three are at module scope (not methods on `Api`) so the sync facade generator does not emit them as public methods.
- `Api` class has 32 new async CRUD methods: `list_*`, `create_*`, `update_*`, `delete_*` for each of the 8 domains.
- `Api` class has 3 new async counter shortcut methods: `increment_counter`, `decrement_counter`, `reset_counter` — each calls `self.call_service(..., return_response=True)`.
- Counter shortcut methods live in their own section of `api.py` with a leading comment explaining they operate on live entity state, not stored config. `timer.start` / `timer.pause` / `timer.cancel` are deliberately excluded — add a brief comment in the section header.
- Every CRUD method uses `_ws_helper_call(self, domain, operation, **data)` — not `self.ws_send_and_wait(...)` directly.
- Every create/update uses `params.model_dump(exclude_unset=True)` — **not** `exclude_none=True`.
- Every method emits the logging specified in the design's "Observability / logging contract" section (list_* DEBUG with count, create_* INFO with id, update_*/delete_* DEBUG with helper_id).
- `src/hassette/api/sync.py` is regenerated via `uv run tools/generate_sync_facade.py --target api` and contains sync wrappers for all 35 new async methods.
- `uv run tools/generate_sync_facade.py --target api --check` exits 0 (drift gate passes).
- `tests/integration/test_api_helpers.py` exists with tests covering: 32 CRUD methods × 4 shape assertions each, plus 3 counter shortcuts × 2 tests each, plus helper-helper tests for `_expect_list`/`_expect_dict`/`_ws_helper_call`.
- `uv run pytest tests/integration/test_api_helpers.py -n 2` passes.
- `uv run pyright src/hassette/api/api.py src/hassette/api/sync.py tests/integration/test_api_helpers.py` passes.

## Test Strategy

**New file: `tests/integration/test_api_helpers.py`**

- **Per-domain CRUD tests**: 8 domains × 4 ops = 32 tests, each asserting transport-level shape (type string, payload fields, ID key naming) and return-type parsing.
- **Module helper tests**: 2 for `_expect_list`, 2 for `_expect_dict`, 2 for `_ws_helper_call` (success propagation + error chaining).
- **Counter shortcut tests**: 6 tests (3 methods × 2 assertions each — correct call + error propagation).

Total: **44 tests** in this WP.

Mock strategy: patch `api.ws_send_and_wait` directly on the `api` instance via `AsyncMock(return_value=...)`. No `SimpleTestServer` or REST mocking — helper CRUD is WebSocket-only. For counter shortcut tests, patch `api.call_service` instead.

Run command: `uv run pytest tests/integration/test_api_helpers.py -n 2`.

**Drift gate**: `uv run tools/generate_sync_facade.py --target api --check` must exit 0. This is the regenerator freshness test at `tests/unit/test_recording_sync_facade_generation.py::test_generator_check_mode_api_exits_zero`.

**Live-HA smoke test (stretch, highest-risk coverage)**: Add ONE `@pytest.mark.requires_ha` integration test to `tests/integration/test_api_helpers.py` that exercises the full create → list → update → delete cycle for `input_boolean` against a real HA instance. This is the only test that can catch ID-key naming mismatches that mocked tests cannot — the design identifies ID-key verification as the single highest-risk item in the whole feature (see WP01 and the "HA Source Verification" design section). The test uses the existing `pytest.mark.requires_ha` marker (search `grep -r "requires_ha" tests/` to find prior usage patterns; if none, add a new marker in `conftest.py` that skips by default and runs only when a specific env var is set). This is a stretch subtask — if `requires_ha` infrastructure does not exist yet, document the gap and file a follow-up issue rather than blocking WP04 on marker setup.

## Review Guidance

**Code reviewer must verify:**

- `_expect_list` and `_expect_dict` are module-level functions, NOT methods on `Api`. If they are methods, the sync facade generator will emit them as public `ApiSyncFacade._expect_list` / `._expect_dict` methods — pollution.
- `_ws_helper_call` is also module-level, takes `api` as its first positional argument (not `self`), and is NOT an `async def` on `Api`. Same reasoning.
- Every CRUD method calls `_ws_helper_call(self, domain, operation, ...)` — not `self.ws_send_and_wait(...)` directly. Missing the wrapper means missing error context, which defeats the whole point.
- Every create/update uses `exclude_unset=True` — NOT `exclude_none=True`. The distinction matters for `initial=False` payloads (round-1 finding F3).
- Counter shortcut methods use `return_response=True` — without it, HA errors are fire-and-forget and silently dropped (round-1 finding F10).
- The regenerated `src/hassette/api/sync.py` contains exactly 35 new methods (check with grep or the diff). Check that none of `_expect_list`, `_expect_dict`, `_ws_helper_call` appear in `sync.py`.
- ID key naming matches WP01's verified table. If WP01 documented all 8 domains using `{domain}_id`, verify all 32 update/delete methods use the consistent pattern. If WP01 documented a mix, verify `_ID_KEYS_BY_DOMAIN` is used.
- Logging follows the design's contract: list_* DEBUG with count, create_* INFO with id, update_*/delete_* DEBUG.

**Integration reviewer must verify:**

- The generated `sync.py` diff is purely additive — no existing sync methods are modified.
- The drift gate passes. Run `uv run pytest tests/unit/test_recording_sync_facade_generation.py::test_generator_check_mode_api_exits_zero -n 2` locally to confirm.
- The module-level helpers `_expect_list` etc. are NOT accessible via `api.sync._expect_list` (if they are, the underscore filter is being ignored somewhere).
- `tests/integration/test_api_helpers.py` does not share fixture state with `test_api.py` in a way that would cause test ordering issues.

**Blocker-level deviations:**

- `exclude_none=True` used anywhere → BLOCKER. Silent field-dropping.
- `_ws_helper_call` defined as `async def _ws_helper_call(self, ...)` on `Api` → BLOCKER. Generator leaks it.
- Any new method missing `_ws_helper_call` → BLOCKER. Error context lost.
- Drift gate failing → BLOCKER.
