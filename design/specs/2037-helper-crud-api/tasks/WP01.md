## Objectives & Success Criteria

Document the exact shape of Home Assistant's WebSocket helper commands for all 8 domains covered by this feature so subsequent WPs can implement CRUD methods against a verified contract rather than assumptions. This WP's deliverable is an updated section in `design/specs/2037-helper-crud-api/design.md` — no production code changes.

Concrete outcomes:

- A new `## HA WebSocket Commands` section is added to `design.md` before the `## Alternatives Considered` section.
- The section contains one sub-heading per domain (`input_boolean`, `input_number`, `input_text`, `input_select`, `input_datetime`, `input_button`, `counter`, `timer`) with a verified field table.
- A repo-wide table summarizes the `_ID_KEYS_BY_DOMAIN` map to be used by `Api` update/delete methods in WP04.
- The slug-derivation rule HA applies to helper `name` → `id` is documented, including exact transformation (lowercase, space handling, punctuation handling).
- HA's error envelope structure is documented, including whether a `code` field is present alongside `message`.
- HA's behavior for `{domain}/delete` of a nonexistent ID is documented (error response vs silent success).
- WP04 and WP05 can be implemented without further HA source reads.

## Subtasks

1. Clone or read HA source for `homeassistant/components/input_boolean/websocket_api.py` (GitHub: https://github.com/home-assistant/core). If the file does not exist at that path, find the `input_boolean` helper registration in `__init__.py` or `storage.py`. Read the `DictStorageCollectionWebsocket` base class registration to find the command names, ID key, and response shape.
2. Repeat for the other 7 domains: `input_number`, `input_text`, `input_select`, `input_datetime`, `input_button`, `counter`, `timer`.
3. Read HA's WebSocket API error envelope handling: search HA source for `"success": false` handling in `homeassistant/helpers/websocket_api.py` (or equivalent) and document whether the error object includes a `code` field.
4. Read HA's delete-of-nonexistent behavior by tracing `DictStorageCollectionWebsocket.delete` or the equivalent collection method. Note whether it raises `ItemNotFound` / returns an error response / silently succeeds.
5. Read HA's slug-derivation rule — typically via `homeassistant.util.slugify` or a local helper in each domain's storage setup. Document the exact transformation rules (lowercase, Unicode handling, special character stripping).
6. Populate a table in `design.md` for each domain with columns: `list command`, `create command`, `update command`, `delete command`, `update/delete ID key`, `create response shape`, `list response shape`.
7. Add a consolidated `_ID_KEYS_BY_DOMAIN` dict in the new section showing the expected lookup (or note that all 8 domains use the same pattern, in which case no lookup table is needed).
8. If HA uses `object_id` rather than `{domain}_id` for any update/delete command, add a warning block noting that WP04's `Api.update_*` / `Api.delete_*` methods must use the per-domain key from the lookup table.
9. Update `design.md`'s `## HA Source Verification` section to reference the new `## HA WebSocket Commands` section as the authoritative output.
10. Commit the design.md update as a single commit.

## Test Strategy

N/A — this WP produces documentation, not executable code. The design doc update is the deliverable. WP04 and WP05 will catch any errors in the documented contract via integration tests that patch the WebSocket transport with the documented payload shape.

## Review Guidance

The spec reviewer should verify:

- Every one of the 8 domains is covered (no omissions).
- The documented command names match the exact strings that HA registers (e.g., `input_boolean/create`, not `input_boolean.create` or `create_input_boolean`).
- The `update/delete ID key` column is sourced from HA code, not inferred from the design doc's assumption that `{domain}_id` is the pattern.
- If HA's `code` field on error envelopes is absent, the design doc's `FailedMessageError.code` population is noted as "always None in practice" and WP03's tests reflect that.
- If HA's delete-of-nonexistent returns an error, the design doc's RecordingApi `delete_*` behavior (raise `KeyError`) stays consistent; if HA silently succeeds, WP05 must change RecordingApi to match, and the doc must explicitly acknowledge the divergence from internal `update_*` semantics.

A FAIL on this review is any unverified claim ("I assume HA uses `input_boolean_id`..."). Every row in the verification table must cite an HA source file and line number or commit SHA.
