## Objectives & Success Criteria

`get_all_app_summaries()` returns correct aggregated telemetry for multi-instance apps. Activity counts (invocations, errors, executions, job errors, duration averages) reflect all instances. Registration counts (handler_count, job_count) reflect instance 0 only. Single-instance apps produce equivalent results to current behavior (within float tolerance for duration averages).

## Subtasks

1. Read `src/hassette/core/telemetry_query_service.py` and understand the four SQL branches in `get_all_app_summaries()` (session-scoped and non-session, listener and job). Each currently has `WHERE instance_index = 0`.

2. Write failing tests in `tests/integration/test_telemetry_query_service.py`:
   - `test_get_all_app_summaries_multi_instance_activity_aggregation` — seed 3 instances with different invocation counts, verify `total_invocations` sums across all, `handler_count` reflects instance 0 only.
   - `test_get_all_app_summaries_multi_instance_job_aggregation` — same pattern for jobs: `total_executions` sums across all, `job_count` reflects instance 0 only.
   - `test_get_all_app_summaries_single_instance_equivalence` — verify single-instance results are equivalent to current behavior (delta < 0.001 for `avg_duration_ms`).
   - `test_get_all_app_summaries_multi_instance_session_scoped` — seed multi-instance data across sessions, verify session-scoped variant aggregates correctly.

3. Split each of the four SQL branches into two queries:
   - **Registration query**: keeps `WHERE instance_index = 0`, grouped by `app_key`. Returns `handler_count` / `job_count`.
   - **Activity query**: removes the `instance_index` filter entirely, grouped by `app_key`. Returns invocation/execution totals, error counts, duration averages.

4. Add merge logic: combine registration and activity results by `app_key` into `AppHealthSummary` objects. Apps that appear in activity but not registration (all instances > 0) should still appear with `handler_count=0` / `job_count=0`.

5. Run all existing `get_all_app_summaries` tests to confirm no regressions. Run new tests to confirm they pass.

## Review Guidance

- **Design constraint**: Registration counts MUST use `WHERE instance_index = 0`. Activity counts MUST NOT filter by instance_index. Verify in the SQL strings.
- **Design constraint**: `AVG(hi.duration_ms)` must be computed over raw rows (not averaging per-instance averages). The SQL AVG in the activity query is correct — verify no intermediate aggregation step was added.
- **Blocker**: If any existing test changes its expected values, that's a regression — FR-9 requires equivalent single-instance behavior.
- **Warning**: The merge logic should handle apps that exist in one query but not the other (edge case: app with jobs but no listeners, or vice versa).
