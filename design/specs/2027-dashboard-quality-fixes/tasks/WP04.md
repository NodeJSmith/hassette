## Objectives & Success Criteria

The status bar displays a telemetry degraded indicator (e.g., amber dot with "DB degraded" label) when the `/api/telemetry/status` endpoint reports degradation. The indicator clears when the endpoint reports recovery. The poller runs regardless of which page is active, polls every 30s with exponential backoff on failure (30s тЖТ 60s тЖТ 120s cap), resets on success and on page navigation. When both WebSocket is disconnected and telemetry is degraded, "Disconnected" takes visual precedence.

## Subtasks

1. Add `telemetryDegraded: Signal<boolean>` to `AppState` in `frontend/src/state/create-app-state.ts`, initialized to `false`. Follow the existing signal pattern (e.g., how `connection` or `sessionId` are defined).

2. Create `frontend/src/hooks/use-telemetry-health.ts`:
   - Export a `useTelemetryHealth(appState: AppState)` hook.
   - Poll `getTelemetryStatus()` every 30s via `setInterval`.
   - On success: write `appState.telemetryDegraded.value = response.degraded`. Reset interval to 30s if backed off.
   - On fetch failure: write `appState.telemetryDegraded.value = true`. Apply exponential backoff (30s тЖТ 60s тЖТ 120s cap).
   - On page navigation (listen to route changes or use a `useEffect` that triggers on location): poll immediately and reset backoff to 30s.
   - Cleanup: clear interval on unmount.

3. Wire `useTelemetryHealth` in `frontend/src/app.tsx` тАФ call it from the app shell so it runs on every page.

4. Update `frontend/src/components/layout/status-bar.tsx`:
   - Read `telemetryDegraded` from `useAppState()`.
   - When `telemetryDegraded.value === true` AND `connection.value === "connected"`, show a degraded indicator (e.g., amber dot class + "DB degraded" label). Follow the existing `statusConfig` pattern.
   - When `connection.value !== "connected"`, show the connection status only тАФ "Disconnected" takes precedence over "DB degraded".

5. Add CSS classes for the degraded indicator state (e.g., `is-degraded`, `ht-pulse-dot degraded`) to the appropriate stylesheet. Follow existing patterns for `is-disconnected` / `ht-pulse-dot disconnected`.

## Test Strategy

Unit test for the hook in `frontend/src/hooks/use-telemetry-health.test.ts`:
- `test_polls_on_mount_and_interval` тАФ verify initial fetch and 30s interval.
- `test_sets_degraded_true_on_fetch_failure` тАФ verify signal is set on network error.
- `test_backs_off_on_consecutive_failures` тАФ verify interval increases.
- `test_resets_backoff_on_success` тАФ verify interval returns to 30s.

Follow the existing hook test pattern from `use-api.test.ts` and `use-websocket.test.ts`.

Integration testing of the status bar rendering is covered visually (see Visual Verification).

## Review Guidance

- **Design constraint**: Single source of truth тАФ `telemetryDegraded` is the ONLY signal the status bar reads for degraded state. The dashboard page must NOT also write to this signal.
- **Design constraint**: "Disconnected" takes visual precedence over "DB degraded" when both are true.
- **Design constraint**: Backoff must cap at 120s and reset on success AND on page navigation.
- **Blocker**: If the hook imports or depends on dashboard-specific code, that violates the single-source-of-truth architecture.
- **Warning**: The `appState` invariant тАФ `reconnectVersion` and `appStatus` must remain independent refetch paths. Adding `telemetryDegraded` must not interfere.
