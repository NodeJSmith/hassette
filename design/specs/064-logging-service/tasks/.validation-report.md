### Warnings

**W1 вҖ” `test_log_records_retention.py` mentioned in design Test Strategy but not in T01 title/summary.**
The design doc's Test Strategy section says "~40 tests affected" in `test_logging.py` and separately calls out `test_log_records_retention.py`. T01 Prompt step 5 does explicitly mention `tests/unit/core/test_log_records_retention.py` with the note "don't miss it." The Focus section also calls this out: "`test_log_records_retention.py` also uses `repo.insert_log_records()` (4 call sites)." Covered вҖ” no gap, but worth noting that the Summary doesn't mention it; the Prompt and Focus do.

**W2 вҖ” Edge case "post-shutdown logging" (StreamHandler restoration) is tested in T03 but not explicitly in a Verify checkbox.**
The design doc Edge Cases section states: "After the logging service shuts down, further log records from other services still shutting down need a path. Restoring the synchronous handler ensures late logs reach the console." T03's Prompt step 4 includes "Test on_shutdown stops listener and restores StreamHandler" in the test list, and the design Test Strategy includes "Post-shutdown logging" as new coverage. However, there is no Verify checkbox item in T03 or T04 that explicitly checks this behavior. This is a minor gap in verification signal, not a coverage gap вҖ” the test is described in the Prompt but no Verify checkbox validates it.

**W3 вҖ” `hassette.config.logging.log_queue_max` config field assumed in T03.**
T03 Prompt references `self.hassette.config.logging.log_queue_max` as an existing config field. The design doc mentions `hassette.config.web_api.log_buffer_size` (for LogCaptureHandler) and `hassette.config.logging.log_persistence_level` and `log_queue_max` implicitly in the Architecture section. If these fields don't already exist on the config model, T03 would need to add them. This is a potential implementation-time surprise but not a plan-level defect вҖ” it's the implementer's responsibility to verify.

**W4 вҖ” `Hassette.get_log_records_dropped()` change is split across T03 and T04 with a "verify already done" note.**
T03 Prompt step 3 says to update `get_log_records_dropped()` to use `self.logging_service.dropped_count`. T04 Prompt step 2 says "should already be done in T03, verify." This cross-task verification note is safe (T04 depends on T03 so the implementer will have context), but it creates a soft ambiguity about ownership. No structural problem.
