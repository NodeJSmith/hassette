## Objectives & Success Criteria

- `job-list.tsx` shows a horizontal filter chip bar when 2+ distinct groups exist. Chips include "All" + each group name.
- Active group filter is URL-persisted as `?group=<name>` via `wouter`'s `useSearch`. Survives instance switch and page refresh.
- Filter resets to "All" when `jobs` prop changes (via `useEffect([jobs])`) to prevent stale empty lists on instance switch.
- When a filter is active but produces zero results, a "No jobs match this filter" message is shown.
- Group pill clicks in `job-row.tsx` set the filter (via `onGroupClick` callback from `job-list.tsx`).
- App-detail heading updated from "Scheduled Jobs (N active)" to "Scheduled Jobs ({N} registered)" with a secondary "N currently scheduled" count.
- New `job-list.test.tsx` and updated `app-detail.test.tsx` cover all behavior.

## Subtasks

1. In `job-list.tsx`: add local `signal<string | null>` for active group filter. Initialize from `?group=` URL param on mount via `wouter`'s `useSearch` (or `useLocation` + `URLSearchParams` if `useSearch` is unavailable — check wouter's API). On chip click, update the URL param via `history.replaceState` (not `pushState`, to avoid polluting browser history with filter changes) and update the signal.
2. Compute `groups = [...new Set(jobs.filter(j => j.group).map(j => j.group))]`. When `groups.length >= 2`, render a chip bar above the list: "All" chip + one chip per group. Active chip gets a pressed visual state (e.g., `ht-badge--active` class).
3. Filter the displayed jobs list by active group signal. When filter is active and filtered list is empty, render a "No jobs match this filter" message instead of the job list.
4. Add `useEffect` with `[jobs]` dependency that resets the filter signal to `null` when jobs change (instance switch, session change).
5. Pass `onGroupClick` callback from `job-list.tsx` to each `JobRow`. When a pill is clicked, set the filter signal and update the URL param.
6. In `app-detail.tsx`: update heading at line 141 from `Scheduled Jobs ({jobCount} active)` to `Scheduled Jobs ({jobCount} registered)`. Add a secondary computed label: `{jobs.data.value?.filter(j => j.next_run != null).length ?? 0} currently scheduled`.
7. Write `frontend/src/components/app-detail/job-list.test.tsx` and update `frontend/src/pages/app-detail.test.tsx`.

## Review Guidance

- Filter state MUST be local to `job-list.tsx` (not lifted to app-detail). URL persistence is the durability mechanism, not state lifting.
- `useEffect([jobs])` reset prevents stale filters after instance switch. Must run even if the URL param is set (override URL with null, clear the param).
- Heading MUST show both counts: registered (DB total) and currently scheduled (`next_run != null`).
- The chip bar must NOT appear when fewer than 2 groups exist — no unnecessary UI for the common case.
