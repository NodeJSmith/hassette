## Objectives & Success Criteria

The web API layer and frontend UI correctly handle and display the new `EXHAUSTED_DEAD` and `EXHAUSTED_COOLING` service statuses, including the cooldown countdown timer.

- `ServiceStatusData` in `src/hassette/core/domain_models.py` includes `retry_at: float | None = None`
- `ServiceStatusWsMessage` in `src/hassette/web/models.py` includes `retry_at` if not already inherited
- Web mappers in `src/hassette/web/mappers.py` handle the new `ResourceStatus` enum values without errors
- Frontend TypeScript types are regenerated from the updated OpenAPI spec
- Frontend service status components render `EXHAUSTED_DEAD` as a permanent failure indicator (distinct from CRASHED/FAILED)
- Frontend service status components render `EXHAUSTED_COOLING` with a countdown timer showing time until next retry (using `retry_at` timestamp)
- OpenAPI spec is fresh (`scripts/export_schemas.py` run, frontend types regenerated)

## Subtasks

1. Add `retry_at: float | None = None` to `ServiceStatusData` in `src/hassette/core/domain_models.py`
2. Update `src/hassette/web/mappers.py` to map `retry_at` from `ServiceStatusPayload` to `ServiceStatusData` (pass through the value)
3. Verify `src/hassette/web/models.py` `ServiceStatusWsMessage` handles the new fields (update if needed)
4. Run `uv run python scripts/export_schemas.py` to regenerate OpenAPI spec
5. Run `cd frontend && npx openapi-typescript openapi.json -o src/api/generated-types.ts` to regenerate TypeScript types
6. Update the frontend service status component to render `EXHAUSTED_DEAD` ŌĆö display as a permanent failure with appropriate styling (red/error indicator, "Permanently failed" label)
7. Update the frontend service status component to render `EXHAUSTED_COOLING` ŌĆö display a countdown timer using `retry_at` timestamp ("Retrying in Xm Ys"), with appropriate styling (amber/warning indicator)
8. Build frontend to verify: `cd frontend && npm run build`

## Test Strategy

- `test_service_status_exhausted_dead_serialization` ŌĆö verify `EXHAUSTED_DEAD` status serializes correctly in WebSocket messages
- `test_service_status_exhausted_cooling_with_retry_at` ŌĆö verify `EXHAUSTED_COOLING` includes `retry_at` timestamp in serialized output

## Review Guidance

- Verify `retry_at` is a Unix timestamp (float), not an ISO datetime string ŌĆö the frontend calculates the countdown from the difference between `Date.now()/1000` and `retry_at`
- Verify the countdown timer updates live (requestAnimationFrame or setInterval), not just on initial render
- Verify the frontend handles `retry_at = null` gracefully (for `EXHAUSTED_DEAD` events)
- Verify the OpenAPI spec and generated types are fresh ŌĆö CI checks this via `tools/check_schemas_fresh.py`
