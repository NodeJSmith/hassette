## Objectives & Success Criteria

- The old Jinja2 UI Docker image (`ghcr.io/nodejsmith/hassette:sha-7d8c541-py3.13`) is running against live HA on port 8126.
- A screenshotter agent navigates to every page and interactive state in the 42-scenario inventory and saves full-page screenshots to `design/audits/visual-parity/old/`.
- A verification agent reads every captured screenshot and confirms the correct content is visible (expanded rows actually show expanded content, filtered tabs show filtered results, etc.).
- All screenshots use structured naming matching the scenario IDs (D1, D2, A1, AD5, etc.).
- After capture, the old container is stopped and the current hassette container is restarted.

## Subtasks

1. Stop the current hassette container: `docker stop hassette`.
2. Start the old Docker image with unquoted HA token (see design doc for exact command).
3. Wait for the old container to be healthy (`curl /api/healthz`).
4. Launch a screenshotter agent with the complete 42-scenario inventory. The agent navigates the old UI at `http://127.0.0.1:8126/ui/` and captures each scenario with Playwright MCP tools. For scenarios requiring specific data (failed apps, multi-instance), the agent checks if the data exists and skips with a note if not.
5. Launch a verification agent that reads every saved screenshot and confirms: file exists, content matches the scenario description, interactive states (expanded rows, active filters) are actually visible.
6. Stop the old container (`docker stop hassette-old && docker rm hassette-old`).
7. Restart the current container (`docker start hassette`).

## Test Strategy

- Verification agent produces a checklist of all 42 scenarios with PASS/SKIP/FAIL for each.
- Any FAIL triggers re-capture of that specific scenario.
- SKIPped scenarios are documented with the reason (e.g., "no failed apps in current HA data").

## Avoid

- **Viewport-only screenshots** — use full-page captures for pages that scroll (dashboard, app detail, logs)
- **Capturing before the page is fully loaded** — wait for data to render before screenshotting
- **Skipping scenarios without documenting why** — every skip needs a reason logged

## Review Guidance

- Every scenario from the design doc's inventory table must have either a screenshot file or a documented skip reason.
- Screenshots must be full-page captures, not viewport-only (some pages scroll).
- The verification agent must be a SEPARATE agent launch from the screenshotter (not the same agent self-verifying).
