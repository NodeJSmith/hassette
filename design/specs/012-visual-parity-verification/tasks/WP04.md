## Objectives & Success Criteria

- Every GAP and REGRESSION item in the canonical gap checklist (`design/audits/visual-parity/gap-checklist.md`) is fixed in the Preact components.
- Each fix is verified with a screenshot comparison against the old UI screenshot for that scenario.
- The gap checklist is updated with `[x] Fixed` for each resolved item.
- `npm run build` succeeds after all fixes.
- `uv run nox -s dev` passes (no Python test regressions).
- `npm test` passes (no Vitest regressions).

## Subtasks

These will be populated from the gap checklist produced by WP03. The executor reads the checklist and implements each gap fix, grouping related gaps (e.g., all app-card fixes together, all log-table fixes together).

Expected fix areas based on prior analysis:
1. **App detail page**: error display + traceback, instance metadata, instance switcher, health strip (Job Avg card, capitalize status), handler subtitle (add topic), job subtitle (add trigger_value), invocation table column order + tracebacks
2. **Apps list page**: App Key column, Error column, multi-instance expand/collapse rows, card wrapper, tab formatting
3. **Dashboard**: app card error rate, error feed (handler_method, error_type badge), session info bar, empty state text
4. **Logs page**: app filter dropdown, "All Levels" option, app column links, card wrapper, sticky header, multi-column sort
5. **Layout**: sidebar logo image, wire AlertBanner into app.tsx, action button Reload style
6. **Cross-cutting**: any CSS/spacing/font gaps found in the visual comparison

For each GAP-NNN entry from the checklist:
1. Implement the fix in the referenced Preact file
2. Take a screenshot of the fixed scenario
3. Compare against the old UI screenshot for that scenario
4. Mark `[x] Fixed` in gap-checklist.md

## Avoid

- **Raw hex colors or magic spacing numbers** — every CSS value must reference `--ht-*` tokens from `design/direction.md`
- **Fixing DIFFERENT items** — only fix GAP and REGRESSION classifications. DIFFERENT items are intentional design choices.
- **Skipping visual verification** — every fix must be screenshotted and compared, not just code-reviewed

## Test Strategy

- Each fix is visually verified by taking a screenshot and comparing against the corresponding old UI screenshot.
- Grouped fixes are batch-verified (e.g., all app-detail changes verified together).
- Full `npm run build`, `uv run nox -s dev`, and `npm test` after all fixes.
