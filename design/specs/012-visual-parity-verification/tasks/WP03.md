## Objectives & Success Criteria

- A comparison agent reads every old/new screenshot pair (42 pairs) plus the old Jinja2 template source from git history and produces a structured comparison document at `design/audits/visual-parity/comparison.md`.
- Every visual difference is classified as GAP / REGRESSION / DIFFERENT / IMPROVEMENT.
- An adversarial verifier agent independently re-reads every old screenshot and checks whether all visible elements appear in the comparison findings. It appends any missed findings.
- The final merged output is a canonical gap checklist at `design/audits/visual-parity/gap-checklist.md` with per-gap file references and checkboxes.
- The prior gap analysis (`design/research/2026-03-20-visual-parity-gaps.md`) is cross-referenced — every gap from the prior analysis should appear in the new checklist, plus any new gaps discovered from screenshots.

## Subtasks

1. Launch a comparison agent. It receives:
   - All screenshot file paths from `design/audits/visual-parity/old/` and `new/`
   - The old template source (retrieved via `git show 01a3889~1:src/hassette/web/templates/...`)
   - Instructions to compare each pair and note every difference
   - The classification scheme (GAP/REGRESSION/DIFFERENT/IMPROVEMENT)
   Output: `design/audits/visual-parity/comparison.md`

2. Launch an adversarial verifier agent (separate, independent). It receives:
   - All old screenshots only (NOT the comparison doc — to avoid anchoring bias)
   - The old template macros file for structural context
   - Instructions to enumerate every visible UI element in each screenshot and check if the new UI has it
   Output: `design/audits/visual-parity/verifier-findings.md`

3. Merge both outputs into `design/audits/visual-parity/gap-checklist.md`. Each gap entry includes:
   - Scenario ID (D1, AD5, etc.)
   - What the old UI showed
   - What the new UI shows (or is missing)
   - File to fix in the Preact SPA
   - Old template reference
   - Status checkbox

4. Cross-reference against the prior gap analysis. Every prior gap should map to a checklist entry. Flag any prior gaps NOT confirmed by the visual comparison.

## Avoid

- **Giving the adversarial verifier access to the comparison doc** — it must read old screenshots independently to avoid anchoring bias
- **Normalizing visual differences** — if something looks different, log it. Classification (GAP vs DIFFERENT) happens after, not during, comparison
- **Ignoring prior gap analysis** — every prior gap must appear in the checklist or be explicitly marked as unconfirmed

## Test Strategy

- The gap checklist must have entries for every GAP and REGRESSION found by either agent.
- The adversarial verifier must document what it independently checked. If it finds additional items the comparison agent missed, those are high-confidence gaps. If it confirms the comparison was thorough, that is also a valid (and useful) outcome.
- Every entry in the prior gap analysis must map to a checklist entry or be explicitly marked "NOT CONFIRMED — visual data insufficient."

## Review Guidance

- The comparison agent and adversarial verifier MUST be separate agent launches (not the same agent doing both passes).
- The adversarial verifier must NOT receive the comparison doc as input — it reads screenshots independently.
- The gap checklist must reference specific Preact component files and old template files for every entry.
- DIFFERENT and IMPROVEMENT entries should be included in the checklist but marked as non-actionable.
