## Objectives & Success Criteria

`HassetteHarness` stubs mirror error handler dispatch logic for precedence testing. Integration tests exercise the full error handler flow through real components. After this WP, harness-based tests verify precedence routing, and CommandExecutor-based tests verify execution path correctness including timeout, double-failure, and sync/async handler behavior.

## Subtasks

1. Extend `HassetteHarness` bus stub in `src/hassette/test_utils/harness.py` to check `listener.error_handler` and the app-level handler after a failed invocation, mirroring the real executor's dispatch logic (per-registration wins, app-level fallback).
2. Extend `HassetteHarness` scheduler stub similarly for `job.error_handler` and app-level.
3. Write harness-based integration tests in `tests/integration/test_bus_error_handler.py` and `tests/integration/test_scheduler_error_handler.py` for precedence and routing (see Test Strategy for named functions).
4. Write executor-based integration tests in `tests/integration/test_command_executor_error_handler.py` using real `CommandExecutor` with mocked `TelemetryRepository` for execution path correctness (see Test Strategy for named functions).
5. Update documentation: `docs/pages/core-concepts/bus/handlers.md` — add error handling section with examples. `docs/pages/core-concepts/scheduler/` — add error handling section. Include `on_error()` docstring note about reload gap.

## Test Strategy

- `tests/integration/test_bus_error_handler.py`: `test_app_level_error_handler_called_on_failure`, `test_per_listener_error_handler_wins`, `test_no_handler_framework_default_behavior`, `test_multiple_listeners_different_handlers`, `test_on_error_registered_after_listeners_still_works` (invocation-time resolution).
- `tests/integration/test_scheduler_error_handler.py`: `test_app_level_error_handler_called_on_job_failure`, `test_per_job_error_handler_wins`, `test_no_handler_framework_default`, `test_error_context_contains_args_kwargs`.
- `tests/integration/test_command_executor_error_handler.py`: `test_error_handler_timeout_fires_after_5s`, `test_sync_error_handler_wraps_in_thread`, `test_double_failure_logged_and_counted`, `test_error_handler_runs_after_framework_log`, `test_cancelled_error_not_routed_to_handler`, `test_timeout_error_not_routed_to_handler`.

## Review Guidance

- Verify harness stubs check `listener.error_handler` first (per-registration), then app-level fallback — matching the real executor's resolution order.
- Verify integration tests exercise invocation-time resolution: register listeners BEFORE calling `on_error()`, then verify the handler still fires (FR11).
- Verify `test_error_handler_timeout_fires_after_5s` uses a handler that sleeps > 5s and asserts the WARNING log.
- Verify documentation examples show both `bus.on_error()` and `on_error=` patterns, plus the reload gap note.
