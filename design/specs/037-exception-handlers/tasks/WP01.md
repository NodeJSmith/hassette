## Objectives & Success Criteria

Establish the foundational types that all subsequent WPs depend on: domain-specific error handler type aliases, frozen error context dataclasses, and the `exc` field on `ExecutionResult`. After this WP, `BusErrorContext`, `SchedulerErrorContext`, `BusErrorHandlerType`, `SchedulerErrorHandlerType`, and `ExecutionResult.exc` are importable and usable. Unit tests verify construction, field access, and traceback generation.

## Subtasks

1. Add `BusErrorHandlerType` and `SchedulerErrorHandlerType` type aliases to `src/hassette/types/types.py`. Import `BusErrorContext` and `SchedulerErrorContext` under `TYPE_CHECKING` to avoid circular imports. Export from `src/hassette/types/__init__.py`.
2. Create `src/hassette/bus/error_context.py` with frozen dataclass `BusErrorContext`: fields `exception: BaseException`, `traceback: str`, `topic: str`, `listener_name: str`, `event: Event[Any]`. Export from `src/hassette/bus/__init__.py`.
3. Create `src/hassette/scheduler/error_context.py` with frozen dataclass `SchedulerErrorContext`: fields `exception: BaseException`, `traceback: str`, `job_name: str`, `job_group: str | None`, `args: tuple[Any, ...]`, `kwargs: dict[str, Any]`. Export from `src/hassette/scheduler/__init__.py`.
4. Add `exc: BaseException | None = None` field to `ExecutionResult` in `src/hassette/utils/execution.py`. In `track_execution()`, assign `result.exc = exc` before `raise` in the `except Exception as exc:` branch only. Do not populate for `CancelledError` or `TimeoutError`.
5. Add `error_handler_timeout_seconds: float = 5.0` to the config class in `src/hassette/config/config.py`, following the existing `event_handler_timeout_seconds` / `scheduler_job_timeout_seconds` pattern with the same validator.

## Test Strategy

- `tests/unit/bus/test_error_context.py`: `test_bus_error_context_construction`, `test_bus_error_context_frozen`, `test_bus_error_context_traceback_always_populated` — verify field access, immutability, and that traceback is always a non-empty string regardless of exception type.
- `tests/unit/scheduler/test_error_context.py`: `test_scheduler_error_context_construction`, `test_scheduler_error_context_frozen`, `test_scheduler_error_context_args_kwargs` — verify field access, immutability, and registration-time args/kwargs.
- `tests/unit/utils/test_execution.py` (extend existing): `test_execution_result_exc_populated_on_exception`, `test_execution_result_exc_none_on_success`, `test_execution_result_exc_none_on_cancelled`, `test_execution_result_exc_none_on_timeout` — verify `exc` is set only in the `except Exception` branch.

## Review Guidance

- Verify `BusErrorContext.traceback` is typed `str` (not `str | None`) — the design explicitly requires always-populated tracebacks in the user-facing context, unlike the framework's own log suppression.
- Verify `ExecutionResult.exc` is only assigned in the `except Exception as exc:` branch, not in `CancelledError` or `TimeoutError` branches (FR10 compliance).
- Verify type aliases use `Callable[[BusErrorContext], Awaitable[None]] | Callable[[BusErrorContext], None]` form, not `Callable[...]`.
- Check that `error_handler_timeout_seconds` config field has the same validator as `event_handler_timeout_seconds`.
