## Objectives & Success Criteria

The 1039-line `docs/pages/appdaemon-comparison.md` is replaced by a `Migration Guide` section with 8 pages under `docs/pages/migration/`:

- `migration/index.md` — Overview, migration decision guide, "is it worth migrating?", quick start checklist
- `migration/concepts.md` — Mental model differences (callback model vs. DI, blocking vs. async, app isolation)
- `migration/bus.md` — AppDaemon listeners → Hassette Bus (on_state_change, on_call_service, on)
- `migration/scheduler.md` — AppDaemon scheduler → Hassette Scheduler (run_in, run_every, run_at equivalents)
- `migration/api.md` — AppDaemon AD API → Hassette Api (call_service, get_state, etc.)
- `migration/configuration.md` — apps.yaml → hassette.toml + AppConfig pattern
- `migration/testing.md` — AppDaemon testing → Hassette test_utils harness
- `migration/checklist.md` — Step-by-step migration checklist

All ~48 inline code blocks in the original page are extracted to snippet files under `docs/pages/migration/snippets/`. `mkdocs.yml` nav is updated. All inbound cross-links to the old `appdaemon-comparison.md` are updated. `mkdocs build --strict` passes.

An AppDaemon user can land on `migration/index.md`, navigate directly to the concept area most relevant to their migration, and find side-by-side AppDaemon → Hassette code examples on each page.

## Subtasks

1. Read the nav audit (`design/audits/2026-04-11-docs-nav-audit/audit.md`) Section 1 (proposed nav diff) for the Migration Guide nav structure, and Section 3 (redirect table) for the inbound link inventory.

2. Read `docs/pages/appdaemon-comparison.md` in full. Map each top-level `###` section to one of the eight migration pages. Record the mapping explicitly before writing any new file.

3. Create `docs/pages/migration/snippets/` directory. Extract all inline code blocks from `appdaemon-comparison.md` into snippet files. Name each snippet file after its concept (e.g., `migration/snippets/bus_on_state_change.py`). Every snippet must be a valid, parseable Python file — use section markers and a minimal `App` subclass wrapper for snippets that reference `self.bus` / `self.api` / `self.scheduler`.

4. Write each of the eight migration pages using the content mapped in Step 2. Each page should:
   - Open with a one-paragraph orientation ("This page covers how to migrate [concept] from AppDaemon to Hassette")
   - Use side-by-side comparisons (via `pymdownx.tabbed` with "AppDaemon" and "Hassette" tabs, or adjacent code blocks with captions)
   - Reference snippet files via `--8<--` with section markers
   - Be accurate against current Hassette source (verify each Hassette snippet against `src/hassette/`)
   - Flag anything that cannot be confidently verified with `!!! warning "Needs human review"`

5. Update `mkdocs.yml` nav. Replace:
   ```yaml
   - AppDaemon Comparison: pages/appdaemon-comparison.md
   ```
   with:
   ```yaml
   - Migration Guide:
       - Overview: pages/migration/index.md
       - Mental Model: pages/migration/concepts.md
       - Bus & Events: pages/migration/bus.md
       - Scheduler: pages/migration/scheduler.md
       - API Calls: pages/migration/api.md
       - Configuration: pages/migration/configuration.md
       - Testing: pages/migration/testing.md
       - Migration Checklist: pages/migration/checklist.md
   ```

6. Delete `docs/pages/appdaemon-comparison.md`.

7. Update all inbound cross-links to the deleted page. At minimum: `docs/index.md` (check for the link at approximately line 71). Search for any other references with `grep -r "appdaemon-comparison" docs/`.

8. Run `python -m py_compile` on every new `.py` snippet file in `docs/pages/migration/snippets/`.

9. Run `mkdocs build --strict`. Fix any errors or warnings before committing.

10. Commit all changes.
