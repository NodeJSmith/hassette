## Objectives & Success Criteria

A committed file `docs/nav-audit.md` exists and contains all eight required sections (see Subtasks). `mkdocs build` (non-strict) passes on the current codebase — confirming the build is clean before any content changes begin. The orchestrator is paused after this WP completes; no content WP starts until the author records an explicit approval signal in `docs/nav-audit.md` (add `**APPROVED**` and a date + note on accepted vs. deferred recommendations).

The audit artifact is the authoritative input for all downstream WPs. Every subsequent WP that references nav structure, split points, or the API reference allowlist must cite a section of this document.

## Subtasks

1. Read `mkdocs.yml` nav block in full (lines 22–78) and record the current nav tree verbatim.

2. Read every page listed in the nav (use Glob `docs/pages/**/*.md`). For each page, note: current line count, snippet-include count vs. inline-fence count, identified issues (thin, dense, misplaced, stale content). Use the research brief's page-size table as a starting point; verify line counts and add quality notes.

3. Produce Section 1 of `docs/nav-audit.md`: **Nav structure evaluation** — current nav tree, recommended restructuring, rationale for each change. Include a concrete before/after `mkdocs.yml` nav block diff showing the proposed new nav. Key decisions to reflect:
   - `Hassette vs. YAML Automations` stays in Getting Started.
   - `AppDaemon Comparison` is replaced by a `Migration Guide:` section with 8 pages: `migration/index.md`, `migration/concepts.md`, `migration/bus.md`, `migration/scheduler.md`, `migration/api.md`, `migration/configuration.md`, `migration/testing.md`, `migration/checklist.md`.
   - `Testing:` nav folder gains three subpages: `testing/time-control.md`, `testing/concurrency.md`, `testing/factories.md`.
   - Any thin pages that should be merged or expanded — note the recommendation.

4. Produce Section 2: **Per-page recommendations** — a table with columns: page path, recommendation (keep/rewrite/split/merge/delete), rationale.

5. Produce Section 3: **Redirect table** — for every page the audit recommends deleting, merging, or moving, list the old path, the new path (or `DELETED`), and any pages with inbound links that must be updated. At minimum include `pages/appdaemon-comparison.md` → `pages/migration/index.md`.

6. Produce Section 4: **Theme and CSS audit** — check `mkdocs.yml` for `extra_css:` (currently absent), evaluate the dead `docs/_static/style.css` (Sphinx selectors, not registered), and assess whether the current Material feature configuration supports side-by-side comparison tables and tabbed content. Recommendation: delete `_static/style.css`; replace any `.hero` usage on `index.md:1` with a valid Material `attr_list` approach.

7. Produce Section 5: **Snippet dependency map** — reproduce the cross-section shared snippets table from the research brief and confirm it is still accurate against the current codebase. List the four cross-section shared snippets that require completion-report notes in any WP that touches them.

8. Produce Section 6: **mkdocstrings API reference scope** — enumerate the proposed public-reference allowlist. Seed it from `hassette.__all__` (read `src/hassette/__init__.py`) and `hassette.test_utils.__init__.py` Tier 1 entries. Extend with a curated set of types users commonly reference via autorefs (e.g., `ScheduledJob`, event model classes, state model base classes). For each addition beyond `__all__`, note why it belongs in the public reference.

9. Produce Section 7: **Docstring gap list** — for each class/function on the allowlist produced in Step 8, check whether it has at least a one-line summary and parameter annotations for non-obvious types. List gaps in a table: class/function, current state, what's missing.

10. Produce Section 8: **`mkdocs build --strict` readiness** — run `mkdocs build` (non-strict) and report any existing warnings or errors. Note what would fail under `--strict` (likely unresolved autorefs pointing to internal modules). This baseline informs WP10 and WP11. Also explicitly evaluate whether `STATE_REGISTRY`, `TYPE_REGISTRY`, `TypeConverterEntry`, and `register_*` functions (all in `hassette.__all__`) should be documented as public API or treated as implementation detail — these have dedicated Advanced pages but their public status must be confirmed per spec FR#3. Record the decision in this section so WP08 and WP10 have an authoritative answer.

11. Add an approval placeholder at the top of `docs/nav-audit.md`:
    ```
    **Approval status:** PENDING — author must add `**APPROVED**` and a date before content WPs run.
    ```

12. Commit `docs/nav-audit.md`.

13. Notify the orchestrator that this WP is complete and that the workflow is paused pending author approval.
