## Objectives & Success Criteria

The 637-line `docs/pages/testing/index.md` is restructured into a four-page Testing section:

- `docs/pages/testing/index.md` — Quick Start + harness setup + basic assertions (beginner-complete page)
- `docs/pages/testing/time-control.md` — Time Control API
- `docs/pages/testing/concurrency.md` — Concurrency rules, same-class lock, pytest-xdist interaction
- `docs/pages/testing/factories.md` — Event factories, state factories, RecordingApi coverage boundary, internals

`mkdocs.yml` nav under `Testing:` is updated to include the three new subpages. All inline fenced code blocks across all four pages are converted to snippet files under `docs/pages/testing/snippets/`. `mkdocs build --strict` passes.

A beginner reading only `testing/index.md` gets a complete, working mental model of how to write tests — they can stop reading at the end of that page and write useful tests.

## Subtasks

1. Read the nav audit (`design/audits/2026-04-11-docs-nav-audit/audit.md`) Section 2 for any recommendations on the Testing section.

2. Read `docs/pages/testing/index.md` in full alongside `src/hassette/test_utils/__init__.py` (Tier 1 public surface), `src/hassette/test_utils/` implementation, and `tests/` to understand the testing model.

3. Identify the split boundary:
   - **Main page content**: `AppTestHarness` setup, `make_test_config`, harness constructor parameters, `trigger_state_change` / `trigger_event`, the drain pattern (`async with harness.drain()`), basic `RecordingApi` assertions (`assert_called`, `assert_not_called`), startup failure testing.
   - **`time-control.md`**: `harness.time` API, `tick()`, `advance()`, any caveats about async timing.
   - **`concurrency.md`**: same-class lock behavior, time-control lock, pytest-xdist interaction, `pytest-asyncio` mode, `DrainError`/`DrainFailure`/`DrainTimeout` semantics.
   - **`factories.md`**: event factory functions (`make_state_change_event`, etc.), state factory functions, `RecordingApi` internals and coverage boundary (`Tier 2` re-exports), `AppConfigurationError`.

4. Create the three new page files: `docs/pages/testing/time-control.md`, `docs/pages/testing/concurrency.md`, `docs/pages/testing/factories.md`.

5. Rewrite `docs/pages/testing/index.md` with only the beginner-complete content identified in Step 3. Lead with a minimal working example (harness setup → trigger event → drain → assert). Add a prominent navigation block at the bottom pointing to the advanced subpages.

6. Write `docs/pages/testing/time-control.md`, `testing/concurrency.md`, and `testing/factories.md` with the content assigned in Step 3. Each page should assume the reader has read the main page.

7. Convert all inline fenced code blocks across all four pages to snippet files under `docs/pages/testing/snippets/`. Each snippet must be a valid Python file (imports + class wrapper where needed).

8. Update `mkdocs.yml` nav. Replace:
   ```yaml
   - Testing:
       - Testing Your Apps: pages/testing/index.md
   ```
   with:
   ```yaml
   - Testing:
       - Quick Start: pages/testing/index.md
       - Time Control: pages/testing/time-control.md
       - Concurrency: pages/testing/concurrency.md
       - Factories & Internals: pages/testing/factories.md
   ```

9. Run `python -m py_compile` on every new or modified `.py` snippet file in `docs/pages/testing/snippets/`.

10. Run `mkdocs build --strict`. Fix any errors or warnings before committing.

11. Commit all changes.
