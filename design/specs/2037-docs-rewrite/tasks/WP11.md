## Objectives & Success Criteria

`mkdocs build --strict` passes with zero errors and zero warnings on the final state of the docs. The Getting Started 30-minute walkthrough is completed successfully (a technical writer or the author reads the path and confirms no step requires off-page knowledge beyond Python basics). The CI pipeline is updated to run `mkdocs build --strict` on every PR. No dead snippet files exist anywhere under `docs/`. The `Troubleshooting` and `Changelog` pages are reviewed and confirmed accurate.

## Subtasks

1. Run `mkdocs build --strict` on the full docs. Capture all errors and warnings. Fix every one before proceeding — this WP does not complete until the build is clean.

2. Conduct the Getting Started walkthrough. Read the following pages in order as a new Python developer with no prior Hassette experience would:
   - Home page (`docs/index.md`) — Does it clearly explain what Hassette is and who it's for?
   - `getting-started/index.md` (Local Setup) — Can you follow the steps without prior knowledge? Are all commands accurate?
   - `getting-started/hassette-vs-ha-yaml.md` — Does this help a reader decide whether to proceed?
   - `getting-started/ha_token.md` — Is the token acquisition process clear?
   - `core-concepts/index.md` (Architecture) — Does this page correctly orient a user for what comes next?

   Document the walkthrough results: any step that requires off-page knowledge (beyond Python basics) must be fixed before this WP is complete. Note any gaps, unclear steps, or missing prerequisite explanations.

3. Review `docs/pages/troubleshooting.md` (59 lines). Verify that each listed troubleshooting scenario is still relevant and the diagnostic steps are accurate. Remove or update stale entries. If the page has inline code blocks, convert them to snippets.

4. Verify `docs/CHANGELOG.md` (symlinked from repo root). Confirm the symlink resolves and the page renders correctly. No content changes needed — this is a build-time verification only.

5. Audit for dead snippet files: run `grep -r '\-\-8<\-\-' docs/pages/ | grep -oP '"[^"]*"' | sort -u` to get the set of all referenced snippet files. Then `find docs/pages/ -path '*/snippets/*' -name '*.py'` to get all snippet files on disk. Any snippet file on disk but not in the referenced set is dead — delete it.

6. Update the CI pipeline to run `mkdocs build --strict` on every PR. Create a new workflow file at `.github/workflows/docs.yml` (no existing workflow covers docs builds). The command to use:
   ```bash
   uv run mkdocs build --strict
   ```
   Ensure the job installs the `docs` optional dependencies (check `pyproject.toml` for a `[project.optional-dependencies]` docs group or equivalent).

7. Run `mkdocs build --strict` one final time after all changes in this WP. Confirm zero errors and zero warnings.

8. Update `design/audits/2026-04-11-docs-nav-audit/audit.md` approval status (if applicable) or confirm it is already approved.

9. Commit all changes. The commit message should summarize the verification results and confirm the 30-minute path is functional.

- **Walkthrough result**: confirm the 30-minute Getting Started path is functional. List any steps that were fixed during the walkthrough.
- **Dead snippets removed**: list any additional dead snippet files deleted in Step 5.
- **CI pipeline**: confirm `.github/workflows/docs.yml` was created and the job name.
- **Flags summary**: grep `docs/pages` for any remaining `!!! warning "Needs human review"` admonitions across all pages and list them — these are the known uncertainties that still require author attention.
