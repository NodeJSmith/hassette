## Objectives & Success Criteria

Ten Core Concepts pages are reviewed, rewritten where needed, and made snippet-compliant: `docs/pages/core-concepts/index.md` (Architecture), all three Apps pages, all three Bus pages, and all three Scheduler pages. All inline fenced code blocks on these pages are converted to snippet files. `mkdocs build --strict` passes after all changes.

Each page is accurate against the current source code. Thin pages are expanded to cover their topic adequately. Dense pages are restructured for clarity.

## Subtasks

1. Read the nav audit (`design/audits/2026-04-11-docs-nav-audit/audit.md`) Section 2 for per-page recommendations on the 10 pages in scope.

2. Read and rewrite `docs/pages/core-concepts/index.md` (Architecture, 94 lines). Verify the architecture description against `src/hassette/core/core.py`, `src/hassette/resources/base.py`, and `src/hassette/app/app.py`. This page sets the mental model for all of Core Concepts — it must be accurate and clear.

3. Read `docs/pages/core-concepts/apps/index.md`, `apps/lifecycle.md`, and `apps/configuration.md` alongside `src/hassette/app/app.py`. Verify App class lifecycle hooks (`on_initialize`, `on_ready`, `on_shutdown`), ClassVar attributes, and `AppConfig` pattern. The existing snippets in `apps/snippets/` are largely accurate but check for any stale method names or missing parameters. Convert any remaining inline blocks to snippet files.
   - **Shared snippet note**: `apps/snippets/app_config_definition.py` and `apps/snippets/app_config.toml` are also referenced by `core-concepts/configuration/applications.md` (WP05). If you modify these files, list `configuration/applications.md` in the completion report.

4. Read `docs/pages/core-concepts/bus/index.md` (135 lines), `bus/handlers.md` (89 lines), and `bus/filtering.md` (99 lines) alongside `src/hassette/bus/bus.py`. The bus module has good docstrings — use them as the source of truth. Verify the filtering predicates, conditions, and handler patterns. Convert any remaining inline blocks. `bus/index.md` currently has inline blocks mixed with cross-references — convert those inline blocks.

5. Read `docs/pages/core-concepts/scheduler/index.md` (47 lines), `scheduler/methods.md` (175 lines), and `scheduler/management.md` (57 lines) alongside `src/hassette/scheduler/`. Verify all scheduling methods (`run_in`, `run_once`, `run_every`, `run_hourly`, `run_daily`, `run_cron`), parameter signatures, and job management API. These pages are mostly snippet-based already; spot-check accuracy and expand any thin explanatory prose.

6. Run `python -m py_compile` on every new or modified `.py` snippet file touched in this WP.

7. Run `mkdocs build --strict`. Fix any errors or warnings before committing.

8. Commit all changes.

## Review Guidance

Method signatures in snippet files must match the current source — a snippet showing `run_in(callback, delay)` when the actual signature has changed is a BLOCKER.

Shared snippet modifications must be called out in the completion report — silent modification of a shared snippet without noting downstream pages is a WARNING.

The Architecture page must present an accurate resource hierarchy (Resource → Service → App, Bus, Scheduler, Api, StateManager) — a description that doesn't match the actual class hierarchy is a BLOCKER.
