This matches design.md. But the same task's **Focus** section says the opposite:
> "`conftest.py:make_mock_app_instance` (line 207) creates a full mock App ŌĆö the `mark_ready = Mock()` there is part of a mock factory, not a spy on a real instance. **Adjust the factory to not set `mark_ready` at all** (since production code calls the module function, not `inst.mark_ready()`)."

The Focus section instructs the executor to remove the `mark_ready` assignment from the mock factory entirely ŌĆö not to patch `hassette.resources.lifecycle.mark_ready` as both the design doc and the task's own Prompt-adjacent table specify. This is a direct internal contradiction (Prompt table vs. Focus) and a divergence from design.md's prescribed fix for this file. An executor following the Focus section's guidance would not perform the `patch()`-based migration that FR#7's Verify criterion and AC#7's grep check expect for this call site, potentially causing AC#7's grep check to pass for the wrong reason (no `mark_ready = Mock()` at all, rather than a `patch()` migration) while silently diverging from the documented approach.

**Recommendation:** Reconcile before execution ŌĆö either update the Focus text to align with "patch the lifecycle module" (matching design.md), or update design.md's Test Strategy table row and T04's per-file table to reflect "remove the attribute from the mock factory" as the intended fix, and confirm this doesn't break other tests in `conftest.py` that may rely on `app.mark_ready` being a settable mock.
