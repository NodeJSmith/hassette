## Objectives & Success Criteria

WebSocketService calls `_emit_readiness_event()` after `mark_ready()` and `mark_not_ready()` in its `serve()` reconnect paths. When the WebSocket disconnects mid-operation, a `service_status` event with `ready=false` is emitted immediately. When it reconnects and re-authenticates, a `service_status` event with `ready=true` is emitted. The existing `handle_running()` event (from `_serve_wrapper`) already carries the correct readiness snapshot from WP01 — this WP adds the mid-operation transitions.

## Subtasks

1. In `src/hassette/core/websocket_service.py`, after `self.mark_not_ready(reason="Early drop detected")` (line ~206), add `await self._emit_readiness_event()`
2. After `self.mark_not_ready(reason="WebSocket recv loop failed")` (line ~212), add `await self._emit_readiness_event()`
3. After `self.mark_ready(reason="WebSocket connected, authenticated, and subscribed")` (line ~252), add `await self._emit_readiness_event()`
4. Verify no other `mark_ready()` / `mark_not_ready()` calls in `websocket_service.py` need emission — the shutdown path does not, because `handle_stop()` carries `ready=false`

## Test Strategy

**New tests in `tests/unit/core/test_websocket_readiness_events.py`:**
- `test_mark_not_ready_early_drop_emits_event`: mock the service in RUNNING state, simulate the early-drop path, capture `hassette.send_event` calls — assert the emitted `ServiceStatusPayload` has `ready=False` and `ready_phase` matching the early-drop reason
- `test_mark_ready_after_connect_emits_event`: mock the service in RUNNING state, simulate successful connection, capture `hassette.send_event` — assert the emitted payload has `ready=True` and `ready_phase` containing "connected"
- `test_mark_not_ready_recv_loop_failed_emits_event`: mock the recv-loop failure path, capture `hassette.send_event` — assert the emitted payload has `ready=False`

**Existing test update:**
- `tests/integration/test_websocket_service.py:test_service_status_stays_running_during_early_drop` — verify it still passes; the new emission is additional behavior, not a replacement

## Review Guidance

- Verify `_emit_readiness_event()` calls are placed AFTER `mark_ready()`/`mark_not_ready()` — ordering matters because the helper reads `is_ready()` and `_ready_reason`
- Verify no `_emit_readiness_event()` calls appear in shutdown/stop paths — those are handled by `handle_stop()` from WP01
- Verify the `serve()` context is always async where these calls are placed — `_emit_readiness_event()` is `async def`
