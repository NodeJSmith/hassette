## Objectives & Success Criteria

The frontend receives, stores, and renders service readiness. Services that are RUNNING but not ready show an amber "Starting" row with the phase description. Services that are RUNNING and ready are hidden (healthy). The `serviceStatus` signal is cleared on WS reconnect. A `readinessVariant` helper encapsulates the combined status+readiness→variant logic.

## Subtasks

1. Add `ready: boolean` and `ready_phase: string | null` to `WsServiceStatusPayload` in `frontend/src/api/ws-types.ts`
2. Add `ready: boolean` and `ready_phase: string | null` to `ServiceStatusEntry` in `frontend/src/state/create-app-state.ts`
3. Update the `service_status` handler in `frontend/src/hooks/use-websocket.ts` to map `ready: msg.data.ready ?? false` and `ready_phase: msg.data.ready_phase ?? null` into the state entry
4. Add `state.serviceStatus.value = {}` on WS reconnect in `use-websocket.ts` alongside the existing `state.logs.clear()` (line ~73)
5. Add `readinessVariant(status: string, ready: boolean): StatusVariant` to `frontend/src/utils/status.ts` — returns `"warning"` when `status === "running" && !ready`, else delegates to `statusToVariant(status)`
6. Update `ServiceStatusPanel` filter in `frontend/src/components/dashboard/service-status-panel.tsx`: a service is visible if `!HEALTHY_STATUSES.has(e.status) || (e.status === "running" && !e.ready)`
7. Update `ServiceRow` to use `readinessVariant(status, entry.ready)` instead of `statusToVariant(status)`
8. Add `"Starting"` label: when `status === "running" && !entry.ready`, use label `"Starting"` instead of the STATUS_LABELS lookup
9. Add phase detail rendering: when `entry.ready_phase` is non-null and `status === "running" && !entry.ready`, render a `<span class="ht-ssp__detail">` with the phase text (e.g., "Connecting to HA...")
10. Fix existing CSS bug: change `var(--ht-text-muted)` to `var(--ht-text-secondary)` in `.ht-ssp__detail` in `frontend/src/global.css` (already applied during challenge — verify it's present)
11. Regenerate frontend types from updated WS schema: `cd frontend && npx openapi-typescript openapi.json -o src/api/generated-types.ts`
12. Build the frontend to verify: `cd frontend && npm run build`

## Review Guidance

- Verify `?? false` / `?? null` guards in the WS handler — without these, buffered pre-schema events produce `undefined` and every service shows as "Starting"
- Verify `serviceStatus` is cleared on reconnect alongside `logs.clear()` — stale "Starting" rows persist indefinitely without this
- Verify `readinessVariant` is used in `ServiceRow` instead of raw `statusToVariant` — the shared utility doesn't know about readiness
- Verify design tokens: amber dot uses `--ht-warning`, phase text uses `--ht-text-secondary` at `--ht-text-sm` (14px), phase text in DM Sans (body), not monospace
- Verify the TS conformance test (`tools/check_schemas_fresh.py`) passes after schema regeneration
