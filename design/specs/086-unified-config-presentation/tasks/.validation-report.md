## Warnings

These findings are non-blocking and do not change the APPROVED status.

### W1: FR#11 frontend half traceable only through AC#10, not through `implements`

**Location:** T06 `implements` field.

**Finding:** FR#11 covers both the backend mechanism ("metadata survives schema export") and the frontend consumption ("renderer applies each hint when present and falls back"). T05 claims FR#11 and verifies the backend half. T06 verifies the frontend rendering half under AC#10 — but FR#11 does not appear in T06's `implements` list. Traceability from FR#11 to T06 is indirect: FR#11 maps to AC#10 (per the design doc), and T06 implements AC#10.

**Risk:** Low — the full requirement is effectively covered; an implementer reading only T06's implements list would not see FR#11 explicitly, but the design doc's "maps FR#11" annotation on AC#10 connects them. A future executor should not be confused, but the traceability is one hop longer than ideal.

**Suggested fix (optional):** Add `"FR#11"` to T06's `implements` list to make the frontend-renderer half of FR#11 directly traceable, since T06's Verify for AC#10 already covers it.

---

### W2: AC#6 is split across T03 and T06 with each covering a different half

**Location:** T03 Verify, T06 Verify.

**Finding:** The design doc's AC#6 has two distinct sub-conditions: (a) `*ConfigResponse` classes and `config_response_from` no longer exist (backend), and (b) `pages/config.tsx` no longer contains a hand-written `groups` array (frontend). T03's Verify item covers only (a); T06's Verify item covers only (b). The split is correct and the full AC is satisfied across the two tasks, but neither task's Verify item quotes the full AC#6 text, so a reviewer auditing T03 alone would not see that the frontend half is deferred to T06.

**Risk:** Very low — the dependency chain (T06 depends on T03, T04, T05) ensures the full AC is verified before merge.

**Suggested fix (optional):** Add a parenthetical to each task's AC#6 Verify item noting which half it covers, e.g. "AC#6 (backend half): …" / "AC#6 (frontend half): …".

---

### W3: T03 Verify combines FR#12 and AC#11 into a single item

**Location:** T03 Verify: `- [ ] FR#12 / AC#11: hassette config (human and --json) …`.

**Finding:** FR#12 and AC#11 are separate identifiers in the design doc. They describe nearly identical success criteria (the CLI shows complete groups and masks the token), so collapsing them is harmless in practice, but the combined item does not explicitly call out the `--json` mode separately as a distinct binary-verifiable check (FR#12 says "both human and `--json` output"; the combined item does include "human and `--json`" in the description, so this is covered).

**Risk:** Negligible.

---

### W4: HTML mockup required by architecture section has no explicit Verify criterion in T06

**Location:** T06 Prompt step 1 vs. T06 Verify section.

**Finding:** The design doc's Architecture section states: "The first build task produces an HTML mockup of the full `HassetteConfig` render against this bar before the renderer is locked." T06 Prompt step 1 says: "Produce an HTML mockup … Save it under the feature dir or `/tmp` and reference it in the PR." However, T06's Verify section contains no item asserting the mockup was produced and referenced in the PR. The Verify items jump straight to the final component assertions (FR#6, FR#7, etc.).

**Risk:** Low — the mockup is a quality gate (preventing a "JSON blob" renderer from being locked), not a functional requirement with an FR/AC identifier. If an executor skips it, nothing in the Verify checklist catches the omission.

**Suggested fix (optional):** Add a Verify item to T06: `- [ ] (quality gate) An HTML mockup of the full HassetteConfig render was produced and referenced in the PR before the shared component was locked.`
