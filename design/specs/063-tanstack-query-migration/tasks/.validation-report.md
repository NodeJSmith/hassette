### Contradictions

#### CONTRADICTION — T03 Verify item for AC#11 is inverted

**AC#11 from design doc:**
> "After a WebSocket reconnection, all data visible on the current page refreshes immediately (within one render cycle, not debounced) — maps to FR#6"

AC#11 is about reconnect triggering an immediate refresh. The correct behavior to verify is that reconnect DOES call `invalidateQueries()`.

**T03 Verify item:**
> `- [ ] AC#11: first connect does NOT call invalidateQueries() — verified by unit test checking spy was not called on initial connection`

This criterion tests the **absence** of a call on first connect — which is the correct behavior for FR#6's second bullet ("On first connect, `invalidateQueries()` is NOT called") but is mislabeled as AC#11. AC#11 requires verifying the **presence** of invalidation on reconnect — which is what the **FR#6** verify item actually covers.

In other words:
- FR#6 Verify item tests the correct thing for AC#11 (reconnect fires `invalidateQueries`)
- AC#11 Verify item tests the correct thing for first-connect, but is attributed to AC#11 instead of being an additional coverage note for FR#6

**Effect:** AC#11 has no verify item that asserts the reconnect does trigger a refresh. The AC#11 verify item asserts a different (correct but separate) behavioral invariant and mislabels it. This creates a gap where the implementer could verify "first connect does not call invalidateQueries" and mark AC#11 complete without ever verifying that reconnect actually causes the data refresh.

**Suggested fix:** Replace the AC#11 verify item with:
```
- [ ] AC#11: on WebSocket reconnect, `queryClient.invalidateQueries()` is called and all active queries are refetched — verified by spy-based unit test showing all active queries become stale after reconnect
```
And retain the first-connect behavior as an additional unlabeled note or a second line under FR#6.

### Vague Verification Criteria

#### WARNING — T01 FR#4 Verify item is not binary-verifiable

**T01 Verify:**
> `- [ ] FR#4: @tanstack/preact-query is installed and QueryClientProvider wraps the app tree — verified by app.tsx rendering without errors`

FR#4 states: "Unmounting a component or changing query parameters cancels any in-flight request for the previous query." The T01 verify item only checks that the provider is installed and renders — it does not verify cancellation behavior. T06 has a more relevant verify item for FR#4 ("changing `appKey` in app-detail cancels the previous in-flight fetch"), but T01 claims FR#4 in its `implements` field with a criterion that only verifies the setup prerequisite, not the requirement itself.

This is not wrong (T01 does create the infrastructure that enables FR#4), but the criterion does not let an implementer confirm the actual FR is met from T01 alone. The combination of T01+T06 together covers FR#4, but the T01 verify item is misleadingly labeled as verifying FR#4 when it only verifies setup.

**Recommendation:** Rename the T01 verify item from `FR#4:` to an infrastructure note (e.g., drop the FR#4 label or change it to "Infrastructure for FR#4: `QueryClientProvider` wraps the app tree — prerequisite for request cancellation").

---

#### WARNING — T06 FR#4 Verify item relies on library behavior without observable test evidence

**T06 Verify:**
> `- [ ] FR#4: changing appKey in app-detail cancels the previous in-flight fetch — verified by TanStack Query's built-in behavior (query key change triggers abort)`

"Verified by TanStack Query's built-in behavior" is not an observable test criterion — it is an assertion of trust in the library. The AC this covers (AC#9: "navigating away from a page while a fetch is in-flight produces no errors, console warnings, or state writes to the unmounted component") has its own verify item. But FR#4 specifically requires cancellation on parameter change, and the verify criterion does not describe what would be observed in a test to confirm it.

**Recommendation:** Make it testable: "changing `appKey` during an in-flight fetch causes the previous request to be aborted — verified by MSW recording no completion of the aborted request and no console errors".

---

#### WARNING — T06 FR#2 Verify item includes a "manual DevTools check" escape hatch

**T06 Verify:**
> `- [ ] FR#2: navigating away from handlers/apps/app-detail and back within 30s serves cached data — verified by test or manual DevTools check`

The "or manual DevTools check" alternative makes this non-deterministic and not suitable for CI. AC#2 has a proper verify item ("verified by test") for the same behavior. The FR#2 verify in T06 should match that precision.

**Recommendation:** Remove "or manual DevTools check" — tests should be the sole verification method.

#### WARNING — T07 implements FR#5 but the verify criterion is code-review-only

**T07 Verify:**
> `- [ ] FR#5: all three components use useQueryInvalidator with the correct debounce timing (500ms/1500ms) — verified by code review confirming WS_DEBOUNCE_DELAY_MS and WS_DEBOUNCE_MAX_WAIT_MS are passed correctly`

Code review of constant names cannot confirm the hook applies the debounce correctly — it only confirms the constants are threaded through. T02 has timer-based tests for the invalidator itself, but T07 adds no behavioral test that the three migrated components actually produce the correct invalidation timing. If `useQueryInvalidator` itself has a bug, the T07 verify passes (correct constant names) while FR#5 fails.

This is acceptable if T02's tests are considered sufficient coverage for the invalidation logic, but noting it here because the criterion describes evidence that is shallower than the requirement.

---

#### WARNING — AC#5 Verify item includes conditional hedge

**T09 Verify:**
> `- [ ] AC#5: E2E test suite passes (run uv run nox -s e2e if available)`

The "if available" qualifier makes this non-deterministic. AC#5 from the design doc states it unconditionally: "All existing E2E tests pass — no visual or behavioral regressions." The design doc's pre-ship verification section also lists E2E as mandatory for frontend changes. The hedge should be removed, or the criterion should state the explicit condition under which E2E is skipped (e.g., "Playwright not installed on this machine — CI covers this").
