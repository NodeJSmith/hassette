## Objectives & Success Criteria

- All helper functions from `web/ui/context.py` are extracted to `src/hassette/web/telemetry_helpers.py` without behavior changes. Existing tests (`tests/unit/test_context_helpers.py`, `tests/integration/test_web_ui.py`) pass unchanged.
- Typed Pydantic models exist for all 6 WebSocket message types (`AppStatusChangedWsMessage`, `LogWsMessage`, `ConnectedWsMessage`, `ConnectivityWsMessage`, `StateChangedWsMessage`, `ServiceStatusWsMessage`) with a `WsServerMessage` discriminated union.
- `ConnectedPayload` includes `session_id` field. `ws.py` sends `session_id` in the `connected` message.
- `RuntimeQueryService` broadcast methods (`_on_app_state_changed`, `_on_state_change`, `_on_service_status`) construct validated Pydantic models via `.model_validate()` and serialize via `.model_dump()` instead of raw `_serialize_payload()`.
- All WS messages use a consistent `{ type, data, timestamp }` envelope. `_on_state_change` is normalized to wrap `entity_id`/`new_state`/`old_state` in a `data` field.
- Typed `HandlerErrorEntry` and `JobErrorEntry` models exist with `RecentErrorEntry` discriminated union.
- `scripts/export_schemas.py` exports `openapi.json` and `ws-schema.json` from the Pydantic models using a stub config (no HA connection required).
- Hand-authored `frontend/src/api/ws-types.ts` exists (~30 lines) with a CI conformance test validating it matches the exported `ws-schema.json`.

## Test Strategy

- **Existing tests pass unchanged**: `tests/unit/test_context_helpers.py`, `tests/integration/test_web_ui.py`, `tests/integration/test_web_api.py`.
- **New test file**: `tests/unit/test_ws_models.py`
  - `test_app_status_changed_payload_matches_dataclass` — construct `AppStateChangePayload` → `asdict()` → `AppStatusChangedPayload.model_validate()` succeeds.
  - `test_state_changed_normalized_envelope` — validate `_on_state_change` now produces `{ type, data: { entity_id, new_state, old_state }, timestamp }`.
  - `test_connected_payload_includes_session_id` — mock `runtime.hassette.session_id = 42`, verify `connected` message data contains `session_id: 42`.
  - `test_ws_server_message_discriminates` — parse each message type through `WsServerMessage` and verify type narrowing.
- **Schema conformance test**: `tests/unit/test_ws_schema_conformance.py`
  - `test_ws_types_match_schema` — load `ws-schema.json`, verify all message type names and required fields match the hand-authored TS types.
