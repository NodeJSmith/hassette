## Objectives & Success Criteria

- Apps list page (`/apps`) renders manifest list with status filter tabs, action buttons (start/stop/reload), and live status updates via WS.
- App Detail page (`/apps/:key`) renders health strip, expandable handler rows with lazy-loaded invocation history, expandable job rows with lazy-loaded execution history, and app-scoped log viewer.
- Expanding a handler/job row and receiving a WS `app_status_changed` event does NOT collapse the row — this is the core problem the migration solves.
- Handler summaries display human-readable descriptions from the server (`format_handler_summary` via telemetry endpoint).
- Action buttons call existing `/api/apps/{key}/start|stop|reload` endpoints and update status signal on response.
- Log viewer streams logs via WS subscription (`subscribe` message), renders from ring buffer, supports level/search filtering client-side.

## Subtasks

1. Create `frontend/src/components/apps/status-filter.tsx` — tab bar with all/running/failed/stopped/disabled/blocked filters.
2. Create `frontend/src/components/apps/action-buttons.tsx` — start/stop/reload buttons calling `/api/apps/{key}/start|stop|reload`. Disable during request, show success/error feedback.
3. Create `frontend/src/components/apps/manifest-row.tsx` — single manifest row with status badge, instance count, action buttons.
4. Create `frontend/src/components/apps/manifest-list.tsx` — filterable manifest table. Reads `appStatus` signal for live badge updates.
5. Create `frontend/src/pages/apps.tsx` — composes status filter + manifest list. Fetches `/api/apps/manifests`. Subscribes to `app_status_changed` for live status updates.
6. Create `frontend/src/components/app-detail/health-strip.tsx` — 4 health KPI cards using `/api/telemetry/app/{key}/health`.
7. Create `frontend/src/components/app-detail/handler-row.tsx` — expandable row with local `expanded` signal. Lazy-loads invocation history from `/api/telemetry/handler/{id}/invocations` on first expand.
8. Create `frontend/src/components/app-detail/handler-invocations.tsx` — invocation history table inside expanded handler row.
9. Create `frontend/src/components/app-detail/handler-list.tsx` — list of handler rows from `/api/telemetry/app/{key}/listeners`.
10. Create `frontend/src/components/app-detail/job-row.tsx` — expandable job row with local `expanded` signal. Lazy-loads execution history.
11. Create `frontend/src/components/app-detail/job-executions.tsx` — execution history table.
12. Create `frontend/src/components/app-detail/job-list.tsx` — list of job rows from `/api/telemetry/app/{key}/jobs`.
13. Create `frontend/src/components/shared/log-table.tsx` — log viewer component. Reads from ring buffer via version signal. Supports level filter, app filter, search. Sends WS `subscribe` message for real-time log streaming.
14. Create `frontend/src/pages/app-detail.tsx` — composes health strip + handler list + job list + log table. Fetches all data sources. Route param `:key` from wouter.
15. Create `frontend/src/utils/format.ts` — `formatTimestamp`, `formatDuration`, `formatRelativeTime` helpers.

## Test Strategy

- **Vitest component tests**:
  - `handler-row.test.tsx`: render collapsed, click to expand, verify invocation fetch triggered. Re-render parent (simulating WS update), verify `expanded` state preserved.
  - `manifest-list.test.tsx`: render with mixed statuses, apply filter, verify correct rows visible.
  - `action-buttons.test.tsx`: click stop, verify API call made, verify button disabled during request.
  - `log-table.test.tsx`: render with mock ring buffer, push new entry, verify it appears without full re-render.
- **Critical regression test** in `handler-row.test.tsx`:
  - `test_expanded_row_survives_parent_rerender` — expand a handler row, update the parent's app status signal (simulating WS event), verify the row stays expanded and invocation history is preserved. This validates the core architectural win.

## Review Guidance

- **THE CRITICAL TEST**: Verify `handler-row.tsx` uses a local signal for `expanded` state, not a prop from the parent. The parent re-rendering (from WS update) must NOT reset the local signal.
- Verify lazy-loaded content (invocations, executions) is fetched on first expand and cached — not re-fetched on every expand/collapse.
- Verify log table reads from the ring buffer via version signal pattern (not array copy).
- Verify action buttons call the correct existing API endpoints (not new ones).
- Verify status filter is client-side (no server round-trip for filtering).
