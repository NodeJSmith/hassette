## Objectives & Success Criteria

- 8 new JSON endpoints exist under `/api/telemetry/*`, all returning typed Pydantic response models.
- All endpoints resolve `session_id` server-side via `safe_session_id(runtime)` from `telemetry_helpers.py`.
- Dashboard data is 3 separate endpoints (`/kpis`, `/app-grid`, `/errors`), not a single combined endpoint.
- Classification labels (`health_status`, `error_rate_class`) are returned alongside raw numbers — no business logic on the client.
- `instance_index` defaults to 0 and is an optional query parameter.
- Integration tests cover every endpoint with expected response shapes, status codes, and query parameter behavior.
- Telemetry router is registered outside the `run_web_ui` guard (available in headless mode).

## Subtasks

1. Create response models in `src/hassette/web/models.py`: `AppHealthResponse` (with `health_status`, `error_rate_class` labels), `DashboardKpisResponse`, `DashboardAppGridResponse`, `DashboardErrorsResponse`.
2. Create `src/hassette/web/routes/telemetry.py` with 8 endpoints:
   - `GET /api/telemetry/app/{app_key}/health` — calls `get_listener_summary` + `get_job_summary` + `compute_health_metrics` from `telemetry_helpers.py`. Returns health metrics with `classify_error_rate` and `classify_health_bar` labels.
   - `GET /api/telemetry/app/{app_key}/listeners` — calls `get_listener_summary`, enriches with `format_handler_summary`.
   - `GET /api/telemetry/app/{app_key}/jobs` — calls `get_job_summary`.
   - `GET /api/telemetry/handler/{listener_id}/invocations` — calls `get_handler_invocations`.
   - `GET /api/telemetry/job/{job_id}/executions` — calls `get_job_executions`.
   - `GET /api/telemetry/dashboard/kpis` — calls `get_global_summary` with `session_id`.
   - `GET /api/telemetry/dashboard/app-grid` — calls `get_all_app_summaries` + manifests with `session_id`.
   - `GET /api/telemetry/dashboard/errors` — calls `get_recent_errors` with `session_id`, returns typed `RecentErrorEntry` list.
3. Register telemetry router in `web/app.py` OUTSIDE the `run_web_ui` guard (after API routes, before the UI block).
4. Write integration tests in `tests/integration/test_web_api.py` (extend existing file).

## Test Strategy

- **New tests in `tests/integration/test_web_api.py`** (extending existing test file and fixtures):
  - `TestTelemetryAppHealth`:
    - `test_app_health_returns_metrics_with_classification` — verify response includes `error_rate`, `error_rate_class`, `health_status`.
    - `test_app_health_unknown_app_returns_empty` — verify graceful handling.
    - `test_app_health_instance_index_param` — verify optional `instance_index` query param.
  - `TestTelemetryListeners`:
    - `test_listeners_returns_summaries_with_handler_descriptions` — verify `format_handler_summary` enrichment.
  - `TestTelemetryDashboard`:
    - `test_dashboard_kpis_returns_global_summary` — verify response shape.
    - `test_dashboard_app_grid_returns_per_app_health` — verify `AppHealthSummary` per app.
    - `test_dashboard_errors_returns_typed_entries` — verify discriminated union (`kind: "handler"` vs `kind: "job"`).
  - `TestTelemetryAvailableWithoutUI`:
    - `test_telemetry_endpoints_work_when_run_web_ui_false` — verify headless mode access.

## Review Guidance

- Every telemetry endpoint MUST call `safe_session_id(runtime)` — check that no endpoint omits session scoping.
- Dashboard endpoints must be 3 separate routes, not one combined endpoint.
- Classification labels must come from `classify_error_rate` / `classify_health_bar` in `telemetry_helpers.py` — no duplication.
- Response models must be in `web/models.py` (not inline) so `openapi-typescript` can generate types.
- Telemetry router must be outside the `if hassette.config.run_web_ui:` block in `app.py`.
