## Objectives & Success Criteria

- Dashboard page (`/`) renders KPI strip, app grid, and error feed using data from the 3 separate telemetry endpoints.
- All 3 data sources are fetched in parallel via `Promise.all` on page mount.
- Each panel renders progressively as its data arrives (not waiting for all 3).
- `app_status_changed` WS events trigger a targeted refetch of the app-grid endpoint only (not all 3).
- App cards show status badge, health bar, and per-app health summary.
- Error feed shows typed error entries (handler vs job) with timestamps.
- KPI strip shows classification labels mapped to CSS classes (server-computed labels, client CSS mapping only).
- Theme toggle (dark/light) works via `data-theme` attribute on root element.

## Subtasks

1. Create `frontend/src/components/shared/status-badge.tsx` — renders status dot + label, accepts `status` and optional `size` props. Uses `ht-status-badge` global class.
2. Create `frontend/src/components/shared/health-bar.tsx` — renders success rate bar with classification class. Uses `ht-health-bar` global class.
3. Create `frontend/src/components/shared/spinner.tsx` — loading indicator for initial data fetch.
4. Create `frontend/src/components/dashboard/kpi-strip.tsx` — 4 KPI cards (app count, error rate, avg duration, last activity). Maps server classification labels to CSS classes.
5. Create `frontend/src/components/dashboard/app-card.tsx` — single app health card with status badge, health bar, handler/job counts.
6. Create `frontend/src/components/dashboard/app-grid.tsx` — grid of app cards. Receives app health data as prop.
7. Create `frontend/src/components/dashboard/error-feed.tsx` — recent errors list. Renders `HandlerErrorEntry` and `JobErrorEntry` with distinct icons/labels.
8. Create `frontend/src/pages/dashboard.tsx` — composes KPI strip + app grid + error feed. Fetches `/api/telemetry/dashboard/kpis`, `/api/telemetry/dashboard/app-grid`, `/api/telemetry/dashboard/errors` in parallel. Subscribes to `app_status_changed` WS events for targeted app-grid refetch.
9. Create CSS module files for any component-local styles (internal layout, animation states). Use global `ht-*` classes for design system primitives.
10. Wire theme toggle: read/write `theme` signal from `useAppState()`, apply `data-theme` attribute on `<html>`.

## Test Strategy

- **Vitest component tests** in `frontend/src/components/dashboard/`:
  - `kpi-strip.test.tsx`: render with mock KPI data, verify 4 cards present with correct labels and classification classes.
  - `app-card.test.tsx`: render with running/failed/stopped status, verify correct badge variant and health bar class.
  - `error-feed.test.tsx`: render with mixed handler/job errors, verify correct `kind` labels.
- **Vitest page test** in `frontend/src/pages/`:
  - `dashboard.test.tsx`: mock API endpoints, verify all 3 panels render after data loads. Verify loading spinners show during fetch.

- Verify `ht-*` global classes are used for design system primitives, not duplicated in component CSS modules.
