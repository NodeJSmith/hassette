## Objectives & Success Criteria

A `useDebouncedEffect` hook provides generic trailing debounce for signal changes. The dashboard page debounces all three data fetches (KPIs, app grid, errors) on `appStatus` changes with 500ms delay. Reconnection refetches remain immediate (via `reconnectVersion`). Unit tests pass for the debounce hook. E2E test verifies debounced behavior.

## Subtasks

1. Create `frontend/src/hooks/use-debounced-effect.ts` with:
   - `useDebouncedEffect(getValue: () => unknown, delayMs: number, callback: () => void): void` â€” watches the return value of `getValue()`, and when it changes, starts a 500ms trailing debounce timer. On timeout, calls `callback()`. Cleans up timer on unmount and on new change (restarts timer).
   - Uses `useEffect` internally with the value as a dependency, plus `useRef` for the timer.
2. Write `frontend/src/hooks/use-debounced-effect.test.ts` â€” unit tests using vitest with fake timers.
3. **Update dashboard** in `frontend/src/pages/dashboard.tsx`:
   - Remove the current `useEffect` block (lines 22â€“31) that watches `lastStatus` and calls `appGrid.refetch()`.
   - Import `useDebouncedEffect`.
   - Add: `useDebouncedEffect(() => appStatus.value, 500, () => { void kpis.refetch(); void appGrid.refetch(); void errors.refetch(); })`.
   - Remove the `lastStatus` const and `refetchAppGrid` callback (no longer needed).

These must remain independent code paths.

## Pitfalls to Avoid

- **Do NOT debounce `reconnectVersion` changes.** The debounce applies only to `appStatus` signal changes. Reconnection refetches go through a separate code path (`useApi`'s `useSignalEffect` on `reconnectVersion`). If someone later routes reconnection through `appStatus` (e.g., "all apps changed status"), the debounce would silently eat the reconnection refetch. These two signals must never be conflated.

## Review Guidance

- The invariant comment on `create-app-state.ts` must explain why the two signals are separate
