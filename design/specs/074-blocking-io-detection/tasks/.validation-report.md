## Coverage Gaps

**AC#6 (row/warning suppression, end-to-end):** The design's AC#6 reads in full:

> Setting `blocking_io_behavior` on an app overrides the global `blocking_io.behavior` for that app only; **`ignore` suppresses both the warning and the DB row for that app.**

T01 covers the resolver unit-test half and explicitly defers the row/warning suppression to "T06 end-to-end." T06 includes a per-app `ignore` end-to-end test in its Prompt (step 2), which would satisfy the DB-row half of AC#6. However, T06's `implements` list is `["AC#4", "FR#8"]` — it does NOT list `AC#6`. Consequently, T06 has no `- [ ] AC#6:` Verify item. The DB-row-suppression half of AC#6 is covered in T06's Prompt text but is not tracked in any task's `implements` + Verify pair. This is a traceability gap: the criterion is partially covered in prose but not formally claimed.

**AC#6 (no task fully owns the complete criterion):** T01 owns the resolver half and says row suppression is "verified end-to-end in T06", but T06 does not declare AC#6 in `implements` and has no AC#6 Verify item. The full acceptance criterion has no single task with both `implements: [..., "AC#6", ...]` and a matching `- [ ] AC#6:` Verify item that covers the complete criterion (resolver + row suppression).

## Warnings

1. **T02 `depends_on: []`** — T02 uses the `BlockingIOBehavior` enum and `HassetteConfig.blocking_io` fields defined in T01 (the resolver and config model). T02's Prompt says "Resolve behavior (T01 resolver)" and references `blocking_io` config. The dependency `T01 → T02` is implicit in the prose but not declared in `depends_on`. This does not block execution if the implementer reads both task files, but the dependency graph is incomplete as declared.

2. **T06 per-app `ignore` end-to-end test scope** — The T06 Prompt includes a per-app `ignore` end-to-end test (step 2: "confirm that with `blocking_io_behavior='ignore'` on an app, a genuine block in that app produces no warning and no row") that directly satisfies the missing half of AC#6. If T06's `implements` is updated to include `AC#6` and a Verify item is added, the gap in Finding #1 above is closed. The content of the test is already planned — the omission is purely in the traceability metadata.

3. **T06 `implements: ["AC#4", "FR#8"]` only** — T06 plans multiple correctness properties (per-app ignore suppression, docs, docstrings, config reference, doc reviews) but formally claims only AC#4 and FR#8. The docs and docstrings deliverables have no design-doc FR/AC counterpart to claim, so no gap exists for those. The per-app ignore gap is captured in Warning #2 above.

4. **Open Question on threshold defaults** — The design's Open Questions lists `lag_threshold_seconds` and `watchdog_interval_seconds` defaults as unresolved. T01 picks `~0.1` and `~0.25` as documented choices in the `BlockingIODetectionConfig` prompt; T03 notes "confirm them against real timing during this task and update the design's Open Question on defaults if they change." This is intentional design-defer, not a task gap, but implementers should be aware that T01's defaults are provisional.
