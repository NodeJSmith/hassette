## Objectives & Success Criteria

Wire the generator into the repo's automation: add a new pre-commit hook entry for the recording target, update the existing entry to use `--target api`, add the `git diff --exit-code` CI drift gate in `lint.yml`, land the required subprocess regression test, then delete the method-name drift test. The WP sequencing rule from the design (`design.md` → Pre-commit and CI integration → WP sequencing) is strict: gate and subprocess test land **before** the drift test deletion, all in this one WP.

Observable outcomes:

- `.pre-commit-config.yaml` has two generator hook entries at the `pre-push` stage:
  - The existing `generate_sync_facade` hook updated to pass `--target api` explicitly.
  - A new `generate_recording_sync_facade` hook with entry `python tools/generate_sync_facade.py --target recording`, `pass_filenames: false`, and `files:` regex matching all three sources: `(src/hassette/test_utils/recording_api\.py|src/hassette/api/api\.py|tools/generate_sync_facade\.py)`.
- `.github/workflows/lint.yml` has a new step after the `uvx pre-commit run` step that runs `git diff --exit-code` and prints the actionable remediation message on failure.
- New test file `tests/unit/test_recording_sync_facade_generation.py` contains `test_generator_check_mode_exits_zero` which invokes `tools/generate_sync_facade.py --target recording --check` via `subprocess.run` and asserts exit code 0.
- `tests/unit/test_recording_sync_facade_drift.py` is **deleted**.
- `uv run pytest tests/unit/test_recording_sync_facade_generation.py tests/unit/test_recording_sync_facade.py tests/unit/test_recording_api_write_parity.py -n 2 --dist loadscope -v` passes.
- `uv run pre-commit run --all-files --hook-stage pre-push` runs without drift (re-running the generator produces no changes to committed files).
- Running the pre-push hook after a no-op edit to `recording_api.py` (or `api.py`, or `generate_sync_facade.py`) triggers the recording facade hook. Verify by adding a whitespace change, running the hook, confirming the hook fires (either by output inspection or by checking the hook's `changes` tracking).
- The CI drift gate in `lint.yml` is invokable locally: `git diff --exit-code` returns 0 on a clean checkout after a successful pre-commit run.

## Subtasks

1. **Update the existing first hook entry** in `.pre-commit-config.yaml`. Change its `entry:` from `python tools/generate_sync_facade.py` to `python tools/generate_sync_facade.py --target api`. Keep `pass_filenames: false`, `files:` pattern, and `stages: [pre-push]` unchanged.
2. **Add the new hook entry** for the recording facade directly after the existing one, inside the `- repo: local` block:
   ```yaml
   - id: generate_recording_sync_facade
     name: Generate _RecordingSyncFacade
     language: python
     entry: python tools/generate_sync_facade.py --target recording
     pass_filenames: false
     files: (src/hassette/test_utils/recording_api\.py|src/hassette/api/api\.py|tools/generate_sync_facade\.py)
     stages:
       - pre-push
   ```
   Note the multi-line `files:` regex — use the exact pattern from the design doc. The backslashes before each `.` are regex literals for the dots.
3. **Add the CI drift gate** to `.github/workflows/lint.yml`. Insert a new step in the `python` job after the existing `SKIP=eslint uvx pre-commit run --all-files --hook-stage pre-push` line. The step runs a shell block that calls `git diff --exit-code` and prints an actionable error on failure:
   ```yaml
   - name: Fail if generators produced drift
     run: |
       if ! git diff --exit-code; then
         echo ""
         echo "ERROR: generated files are out of date. Re-run locally and commit:"
         echo "  uv run tools/generate_sync_facade.py --target both"
         echo ""
         exit 1
       fi
   ```
4. **Add `tests/unit/test_recording_sync_facade_generation.py`** (new file) with `test_generator_check_mode_exits_zero`:
   - Computes the repo root (via `Path(__file__).resolve().parents[2]` or similar).
   - Runs `subprocess.run(["uv", "run", "tools/generate_sync_facade.py", "--target", "recording", "--check"], cwd=repo_root, capture_output=True, timeout=60)`.
   - On non-zero exit, the assertion message includes the captured stderr so a developer debugging a CI failure sees what drifted.
   - Asserts `result.returncode == 0`.
   - Similar second test `test_generator_check_mode_api_exits_zero` for `--target api` — exercises the existing api-generation path in `--check` mode as a smoke test that the WP02 robustness fixes didn't break anything.
5. **Delete `tests/unit/test_recording_sync_facade_drift.py`** using `rm` or `git rm`. Before deletion, verify nothing else in the repo references it:
   - `grep -r 'test_recording_sync_facade_drift' .` — should return no results outside the deleted file
   - Additionally grep `conftest.py` files for any fixture imports or registrations that reference the drift test module: `grep -r 'recording_sync_facade_drift' tests/` (including any `conftest.py` in the tree). The drift test does not currently export fixtures, but a future addition could; this guard catches the case before it bites.
6. **Run the full test suite** to catch any regression: `uv run pytest tests/unit/test_recording_sync_facade.py tests/unit/test_recording_sync_facade_generation.py tests/unit/test_recording_api_write_parity.py tests/unit/test_recording_api.py tests/unit/test_generate_sync_facade.py tests/integration/test_app_test_harness.py -n 2 --dist loadscope -v`.
7. **Run the pre-commit hooks manually** to verify they're wired correctly:
   ```bash
   uv run pre-commit run --all-files --hook-stage pre-push
   ```
   Expected: both `generate_sync_facade` and `generate_recording_sync_facade` hooks run and produce no changes (the committed files are already current).
8. **Verify the hook triggers correctly**: touch `src/hassette/api/api.py` (e.g., add and remove a trailing newline), run the pre-commit hook, confirm the `generate_recording_sync_facade` hook fires (not just `generate_sync_facade`). The `files:` regex must include `api.py` so both hooks trigger on an `Api` change. If only the first hook fires, the regex is wrong — fix subtask 2.
9. **Verify the CI drift gate is functional**: after the pre-commit run completes with no changes, run `git diff --exit-code` in the repo. Should return 0. This simulates what the CI step does.

## Review Guidance

**Design constraints to verify:**

- The new pre-commit hook's `files:` regex matches **all three** source paths: `recording_api.py`, `api.py`, and `tools/generate_sync_facade.py`. Missing any of them is a Finding 2, 20, or the other silent-staleness regressions. Blocker.
- The CI drift gate step runs **after** `uvx pre-commit run`, not before. If the order is reversed, `git diff --exit-code` runs against pre-hook state and doesn't actually detect drift. Blocker.
- The CI drift gate's error message prints the remediation command (`uv run tools/generate_sync_facade.py --target both`). A generic `exit 1` with no guidance is a Finding 7 regression. Blocker.
- The subprocess regression test is actually **required** — not optional, not skipped, not marked with `@pytest.mark.skip`. If the WP delivers the test but the test is disabled, it's a Finding 11 regression. Blocker.
- The drift test is deleted **in the same WP** as the CI gate lands. If the reviewer sees a commit that deletes the drift test without the CI gate also being in place, that's a WP-sequencing violation. Blocker.
- The existing `generate_sync_facade` hook has `--target api` explicitly. If it's still the bare `python tools/generate_sync_facade.py`, the two hooks overlap confusingly. Warning (the default `--target both` would still work but pollutes the regenerated file set).
- No other file imports from the deleted `test_recording_sync_facade_drift.py`. Confirm via grep. Blocker if anything does.

**WP-sequencing (critical — re-verify)**:

The design's explicit required ordering inside this single WP:

1. lint.yml gets the `git diff --exit-code` step (subtask 3)
2. The generated file was already committed in WP03 (verified)
3. The subprocess regression test is added (subtask 4)
4. The drift test is deleted (subtask 5)

Reviewers must confirm the commit order respects this — ideally a single commit, or commits in this sequence. If WP04 lands as a branch, the branch's commit history should reflect it.

**Warning-level deviations (note but not block):**

- Whether the subprocess test uses `subprocess.run` or `subprocess.check_output`. Both work.
- Whether the CI drift gate is a separate step or appended to the pre-commit step's `run:` block. Separate is clearer but both work.
- Formatting of the `files:` regex (single-line vs multi-line) in `.pre-commit-config.yaml`. Both work.
