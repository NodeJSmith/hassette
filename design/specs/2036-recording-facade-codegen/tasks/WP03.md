3. **Import stub message constants in tests** (Finding 19). The generator emits `_STUB_MSG_STATE_CONVERSION` and `_STUB_MSG_GENERIC` as **top-level module constants in the generated `sync_facade.py`** (not as hidden string literals inside method bodies). Tests import these constants directly from `hassette.test_utils.sync_facade` — a normal intra-package import, no `sys.path` manipulation, no `conftest.py` hacks, no `tools/__init__.py`. Update `tests/unit/test_recording_sync_facade.py` tests that currently assert on inline substrings of error messages:
   - Change `assert "Seed state via" in str(exc.value)` → `assert str(exc.value) == _STUB_MSG_GENERIC.format(name="<method_name>")`
   - Import: `from hassette.test_utils.sync_facade import _STUB_MSG_GENERIC, _STUB_MSG_STATE_CONVERSION`
   - Source-of-truth is preserved: the generator's Python template constants and the generated file's module-level constants are byte-identical by construction. CI's `--check` mode fails loudly if they drift.
   - **Blocker verification for the reviewer** (WP02 / WP03 boundary): the generator's class header template and stub emit path must actually write `_STUB_MSG_STATE_CONVERSION = "..."` and `_STUB_MSG_GENERIC = "..."` as top-of-file assignments in the generated file. WP02's subtask 4 (`_RECORDING_CLASS_HEADER`) and subtask 11 (`generate_sync_recording`) must cooperate to emit these constants into the generated module header.
4. **Add the runtime coroutine + async-generator smoke test** to `tests/unit/test_recording_sync_facade.py` as `test_body_copied_methods_are_sync`:

   - Enumerate body-copied method names. Options for enumeration:


## Review Guidance

**Design constraints to verify:**

- The generated class header in the committed `sync_facade.py` exactly matches the "Class header template (pinned)" text in `design.md`. Diff the two literally. Any wording change is a blocker — either update the template in the design doc (with justification) or fix the generator. Blocker.
- All body-copied methods in the generated file contain **no** `await` keywords and **no** `async def`. If any survive, either WP02's rewriter is broken or a `RecordingApi` method slipped through without the WP01 refactor being applied. Blocker.
- The runtime smoke test (`test_body_copied_methods_are_sync`) actually exercises every body-copied method — not just turn_on. The reviewer should count the methods invoked in the test and compare against the body-copied method list in the generated file. Blocker if the count is short.
- The smoke test's failure message includes the method name. A bare `assert not True` is a Finding 21 regression. Blocker.
- The smoke test closes leaked coroutines via `result.close()`. A failing test that leaks coroutines will spam `RuntimeWarning` — Finding 21 regression. Blocker.
- The smoke test checks **both** `CoroutineType` and `AsyncGeneratorType`. Missing either is a Finding 18 regression. Blocker.
- The Api-vs-RecordingApi parity test lives in a **separate file** from the old drift test, so WP04's deletion does not take it down. Blocker if it's in `test_recording_sync_facade_drift.py`.
- Message-string assertions in `test_recording_sync_facade.py` import `_STUB_MSG_*` constants rather than duplicating substrings. If the reviewer sees inline substring assertions after this WP, that's a Finding 19 regression. Blocker for new tests; warning for any pre-existing inline assertions the WP left alone.

**Behavioral parity check (the manual one-shot from `design.md`):**

- The reviewer must walk the `git diff` of `src/hassette/test_utils/sync_facade.py` before/after WP03 and confirm every difference is either (a) a body-copy transformation, (b) a docstring preservation, (c) the pinned class header template, or (d) derived imports. Any other difference is a potential behavioral change and must be justified or fixed. This is a BLOCKER — the WP cannot merge without this review.

**Warning-level deviations (note but not block):**

- Whether the smoke test enumerates method names via introspection or a hardcoded list. Both work; the design prefers introspection if feasible.
- Whether the write-method parity test uses `ApiProtocol` or direct `Api` introspection. Both work.
