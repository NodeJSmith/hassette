## Objectives & Success Criteria

Prepare `RecordingApi` for body-copy codegen by eliminating all internal `await` calls from methods that will be body-copied into `_RecordingSyncFacade`. The refactor must be functionally equivalent to the current implementation â€” existing tests pass unchanged, plus a new regression test pins the `BaseEntity`-subclass behavior of `get_entity_or_none` before the inline rewrite.

Observable outcomes:

- `RecordingApi.get_entity_or_none` no longer calls `await self.get_entity(...)`; its body uses `self._get_raw_state(...)` + `self._convert_state(...)` directly, with the same `BaseEntity`-vs-`BaseState` branching behavior that `get_entity` provides.
- `RecordingApi.get_state_or_none` no longer calls `await self.get_state(...)`; its body uses `self._get_raw_state(...)` + `self._convert_state(...)` directly.
- The `async def` signatures of both methods remain unchanged (callers still `await` them).
- A new test `test_get_entity_or_none_returns_base_entity_subclass` in `tests/unit/test_recording_api.py` passes â€” calls `get_entity_or_none(entity_id, model=SomeLightEntity)` with a seeded state and asserts the returned object `isinstance(result, SomeLightEntity)`.
- A new test `test_get_entity_or_none_returns_none_for_missing_entity_with_model` passes â€” calls `get_entity_or_none("sensor.unseeded", model=SomeLightEntity)` and asserts `result is None`.
- A class-level docstring comment on `RecordingApi` documents the authoring constraint: "Methods on this class must not call other `async def` methods on `self` directly; use sync helpers (`_get_raw_state`, `_convert_state`) instead. Violating this constraint will fail the `_RecordingSyncFacade` generator with a clear error pointing at the offending call site."
- `uv run pytest tests/unit/test_recording_api.py tests/unit/test_recording_sync_facade.py -n 2 --dist loadscope` passes with no regressions.
- `uv run pyright src/hassette/test_utils/recording_api.py tests/unit/test_recording_api.py` is clean.
- `uv run ruff check src/hassette/test_utils/recording_api.py tests/unit/test_recording_api.py` is clean.

## Subtasks

1. Add the authoring-constraint note to the class docstring of `RecordingApi` in `src/hassette/test_utils/recording_api.py` (just after the existing "on_initialize() calls self.mark_ready()..." paragraph). Mention `_get_raw_state` and `_convert_state` as the sanctioned sync helpers.
2. Rewrite `RecordingApi.get_entity_or_none` body in `src/hassette/test_utils/recording_api.py`: replace `return await self.get_entity(entity_id, model)` with an inline try/except that:
   - Calls `self._get_raw_state(entity_id)` inside a try block
   - On `EntityNotFoundError`, returns `None`
   - Otherwise, applies the same `model is not BaseState and issubclass(model, BaseEntity)` branch that `get_entity` uses: `cast("BaseState", model.model_validate({"state": raw}))` for the BaseEntity path; `self._convert_state(raw, entity_id)` for the default path.
3. Rewrite `RecordingApi.get_state_or_none` body: replace `return await self.get_state(entity_id)` with an inline try/except around `self._get_raw_state(entity_id)` + `self._convert_state(raw, entity_id)`.
4. Add `test_get_entity_or_none_returns_base_entity_subclass` to `tests/unit/test_recording_api.py`. Use an existing `BaseEntity` subclass from `hassette.models.entities` (check `tests/unit/test_recording_api.py` and `src/hassette/models/entities/` for one already used in tests â€” prefer reuse over creating a new subclass). Seed state via `harness.set_state(...)` or equivalent, call `get_entity_or_none(entity_id, model=SomeEntity)`, assert `isinstance(result, SomeEntity)`.
5. Add `test_get_entity_or_none_returns_none_for_missing_entity_with_model` to the same file. Call `get_entity_or_none("sensor.unseeded", model=SomeEntity)` with no prior seeding and assert `result is None` (should not raise).
6. Verify both methods' existing tests still pass. Run the full `tests/unit/test_recording_api.py` file and fix any fallout.
7. Run the sync facade test file `tests/unit/test_recording_sync_facade.py` to confirm the refactor did not inadvertently change behavior visible through the sync facade (the hand-written sync facade still delegates to these methods indirectly â€” it should not break).

## Test Strategy

**Unit tests in `tests/unit/test_recording_api.py`:**

- `test_get_entity_or_none_returns_base_entity_subclass` (NEW) â€” seeds state for a known entity ID, calls `get_entity_or_none(entity_id, model=<BaseEntity subclass>)`, asserts the returned object is an instance of that subclass and has the expected state. Pins the `BaseEntity`-branch behavior that the inline refactor must preserve.
- `test_get_entity_or_none_returns_none_for_missing_entity_with_model` (NEW) â€” calls `get_entity_or_none("sensor.unseeded", model=<BaseEntity subclass>)` with no seeding, asserts `result is None` and no exception propagates.
- All existing tests in `tests/unit/test_recording_api.py` must continue to pass without modification. If any fail, the refactor is behaviorally divergent and must be fixed, not the tests.

**Regression coverage in `tests/unit/test_recording_sync_facade.py`:**

- No new tests in this file for WP01. The existing tests exercise the hand-written `_RecordingSyncFacade` which still delegates to `RecordingApi` read methods indirectly via its own hand-copied bodies. They should pass unchanged â€” if any fail, the WP01 refactor has altered externally-visible behavior that the sync facade depended on.

**Exclusions:**

- No tests for the generator or `_RecordingSyncFacade` in this WP. Those land in WP02 and WP03 respectively.
- No tests for the sync-only code path of `_get_raw_state` / `_convert_state` â€” they already have sync-path coverage via existing `get_state` / `get_states` tests.

## Review Guidance

**Design constraints to verify:**

- The refactored `get_entity_or_none` body must **not** contain `await` on any call to `self.get_entity` (or any other `RecordingApi` async method). It may still contain `await` in the `async def` signature and it may call `self._get_raw_state` and `self._convert_state` (both sync). This is the precondition for WP02's generator to body-copy the method without hitting its static async-call check.
- The refactored `get_state_or_none` body must not contain `await` on `self.get_state`. Same constraint.
- The `async def` signatures of both methods are unchanged â€” the `type[Any] = BaseState` default stays, the return type stays. Callers still `await` these methods.
- The class docstring authoring constraint note is present and names `_get_raw_state` / `_convert_state` explicitly. This is documentation that WP02's generator will reinforce with a static check â€” both must agree.
- The new BaseEntity regression test actually exercises the `issubclass(model, BaseEntity)` branch. If the test calls `get_entity_or_none` without a `model` argument, it is not testing what the WP requires.

**Blocker-level deviations (fix before merge):**

- Using `await self.get_entity(...)` or `await self.get_state(...)` anywhere in the refactored method bodies.
- Changing the `async def` signature or default arguments.
- Missing the BaseEntity regression test or testing the wrong code path.
- Adding new tests that mock `_get_raw_state` â€” the test must use real state seeding to pin the end-to-end behavior.

**Warning-level deviations (note but not block):**

- Renaming local variables. Style choices in the inline body are fine as long as the behavior matches `get_entity` exactly.
- Docstring wording variations for the class-level authoring-constraint note, as long as it names the two sync helpers.
