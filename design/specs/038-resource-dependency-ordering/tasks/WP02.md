## Objectives & Success Criteria

Add `depends_on: ClassVar[list[type[Resource]]] = []` to the `Resource` class. Implement `_auto_wait_dependencies()` and wire it into both `Resource.initialize()` and `Service.initialize()` (both `@final`) before lifecycle hook dispatch. Add `_skip_dependency_check` flag to `_HassetteMock` for test harness compatibility. All unit tests pass validating the auto-wait logic, timeout handling, shutdown detection, and harness bypass.

## Subtasks

1. Add `depends_on: ClassVar[list[type["Resource"]]] = []` to the `Resource` class body in `src/hassette/resources/base.py`, alongside existing ClassVars (`role`, `source_tier`).
2. Add `_auto_wait_dependencies()` async method to `Resource` in `src/hassette/resources/base.py`:
   - Early return if `self.depends_on` is empty
   - Early return if `getattr(self.hassette, '_skip_dependency_check', False)` is `True`
   - For each type in `self.depends_on`, find matching instances in `self.hassette.children` via `isinstance()`
   - Raise `RuntimeError` if no match found (naming the declaring class and missing type)
   - Call `self.hassette.wait_for_ready(deps)` with existing timeout/shutdown semantics
   - On `False` return: check `self.hassette.shutdown_event.is_set()` — if set, call `self.mark_not_ready("shutdown during dependency wait")` and return; if not set, raise `RuntimeError` naming the timed-out deps
3. In `Resource.initialize()` (line ~370 in base.py), insert `_auto_wait_dependencies()` call after `await self.handle_starting()` and before `await self._run_hooks(...)`. Wrap in try/except that calls `await self.handle_failed(exc)` before re-raising, matching the pattern used in `_serve_wrapper`.
4. In `Service.initialize()` (line ~510 in base.py), insert the same `_auto_wait_dependencies()` call with the same try/except pattern, after `await self.handle_starting()` and before `await self._run_hooks(...)`.
5. Add `_skip_dependency_check: bool = False` attribute to `_HassetteMock` in `src/hassette/test_utils/harness.py`, set to `True` in `__init__`.
6. Write unit tests in `tests/unit/test_resource_depends_on.py`:
   - `test_empty_depends_on_is_noop` — completes without calling `wait_for_ready`
   - `test_depends_on_waits_for_matching_instance` — calls `hassette.wait_for_ready` with the correct instance
   - `test_depends_on_missing_type_raises` — raises `RuntimeError` with actionable message
   - `test_depends_on_subclass_match` — `depends_on = [BaseType]` finds `ConcreteSubclass` child
   - `test_depends_on_multiple_matches` — all matching instances are waited on
   - `test_depends_on_timeout_raises` — raises `RuntimeError` when `wait_for_ready` returns `False` and shutdown not set
   - `test_depends_on_shutdown_marks_not_ready` — calls `mark_not_ready()` when shutdown is set and `wait_for_ready` returns `False`
   - `test_depends_on_timeout_calls_handle_failed` — verifies `handle_failed` is called before the `RuntimeError` propagates
   - `test_skip_dependency_check_bypasses` — `_skip_dependency_check = True` causes immediate return

## Test Strategy

All tests in `tests/unit/test_resource_depends_on.py`. Uses mock `Hassette` instances with controlled `children` lists and configurable `wait_for_ready` return values. Tests the auto-wait logic in isolation from real services.

## Review Guidance

- The auto-wait MUST be in `initialize()` (both Resource and Service versions), NOT in `before_initialize()` — this is the core design decision that makes it unbypassable
- The try/except wrapping `_auto_wait_dependencies()` MUST call `handle_failed()` before re-raising — without this, a timeout leaves the service stuck in STARTING forever
- `_skip_dependency_check` MUST be checked before the children lookup, not before `wait_for_ready` — the harness's mock objects would fail the `isinstance` check
- Verify both `Resource.initialize()` and `Service.initialize()` are updated identically
- The `NOTE: keep flag resets and child propagation in sync` comments in both methods should be extended to mention `_auto_wait_dependencies`
