## Objectives & Success Criteria

`tests/system/test_app_lifecycle.py` contains 7 passing tests that prove apps loaded from disk get working resources, fire lifecycle hooks, and operate in isolation from each other.

## Subtasks

1. Create `tests/system/test_app_lifecycle.py` with `pytestmark = [pytest.mark.system]` and these tests. Each test creates a `make_system_config` with `autodetect_apps=True` and `app_dir` pointed at the appropriate directory (either `system_app_dir` fixture or a `tmp_path` with inline app files):
   - `test_trivial_app_initializes` ŌĆö point `app_dir` at `system_app_dir`, load `trivial_app.py`, verify `hassette.app_handler.apps` contains an entry with the app's key and the app's status is RUNNING (or equivalent running check)
   - `test_app_gets_working_api` ŌĆö write an inline app to `tmp_path` that calls `await self.api.get_states()` in `on_initialize` and stores the result in `self.fetched_states`, verify `len(app_instance.fetched_states) > 0` after startup
   - `test_app_bus_handler_fires` ŌĆö point `app_dir` at `system_app_dir` to load `bus_handler_app.py`, toggle `light.kitchen_lights` via hassette's API, `wait_for` until the app's `captured_events` list is non-empty
   - `test_app_scheduler_fires` ŌĆö write an inline app that schedules `self.scheduler.run_in(self._callback, 1)` in `on_initialize` and appends to `self.fired`, `wait_for` until `app.fired` is non-empty (timeout 5s)
   - `test_app_state_access` ŌĆö write an inline app that reads `list(self.states.light)` in `on_initialize` and stores the result, verify the stored list is non-empty after startup
   - `test_app_shutdown_hook` ŌĆö write an inline app that sets `self.shutdown_called = True` in `on_shutdown`, verify the flag is True after `startup_context` exits
   - `test_multiple_apps_isolation` ŌĆö write two inline apps to `tmp_path`: AppA registers `bus.on_state_change("light.kitchen_lights")` and appends to `self.captured`, AppB initializes `self.captured = []` but registers no handler. Toggle `light.kitchen_lights`, `wait_for` until `len(app_a.captured) >= 1`, then `await asyncio.sleep(2)` and assert `len(app_b.captured) == 0` ŌĆö proving the bus does not leak events to apps that didn't subscribe
2. For tests that use inline apps, write the app class as a Python string to a `.py` file in `tmp_path / "apps"`, then set `app_dir` to that directory. Verify each inline app actually loaded by checking `hassette.app_handler.apps` after startup.
3. Run `timeout 300 uv run pytest tests/system/test_app_lifecycle.py -m system -v -x -n 0 --tb=short` and verify all tests pass

## Test Strategy

The 7 tests verify the app lifecycle ŌĆö the core user experience of hassette:
- `test_trivial_app_initializes` ŌĆö app discovery and initialization from committed fixture
- `test_app_gets_working_api`, `test_app_state_access` ŌĆö per-app resources are functional
- `test_app_bus_handler_fires` ŌĆö per-app bus receives real HA events
- `test_app_scheduler_fires` ŌĆö per-app scheduler fires against real event loop
- `test_app_shutdown_hook` ŌĆö `on_shutdown` lifecycle hook fires during clean shutdown
- `test_multiple_apps_isolation` ŌĆö apps don't leak events to each other

## Review Guidance

- Every test that uses inline apps must verify the app actually loaded (`hassette.app_handler.apps` check) ŌĆö a test that passes because the app failed to load is a false positive
- Inline app strings must be valid Python ŌĆö verify imports are correct and no f-string interpolation issues
- `test_multiple_apps_isolation` must wait long enough for AppB to NOT receive events (brief `asyncio.sleep` after AppA captures, then assert AppB is empty)
- App fixture files from `tests/system/apps/` should not be modified ŌĆö use them as-is
- Verify `autodetect_apps=True` is set in configs that need app loading (the default `make_system_config` sets it to `False`)
