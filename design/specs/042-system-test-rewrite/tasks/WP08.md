## Objectives & Success Criteria

`tests/system/test_reconnection.py` contains 2 passing tests that prove hassette recovers from an HA connection drop. `tests/TESTING.md` is updated with a System Tests section documenting the suite organization, how to run tests, and the fixture app pattern.

## Subtasks

1. Create `tests/system/test_reconnection.py` with `pytestmark = [pytest.mark.system]` and these tests:
   - `test_websocket_reconnects_after_ha_restart` — start hassette, verify connected (`_session_ready`), run `docker pause hassette-system-ha` via `subprocess.run`, `wait_for(lambda: not hassette.websocket_service.is_ready(), timeout=15)` to confirm disconnect detected, run `docker unpause hassette-system-ha`, `wait_for(lambda: hassette.websocket_service.is_ready() and bool(hassette.websocket_service._subscription_ids), timeout=30)` to confirm reconnect with active subscriptions, then register a handler for `light.kitchen_lights`, toggle the light, `wait_for` event received — proving end-to-end event delivery works after reconnection
   - `test_state_proxy_refreshes_after_reconnect` — start hassette, read initial state of `light.kitchen_lights` from `state_proxy`, run `docker pause hassette-system-ha`, wait for disconnect, run `docker unpause hassette-system-ha`, wait for reconnect, `wait_for` until `state_proxy.states` is non-empty and `state_proxy.is_ready()` returns True, verify the proxy has a valid state for `light.kitchen_lights` (not stale — the state value should be a valid string, not None)
2. Add a `## System Tests` section to `tests/TESTING.md` documenting:
   - Purpose: tests that run against a real HA Docker container
   - How to run: `uv run nox -s system` (or `timeout 300 uv run pytest -m system -v -x -n 0`)
   - Organization: one file per user-visible subsystem
   - Infrastructure: `conftest.py` provides `ha_container` (session-scoped Docker lifecycle), `startup_context` (async context manager for Hassette lifecycle), `make_system_config` / `make_web_system_config` (config factories), `toggle_and_capture` (event capture helper)
   - App fixtures: committed apps in `tests/system/apps/` for common patterns; inline apps in `tmp_path` for test-specific variants
   - Key conventions: all polling via `wait_for`, no caplog assertions, `pytest.mark.system` on every file, tests independent of execution order
3. Run `timeout 300 uv run pytest tests/system/test_reconnection.py -m system -v -x -n 0 --tb=short` and verify both tests pass
4. Run the full suite: `timeout 300 uv run pytest -m system -v -x -n 0 --tb=short` and verify all ~43 tests pass (requires WP01-WP07 to be complete)

## Test Strategy

**test_reconnection.py** (2 tests):
- `test_websocket_reconnects_after_ha_restart` — full disconnect → reconnect → event delivery cycle
- `test_state_proxy_refreshes_after_reconnect` — cache validity after reconnection

**TESTING.md** update: documentation only, no testable code.

## Review Guidance

- `docker pause`/`docker unpause` must use the exact container name `hassette-system-ha` from `docker-compose.yml`
- Reconnection timeouts must be generous (30s for reconnect) — Docker unpause + WebSocket reconnect + event resubscription can take 10-15s
- The disconnect detection check (`not hassette.websocket_service.is_ready()`) must use `wait_for` — the WebSocket may not notice the pause immediately
- Verify `subprocess.run` calls for `docker pause`/`docker unpause` use `check=True` — if the Docker command fails, the test should fail immediately with a clear error, not hang waiting for a disconnect that won't happen
- The TESTING.md update should go in a natural location relative to existing sections (after the mock strategy sections, before or after E2E Seed Data)
- Verify the full suite pass (subtask 4) only after all WP01-WP07 are merged — this is the final validation step
