## Objectives & Success Criteria

`tests/system/test_web_api.py` contains 5 passing tests that verify the monitoring web API endpoints and WebSocket work against a fully-wired Hassette with a real HA backend. Tests use `make_web_system_config` from WP01.

## Subtasks

1. Create `tests/system/test_web_api.py` with `pytestmark = [pytest.mark.system]` and these tests. Each test uses `make_web_system_config` to get a config with `run_web_api=True` and a dynamic port, then `startup_context` to start hassette with the web server:
   - `test_health_endpoint` ‚Äî `httpx.AsyncClient` GET to `{base_url}/api/health`, verify status 200 and response JSON contains a connectivity status field
   - `test_apps_endpoint` ‚Äî GET `{base_url}/api/apps`, verify status 200 and response JSON has the expected list structure (may be empty if no apps loaded)
   - `test_config_endpoint` ‚Äî GET `{base_url}/api/config`, verify status 200 and response JSON contains hassette configuration fields
   - `test_telemetry_after_activity` ‚Äî register a bus handler for `light.kitchen_lights`, toggle the light, `wait_for` the handler to fire, then poll `GET {base_url}/api/dashboard/kpis` via `wait_for` until the response contains non-zero invocation count (the write pipeline is async, so the KPI may not update immediately)
   - `test_websocket_receives_events` ‚Äî connect to `{base_url}/api/ws` via `httpx` or `websockets` library, receive the initial `connected` message (verify `type == "connected"`), then toggle a light and `wait_for` an event message to arrive over the WebSocket. Close the connection cleanly.
2. Run `timeout 300 uv run pytest tests/system/test_web_api.py -m system -v -x -n 0 --tb=short` and verify all 5 tests pass

## Test Strategy

The 5 tests verify the web API monitoring layer works with a real backend:
- `test_health_endpoint`, `test_apps_endpoint`, `test_config_endpoint` ‚Äî basic REST endpoint availability and response structure
- `test_telemetry_after_activity` ‚Äî full pipeline: HA event ‚Üí bus handler ‚Üí command executor ‚Üí DB ‚Üí telemetry query ‚Üí API response
- `test_websocket_receives_events` ‚Äî real-time event streaming over WebSocket

## Review Guidance

- The web server needs time to start after `startup_context` yields ‚Äî add a brief `wait_for` on the health endpoint before running other tests, or accept that the first request may need a retry
- `test_websocket_receives_events` must handle the WebSocket protocol correctly: receive the initial `connected` message first, then wait for event messages. Use `asyncio.wait_for` with a timeout to avoid hanging if no events arrive
- Verify `make_web_system_config` is used (not `make_system_config`) ‚Äî the web server must be enabled
- HTTP requests should use `httpx.AsyncClient` for consistency with the rest of the codebase
- The dynamic port from `make_web_system_config` must be used in all request URLs ‚Äî do not hardcode port 8126
