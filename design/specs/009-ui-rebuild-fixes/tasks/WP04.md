## Objectives & Success Criteria

After this WP:
1. `manifest_list.html` renders rows unconditionally — the server filters via `?status=` param
2. `apps.html` uses `{% include "partials/manifest_list.html" %}` for initial render (single source of truth for row markup)
3. Tab clicks update the `data-live-on-app` URL dynamically via `x-bind` to include `?status=` param
4. After a WS-triggered refresh, the active tab filter is respected (server-side, not client-side)
5. `live-updates.js` re-reads the URL attribute at fire time (not enqueue time) to pick up tab changes within the debounce window

## Subtasks

1. Rewrite `src/hassette/web/templates/partials/manifest_list.html` to render full table rows matching the structure in `apps.html` — but WITHOUT Alpine `x-show` directives. The server filters via the `?status=` query param before the template renders. Include the expand/collapse chevrons for multi-instance apps.

2. Update `src/hassette/web/templates/pages/apps.html`:
   - Replace inline manifest row rendering with `{% include "partials/manifest_list.html" %}`
   - Keep the Alpine.js `x-data` for tab state: `{ activeTab: 'all' }`
   - Tab clicks: (a) set `activeTab`, (b) update `data-live-on-app` URL via `x-bind`, (c) trigger a fetch of the filtered partial to replace the tbody
   - Use `@click` on tabs to trigger `htmx.ajax()` with the filtered URL for immediate visual update (not waiting for a WS event)

3. Update `src/hassette/web/static/js/live-updates.js` to re-read the `data-live-on-app` attribute at fire time inside the `setTimeout` callback, not when `scheduleRefresh` is called. Change line ~91 from using the captured `refreshUrl` to re-reading `target.getAttribute("data-live-on-app")`.

4. Remove the Alpine `x-show` directives from the manifest rows in `apps.html` (they are no longer needed — server filters).
