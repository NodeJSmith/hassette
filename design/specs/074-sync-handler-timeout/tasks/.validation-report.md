## Warnings

1. **FR#8 verify split across T02 and T05.** T02's Verify covers config-field existence and the validator. T05's Verify covers the behavior change. Neither task implements both sides, but the dependency chain (T05 depends on T04, which depends on T03, which depends on T02) guarantees the config is present before the behavior is exercised. No gap, but a reviewer should confirm T05's config-behavior tests cite T02's fields explicitly.

2. **Shutdown ordering test not explicitly in a Verify item.** The design's Test Strategy calls out a specific regression test: "an AppSync shutdown hook submitting sync work during shutdown completes — no `RuntimeError: cannot schedule new futures`." T03's Verify focuses on graph structure (depends_on declarations), not the runtime race. T05's Verify covers the interruption behavior, not the "submit-to-alive-pool" ordering invariant. This test is described in T03's Prompt but absent from its Verify checklist. Not a coverage gap for an FR/AC, but the ordering regression test may be overlooked at done-check time.

3. **`async_raise` `res == 0` path.** T01's Verify says "`res == 0` and `res > 1` paths are handled (ValueError suppressed; SystemError revert path covered)." The design's edge-case note clarifies `res == 0` raises `ValueError` and suppression is intentional. The Verify criterion correctly captures this, but the wording "ValueError suppressed" without the explicit "thread not found" framing could be misread. No structural problem — just worth confirming the test explicitly exercises the `res == 0` path (not just the suppression path in `join_or_interrupt_threads`).

4. **context.md section 6 (`## Convention Examples`) is present and non-empty.** All five required sections (1–5) are also present and non-empty. No issues.

5. **Identifier format:** All FRs and ACs in the design doc use the correct `FR#N` / `AC#N` format. All task `implements` lists use the same format. No violations found.

6. **Scope check:** No task claims an FR or AC not present in the design doc. T07 implements `FR#3` only — correct, since it surfaces the leak marker in the UI, which is part of FR#3's "observable signal" requirement.
