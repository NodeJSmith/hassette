## Objectives & Success Criteria

The startup script replaces `uv sync` with the export-then-install pattern, gates requirements.txt behind `INSTALL_DEPS=1`, adds constraints enforcement, and includes all hardening measures from the design doc.

- Project-based install uses `uv export` → `uv pip install -r <deps> -c constraints.txt` → `uv pip install --no-deps <project>`
- Requirements.txt discovery only runs when `HASSETTE__INSTALL_DEPS=1` is set
- Only exact `requirements.txt` files are discovered (not `requirements-dev.txt` etc.)
- All `uv` network calls are wrapped with `timeout` and explicit exit-code handling (code 124 = actionable timeout message)
- Constraint conflicts produce a user-friendly error banner before the raw uv output
- `HASSETTE__ALLOW_UNLOCKED_PROJECT` detection logs a deprecation warning
- `pyproject.toml` without `uv.lock` logs a clear "run uv lock" message
- Version banner uses `importlib.metadata` and validates venv health
- `set -euo pipefail` is set
- `fd` binary check runs unconditionally (outside `INSTALL_DEPS` gate)
- `fd --max-depth 5` prevents deep traversal
- `ROOTS` uses a bash array (no word-split)
- fd loop uses `read -d ''` (NUL-safe)
- `shellcheck` passes cleanly

## Subtasks

1. Replace `set -eu` with `set -euo pipefail` at the top of `scripts/docker_start.sh`
2. Replace the `uv version --short` version banner with `importlib.metadata.version('hassette')` via a Python one-liner; exit with a clear error if the import fails
3. Add `INSTALL_DEPS` and `CONSTRAINTS` variable declarations; remove `ALLOW_UNLOCKED_PROJECT` variable
4. Move `FD_BIN` detection and check outside the `INSTALL_DEPS` block — warn (not error) if missing, since project-based installs don't need it
5. Add `HASSETTE__ALLOW_UNLOCKED_PROJECT` deprecation detection: if the variable is set, log a warning with migration instructions
6. Replace the `uv sync` project install block with the three-step export-then-install pattern: `uv export --frozen` → `uv pip install -r /tmp/user-deps.txt -c "$CONSTRAINTS"` → `uv pip install --no-deps "$PROJECT_DIR"`; add `--no-default-groups` to the export; wrap each step with `timeout`
7. Add the `pyproject.toml`-exists-but-no-lockfile branch that logs "run uv lock"
8. Replace the unconditional requirements discovery block with an `INSTALL_DEPS=1` gate; change the fd pattern from `'requirements' --extension txt` to exact `'requirements.txt'`; add `--max-depth 5`; wrap `uv pip install` with `timeout 120`
9. Add timeout exit-code handling: wrap each `timeout uv ...` call in a function or inline check that prints "ERROR: dependency install timed out after Ns" on exit code 124, or "ERROR: dependency install failed (exit N)" otherwise
10. Add the constraint-conflict error banner: when `uv pip install -r ... -c constraints.txt` fails (non-timeout), print the user-friendly DEPENDENCY CONFLICT banner before exiting
11. Remove the `uv pip install /app` version protection block and the "Completed sync of found project" / "Completed installation of N" messages; replace with accurate per-branch logging
12. Clean up the temp file: `rm -f /tmp/user-deps.txt` after the project install completes

## Test Strategy

The startup script is a bash script tested by the Docker integration tests in WP03. Unit-level verification for this WP:

- `shellcheck scripts/docker_start.sh` must pass with zero errors/warnings (run as part of pre-commit hooks)
- Manual review of the script structure against the design doc's startup flow diagram

## Review Guidance

- Verify every `uv pip install` call passes `-c "$CONSTRAINTS"`
- Verify `uv sync` does not appear anywhere in the script
- Verify `--no-default-groups` is on both `uv export` calls (locked and potential future unlocked paths)
- Verify the fd pattern matches only `requirements.txt` exactly, not `requirements-dev.txt`
- Verify `timeout` wraps all three steps of the project install and each requirements install
- Verify the deprecation warning for `ALLOW_UNLOCKED_PROJECT` mentions both migration paths (uv lock, or INSTALL_DEPS=1)
- Verify the error banner distinguishes project-based conflicts ("run uv lock") from requirements.txt conflicts ("relax version pin or check constraints.txt")
- Run `shellcheck` — any warning is a blocker
