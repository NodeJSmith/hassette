## Objectives & Success Criteria

Docker integration tests reflect the new startup behavior: `INSTALL_DEPS` gating, exact requirements.txt match, and constraints enforcement. All tests pass against an image built with WP01's Dockerfile.

- Existing tests that exercise requirements discovery pass `HASSETTE__INSTALL_DEPS=1`
- `test_docker_installs_requirements_dev_variants` is inverted to assert dev variants are NOT installed
- A new test verifies requirements are NOT installed when `INSTALL_DEPS` is unset (default-off)
- Pinned versions in test fixtures are relaxed to `>=` specifiers to survive constraint drift
- `test_fd_handles_multiple_requirements_patterns` in `test_docker_requirements_discovery.py` is updated for exact-match fd pattern
- `test_docker_handles_missing_requirements` assertion message matches new script output

## Subtasks

1. Add `-e HASSETTE__INSTALL_DEPS=1` to all Docker run commands in `tests/test_docker_integration.py` that test requirements discovery: `test_docker_installs_user_requirements`, `test_docker_finds_nested_requirements`, `test_docker_installs_from_config_and_apps`, `test_docker_skips_empty_requirements`, `test_docker_handles_missing_requirements`
2. Invert `test_docker_installs_requirements_dev_variants`: assert that `requirements-dev.txt` is NOT in the output and only `requirements.txt` is installed (the fd pattern now matches exact filenames only)
3. Add `test_docker_skips_requirements_by_default`: run the container without `INSTALL_DEPS=1`, with a `requirements.txt` present; assert "Runtime dependency installation disabled" in output and no "Installing requirements from" messages
4. Relax pinned versions in test fixtures: change `pyyaml==6.0.1` to `pyyaml>=6.0`, `httpx==0.25.0` to `httpx>=0.25`, `httpx\n` to `httpx>=0.25\n`
5. Update `test_docker_handles_missing_requirements` assertion from `"Completed installation of 0 found requirements.txt"` to match the new script's output message
6. Update `test_fd_handles_multiple_requirements_patterns` in `tests/test_docker_requirements_discovery.py` to reflect the exact-match fd pattern (only `requirements.txt` should be found, not `requirements-dev.txt` or `requirements_test.txt`)
7. Add `test_docker_constraint_conflict`: create a `requirements.txt` with a version that conflicts with the constraints file (e.g., `pydantic==1.10.0` when hassette constrains `pydantic==2.x`), run the container with `INSTALL_DEPS=1`, assert non-zero exit code and that the output contains the DEPENDENCY CONFLICT error banner

## Test Strategy

These ARE the tests. Verification:

- `uv run pytest tests/test_docker_integration.py -v -m docker` passes (requires a built Docker image)
- `uv run pytest tests/test_docker_requirements_discovery.py -v` passes (requires `fd`/`fdfind` on the host)

Note: Docker integration tests require a built image and Docker daemon. They run in CI via `docker-tests.yml` and locally only when Docker is available.

## Review Guidance

- Verify every test that expects requirements installation passes `HASSETTE__INSTALL_DEPS=1`
- Verify the inverted dev-variant test asserts the ABSENCE of dev file installation
- Verify no test uses exact version pins that could conflict with the constraints file
- Verify the fd discovery test update matches the new pattern in `docker_start.sh`
- Check that the default-off test does NOT pass `INSTALL_DEPS` and asserts the disabled message

## Avoid

- Do not use exact version pins (e.g., `pyyaml==6.0.1`) in test fixtures — use `>=` specifiers to survive constraint drift
- Do not forget `HASSETTE__INSTALL_DEPS=1` on any test that expects requirements to be installed
