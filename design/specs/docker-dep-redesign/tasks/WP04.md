## Objectives & Success Criteria

All Docker documentation accurately reflects the new startup behavior, constraints mechanism, and error UX. Stale references to removed features are eliminated. Users encountering constraint errors find clear, actionable troubleshooting guidance.

- `docs/pages/getting-started/docker/index.md` lists `HASSETTE__INSTALL_DEPS` and removes `HASSETTE__ALLOW_UNLOCKED_PROJECT` from the env var table; adds the graceful shutdown section from PR #479
- `docs/pages/getting-started/docker/dependencies.md` has an updated flow diagram, explains the constraints concept proactively, updates the custom-image example to use export-then-install, documents the local path deps limitation, and uses accurate language for opt-in requirements discovery
- `docs/pages/getting-started/docker/troubleshooting.md` has a dedicated "Dependency Conflicts" section with the error UX framing from the design doc, removes `ALLOW_UNLOCKED_PROJECT` references, and fixes the stale "reinstalls hassette" text
- `examples/docker-compose.yml` has `start_period: 60s` in its healthcheck
- No documentation page references `ALLOW_UNLOCKED_PROJECT` as a supported feature
- No documentation page references `requirements-dev.txt` or `requirements_test.txt` as discoverable files

## Subtasks

1. Update `docs/pages/getting-started/docker/index.md`:
   - Add `HASSETTE__INSTALL_DEPS` to the env var table
   - Remove `HASSETTE__ALLOW_UNLOCKED_PROJECT` from the env var table
   - Verify the graceful shutdown section from PR #479 is present (it should be on main already)
2. Update `docs/pages/getting-started/docker/dependencies.md`:
   - Replace the overview paragraph: "automatically discovers and installs" → "installs project dependencies automatically and optionally discovers requirements.txt files when enabled"
   - Update the Mermaid flow diagram to show the export-then-install steps and `HASSETTE__INSTALL_DEPS=1?` gate (use full env var name in diagram labels)
   - Update "Key Behaviors" to describe export-then-install, opt-in requirements, and constraints protection for ALL installs
   - Add a "How Constraints Work" paragraph in the overview explaining the concept proactively (from design doc section "Dependencies page")
   - Fix the flat-structure description: "automatically discovered and installed" → "discovered and installed when `HASSETTE__INSTALL_DEPS=1` is set"
   - Remove the "Without a Lock File" / `ALLOW_UNLOCKED_PROJECT` section entirely; replace with a note that a lockfile is required
   - Remove the "Multiple Requirements Files" section (only exact `requirements.txt` is discovered now)
   - Add an "Opt-in required" admonition above the requirements.txt section
   - Update the custom-image Dockerfile example to use export-then-install with constraints instead of `uv sync`
   - Add a "Known Limitations" subsection noting local path dependencies and the custom-image workaround
   - Add `HASSETTE__INSTALL_DEPS=1` to Example 1's compose environment section
3. Update `docs/pages/getting-started/docker/troubleshooting.md`:
   - Add a "Dependency Conflicts" section with the full UX framing from the design doc: what the error looks like, why it happens, how to fix it, how to prevent it
   - Remove `ALLOW_UNLOCKED_PROJECT` from the "Unlocked Project Not Installing" section; replace with guidance to generate a lockfile
   - Fix "requirements.txt Not Found" to mention `HASSETTE__INSTALL_DEPS=1` as the first check
   - Fix "Hassette Module Not Found" to remove the stale "script reinstalls hassette" text
   - Update the requirements.txt naming guidance to reflect exact match only
4. Update `examples/docker-compose.yml`: add `start_period: 60s` and change `interval: 10s` to `interval: 30s` in the healthcheck
