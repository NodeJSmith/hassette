## Objectives & Success Criteria

The Dockerfile produces a `/app/constraints.txt` in the final image that pins all hassette dependencies plus hassette itself. A stale lockfile causes a build failure. `git` is removed from the final stage.

- `docker build` succeeds and the resulting image contains `/app/constraints.txt`
- The constraints file contains range-based versions matching `pyproject.toml` (e.g., `pydantic>=2.0,<3`) and an exact `hassette==<version>` pin
- `RUN uv lock --check` fails the build if `uv.lock` is stale relative to `pyproject.toml`
- `git` is not available in the final image (`docker run <image> which git` returns non-zero)

## Subtasks

1. Add `RUN uv lock --check` after copying `pyproject.toml` and `uv.lock` but before `uv sync` in the builder stage of `Dockerfile`
2. Create `tools/generate_constraints.py`: reads `pyproject.toml` `[project].dependencies` list, extracts package names and version specifiers, appends `hassette==<version>` (from `importlib.metadata`), writes pip-constraints-format output to stdout. Must handle extras syntax (e.g., `uvicorn[standard]>=0.30` → `uvicorn>=0.30`).
3. Add `RUN python tools/generate_constraints.py > /app/constraints.txt` after the final `uv sync` in the builder stage
4. Remove `git \` from the `apt-get install` list in the final stage (line ~56)
5. Add `constraints.txt` to `.dockerignore` with a comment explaining it's generated at build time
6. Add unit tests for `generate_constraints.py` in `tests/unit/tools/test_generate_constraints.py`: verify output format, verify hassette pin is exact, verify extras are stripped, verify ranges are preserved

## Test Strategy

- `tests/unit/tools/test_generate_constraints.py`: unit tests for the constraints generator — verify output format, exact hassette pin, extras stripping, range preservation
- Dockerfile changes verified by the existing Docker CI workflow (`docker-tests.yml`) which builds the image
- The `uv lock --check` guard is a build-time check that prevents stale lockfiles from being shipped

## Review Guidance

- Verify `uv lock --check` is positioned after `COPY` of `pyproject.toml`/`uv.lock` but before any `uv sync`
- Verify `generate_constraints.py` runs after the final `uv sync` (so `importlib.metadata` can read the installed hassette version)
- Verify the constraints file uses ranges from `pyproject.toml`, not exact pins from the lockfile
- Verify `git` is not in the final stage's `apt-get install` list
- Confirm the constraints file is included in the `COPY --from=builder` step (it's in `/app` which is already copied)

## Avoid

- Do not remove the `uv sync` steps from the builder stage — those are build-time and correct; only runtime `uv sync` is being eliminated
- Do not use exact pins from the lockfile for constraints — use the declared ranges from `pyproject.toml` (except for hassette itself, which is pinned exactly)
- Do not hardcode dependency names — read them from `pyproject.toml` programmatically
