## Warnings

### WARNING 1 — AC#3 visual verification relies on code inspection, not an executed screenshot comparison

**Design doc Test Strategy:**
> Visual verification: screenshot comparison before/after to confirm no visual regression (existing Playwright infrastructure from #736)

**T05 Verify item:**
> `AC#3: npm run build succeeds and visual appearance is preserved (no CSS value changes)`

The design doc's Test Strategy calls for a Playwright screenshot comparison. T05's Verify item reduces this to "npm run build succeeds" plus a code-inspection claim ("no CSS value changes"). A build-clean result would not catch a misapplied class, a missing CSS selector, or a dark-mode regression. The Playwright comparison step is absent from T05's Prompt and all other tasks.

This is a warning, not a blocking issue — the plan already documents the one known visual delta (4px padding change) in AC#3, and the migration is mechanical. But the specified verification method doesn't match what the design doc requires.

**Suggested resolution:** Add to T05 Verify: `[ ] AC#3: Playwright screenshot comparison before/after confirms no unintended visual regression`. Or explicitly acknowledge in T05 that the Playwright comparison is waived.

### WARNING 2 — T03 Verify item for FR#13 undercounts scope

**T03 Verify item:** `FR#13: All five shared components and error-boundary use Button/Card components instead of raw class strings`

FR#13 is "Every TSX file that currently uses raw `ht-btn`, `ht-badge`, `ht-chip`, or `ht-card` class strings is migrated." T03 handles only the shared-component layer (5 shared components + error-boundary). The remaining 16 consumer files are covered by T04, which has its own FR#13 Verify item. T05 runs the authoritative AC#1 grep that definitively confirms FR#13 across all files.

The T03 Verify phrasing is technically accurate for T03's scope, but a reviewer reading it in isolation could interpret it as fully satisfying FR#13. This is a readability issue, not a coverage gap.

**Suggested resolution:** Amend T03's Verify item to: `[ ] FR#13 (partial — shared layer): All five shared components and error-boundary use Button/Card; remaining consumer files covered by T04`.

### WARNING 3 — Dependencies section of T01 and T02 omits T01→T02 test-infrastructure order dependency

T01 and T02 both declare `depends_on: []` and are intended to run in parallel. This is correct for production code. However, T03's test update instructions cross-reference T04 (T03 note: `job-executions.test.tsx` badge selector update "requires adding `data-testid` to the Badge usage in `job-executions.tsx` (done in T04)"). The cross-task test dependency is documented in prose but not reflected in `depends_on` metadata. T03 depends on T04 for the `job-executions.test.tsx` update to be executable, but `depends_on` only lists T01 and T02. An implementer executing T03 fully before T04 would not be able to complete the `job-executions.test.tsx` update.

This is a warning because it could cause a partial-completion block mid-task, not because any requirement goes uncovered.

**Suggested resolution:** Either move the `job-executions.test.tsx` update from T03 to T04 (where the `data-testid` addition happens), or add T04 to T03's `depends_on`. The current T05 Verify for AC#8 still catches the overall test-pass requirement.
