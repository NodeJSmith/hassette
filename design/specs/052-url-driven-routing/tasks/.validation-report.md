## Contradictions

None found. Spot checks:

- **FR#14 exception for `?instance=0`**: Design doc states `?instance=0` is retained (not omitted) for multi-instance apps because the no-param URL is the parent overview. T01 Verify only checks that default values produce empty query string — this could be read as a gap, but the T01 Prompt explicitly states "Keep `?instance=0` explicit for multi-instance apps (do not omit as default)" and the same constraint appears in T02 Focus and Prompt. The AC#14/FR#14 instance=0 exception is correctly handled by T02, not T01. No contradiction.

- **FR#12 coverage**: T04 implements FR#12 (replace history for filter/sort/search). T05 and T06 also write params with `push: false` (default) per their Prompt text, consistent with FR#12. T05 does not list FR#12 in its `implements` but its Prompt is correct. This is a minor annotation omission (T05 omits FR#12 from its `implements` list), but the implementation guidance is correct. No functional gap.

- **`correctUrl` timing guard**: Design doc FR#16 states "must not fire during loading or while stale data is in use." T03 Focus addresses this explicitly: "The `correctUrl` timing guard is critical — only fire when data is loaded." No contradiction.

- **`cameFromHandlers` signal removal**: Design doc "Architecture > View in Code" says the signal and "back to handlers" button are both removed. T02 Prompt removes both. T03 makes CodeTab read `?line=` from URL. Consistent.

- **`useSearch` import removal**: T02 Prompt explicitly removes the `useSearch` import from `app-detail.tsx` after `focusMethod` removal. Consistent with design doc.

## Warnings (Non-Blocking)

1. **T05 missing FR#12 in `implements` list**: T05's Prompt instructs `qp.set()` with default `push: false` (replace history), which is exactly what FR#12 requires. However FR#12 is not in T05's `implements` list. This is an annotation gap, not a functional gap — the implementation guidance is correct.

2. **FR#12 not listed in T06 `implements`**: Same pattern as T05. T06 writes all log filter changes via `qp.set()` (default replace), satisfying FR#12. But FR#12 does not appear in T06's `implements` list.

3. **T07 time window button — `qp.set` logic looks slightly off in Prompt**: The T07 Prompt says `qp.set({ window: newPreset === timePreset.value ? null : newPreset })`. This would omit `?window=` when the new preset equals the existing persisted value, which seems incorrect — the intent from the design doc is that clicking the button always writes the window to the URL (FR#9, AC#8). An implementer following the Prompt literally could produce a behavior where clicking the time preset button while the URL already matches the persisted preference removes `?window=` from the URL. The Verify checklist (AC#8: "Clicking the time preset button on any page updates both the URL and the persisted global preference") should catch this in testing, but the Prompt wording is a potential implementation trap.

4. **T05 `implements` missing `FR#12`**: As noted above. The Verify section for T05 checks AC#6 (which requires FR#12 behavior) — this provides functional coverage even without the explicit `implements` annotation.

5. **context.md Visual Artifacts section is minimal**: The section reads "None." This is accurate for this feature (no mockups or wireframes were produced), so it is complete per the design doc ("No new UI elements"). Not a defect.
