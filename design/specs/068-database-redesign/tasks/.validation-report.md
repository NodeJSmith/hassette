## Warnings

### W1 — FR#19 verification split: schema vs. model

FR#19 ("Known future columns in initial schema") is claimed by two tasks with different scopes. T05 verifies model-level presence (`AC#12` Verify item: "New columns exist on the model with correct defaults"). T14 verifies the actual SQL schema on disk ("New columns present in `executions` table"). Both checks are needed and both are present. However, T02 also writes 001.sql with the future columns — yet T02 does not list FR#19 or AC#12 in its `implements` field. The schema DDL is where these columns originate; the current attribution (T05, T14) is technically correct but the omission from T02 is a documentation inconsistency. Not a gap, but worth noting.

### W2 — T02 unit tests cover AC#12 without claiming it

T02's Prompt Step 6 explicitly says: "Test: new columns exist in schema after 001 applies." This is the AC#12 check. T02 does not list AC#12 in `implements`, but T05 does. This is consistent but the test coverage for AC#12 will be written in T02, not T05. An implementer working from T05 alone might not realize T02 already writes this test — creating a duplicate test or missing it entirely. Low risk but worth a note in the handoff.

### W3 — FR#16 frontend removal split across T04, T10, T11

FR#16 covers three distinct layers: the API field (T04), the `dropped_no_session` column in the 001.sql sessions table (T02), and the frontend badge/signal (T10, T11). T02 does not list FR#16 in `implements` even though its 001.sql DDL explicitly drops the column from the `sessions` table. T04 handles the Python code removal and lists FR#16. The storage layer component of FR#16 is implemented in T02 but not tracked there. This is an attribution gap in the task plan — the column removal in 001.sql is untracked. Not a functional gap (T02's Prompt explicitly mentions it), but the `implements` field would fail a grep-based audit.

### W4 — AC#1 claimed by T06, T07, T14 but not T03, T04, T05

AC#1 says "All existing tests pass after adaptation." Tasks T03, T04, and T05 each write new tests and update existing ones, but none list AC#1 in `implements`. They could break existing tests if their changes are incomplete. The claim is made by T14 (full suite run), which is the correct final gate — this is fine architecturally, but an auditor looking for "which task makes the full suite green" finds only T06/T07/T14, which may be misleading. Not a gap.

### W5 — T04 Verify item for FR#16 grepping both src/ and frontend/src/ combined

T04's `AC#10` Verify item uses `grep -r "dropped_no_session\|droppedNoSession" src/hassette/` — this grep targets only the backend. The frontend removal of `droppedNoSession` is handled in T11, but T04 also lists FR#16. T04's Verify for FR#16 only checks backend; the frontend check is deferred to T11's Verify. This is correctly split across tasks, but the T04 Verify could make an implementer think FR#16 is fully done after T04. Minor clarity issue.

### W6 — T12 depends_on missing T02

T12 ("Update documentation") depends on T03 and T04 but not T02. The documentation updates include describing the new migration runner (PRAGMA user_version, unified executions table). T12 Step 3 explicitly covers `internals.md` — the database section. If T02 is not complete, the documentation cannot accurately describe the new schema. The missing dependency does not create a functional gap (a human can reorder), but the dependency graph is incomplete. If tasks run in topological order, T12 could begin before T02 finishes.

### W7 — T13 does not depend on T06 or T07

T13 ("Update e2e and system tests") depends on T09, T10, T11 but not T06 or T07. T13 Step 4 includes updating `test_telemetry_execution_id.py`, `test_framework_telemetry.py`, and `test_database_service.py` — tests that exercise repository and command executor behavior. If T06 and T07 are not complete, T13 implementer cannot know the final shape of `ExecutionRecord`, persist signatures, or the unified command executor. In practice a human would reorder, but the dependency graph would trigger parallel execution in an orchestrator where that's incorrect.
