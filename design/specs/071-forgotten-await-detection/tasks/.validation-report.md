## Warnings

**W1 — FR#5 split across three tasks with no Verify at T03 (pre-existing):**
FR#5 covers Pyright's `reportUnusedCoroutine` for all protected methods. T03 does not include FR#5 in its `implements` list and has no Pyright-probe Verify item for scheduler methods. Coverage is compensated by T06's AC#3/FR#5 probe fixture, which covers the full canonical list including scheduler methods. Not a gap, but implementers should not consider FR#5 satisfied for scheduler methods until T06 lands.

**W2 — T01 `implements` AC#1/AC#5 but not AC#2 (pre-existing):**
AC#2 requires verifying awaiting a converted public method returns the right type with no warning. T01 builds the handle machinery in isolation; no converted public methods exist yet, so a full AC#2 test cannot be written in T01. AC#2 is correctly deferred to T02/T03/T04. This is consistent with the task split.

**W3 — FR#12 documentation split across T01 and T06 (pre-existing):**
T01's FR#12 Verify covers the runtime observable guarantee (stored handle defers warning to collection). T06's FR#12 Verify covers the troubleshooting documentation. Both are required; neither is redundant. Implementers should note that T01 does not produce the docs, only the testable runtime behavior.

**W4 — T07 entity methods not in AC#6 completeness guard scope:**
The AC#6 completeness/drift guard in T06 enumerates public methods on `Bus`, `Scheduler`, and `Api` specifically. Entity methods (`LightEntity.turn_on`, etc.) are on separate classes and are not enumerated by this guard. FR#13/AC#11 in T07 provides the separate regen-check and representative tests for entity coverage. This is a deliberate split in the design (entity protection via codegen template, not the Bus/Scheduler/Api canonical list), not a coverage gap — but implementers should be aware that the completeness guard does not catch a future unprotected entity method; the codegen freshness gate and template review are the structural protection there.
