# Context: Namespace Helper CRUD Behind Api.helpers

## Problem & Motivation

The `Api` class exposes 35 flat helper CRUD methods (8 domains × 4 ops + 3 counter shortcuts) that bloat its public surface to ~72 methods. This makes autocomplete noisy, the class hard to navigate, and violates the minimize-public-API-surface principle. The file is also 1477 lines, well past the 800-line cap. With the v1.0 API freeze approaching, the shape must be right before it becomes a compatibility commitment. The refactor replaces these 35 methods with a single `api.helpers` attribute exposing a `HelperClient` class with 4 generic CRUD methods, 3 counter shortcuts, and `@overload` declarations for full type safety.

## Key Decisions

1. **Params-model dispatch** — `create()` and `update()` dispatch on `type(params)` to determine the domain and return type. This preserves the existing params model validation (e.g., `input_datetime`'s `has_date or has_time` validator) and gives fully typed inputs without kwargs.
2. **Literal-string dispatch for list/delete** — `list()` and `delete()` take a `HelperDomain` literal union because they have no params object to dispatch on. Minor asymmetry, ergonomically natural.
3. **Hand-maintained overloads** — 8 overloads per method, ~32 two-line signatures total. HA helper schemas change roughly once a year (3 commits in 2 years), so codegen for overloads would add complexity for negligible savings.
4. **HelperClient as Resource** — must be a `Resource` subclass for the sync facade's `task_bucket.run_sync()`. Added via `self.helpers = self.add_child(HelperClient, api=self)`, matching the `self.sync` convention.
5. **Separate sync facade** — `HelperClientSyncFacade` generated by a new `generate_sync_helpers()` in the existing codegen pipeline, following the `generate_sync()` pattern exactly.
6. **Counter shortcuts on HelperClient** — `increment()`, `decrement()`, `reset()` move to HelperClient despite using `call_service()` internally, because they're conceptually helper operations.
7. **Breaking change, no compatibility layer** — per repo convention.

## Constraints & Anti-Patterns

- Do NOT modify the helper Pydantic models in `src/hassette/models/helpers/` — reuse as-is
- Do NOT modify `_ws_helper_call()`, `_expect_list()`, or `_expect_dict()` in `api.py` — the shared infrastructure stays
- Do NOT add compatibility shims for the old flat methods
- Do NOT generate overloads via codegen — hand-maintain them
- The sync facade generator walks a single `ast.ClassDef` by exact name in `module.body` — `HelperClient` must be in its own file, not an inner class
- The `ID_KEYS` pattern is uniform: `{domain}_id` (e.g. `input_boolean_id`, `counter_id`, `timer_id`)
