## Step 1: Requirements Extracted from design.md

### Functional Requirements (10 total)

- FR#1 — The tool queries a running hassette instance's REST API and displays results in the terminal
- FR#2 — Each readable API resource is accessible as a noun-based subcommand
- FR#3 — The default output format is a human-readable table for collections and a key-value panel for single objects
- FR#4 — A flag switches output to structured data containing the complete response model
- FR#5 — When the structured output flag is active, stdout contains exactly one valid document — no other output may appear on stdout
- FR#6 — Commands that return collections support filtering by app key, instance, time window, and result count limit via flags
- FR#7 — The tool discovers the server address from the same configuration sources used by the server itself
- FR#8 — Running the binary with no subcommand starts the server, preserving backward compatibility
- FR#9 — Commands return structured data objects; a single rendering layer handles all output formatting
- FR#10 — The tool supports shell tab completion for commands and subcommands

### Acceptance Criteria (12 total)

- AC#1 — All GET endpoints in the REST API are queryable via CLI subcommands
- AC#2 — Human-readable output fits within an 80-column terminal without horizontal scrolling for the default column set
- AC#3 — Structured output mode produces valid parseable output on stdout with no other content on stdout
- AC#4 — `hassette listener --app my-app` returns only listeners belonging to `my-app`
- AC#4a — `hassette listener --app my-app --instance 1` returns only listeners for instance index 1; `--instance office` resolves name to index
- AC#5 — `hassette log --since 1h --limit 20` returns at most 20 log entries from the last hour
- AC#6 — Running `hassette` with no arguments starts the framework server as before
- AC#7 — Adding a new output format requires changes only to the rendering layer, not to individual commands
- AC#8 — When the server is unreachable, the tool exits with a non-zero code and a human-readable error message on stderr
- AC#9 — When a configuration file or environment variable specifies a non-default server address, the tool queries that address instead of the default
- AC#10 — Tab completion for command and subcommand names works after running the shell-specific installation command

**Identifier format check:** All identifiers are correctly formatted as `FR#N` and `AC#N`. No violations. Note: `AC#4a` is non-standard (uses a letter suffix rather than an integer). It is treated as a valid identifier here since it appears consistently in both design doc and task files.

### GAP-1: `context.md` missing `## Visual Artifacts` section

The required `## Visual Artifacts` section is entirely absent from `context.md`. The file jumps from `## Problem & Motivation` directly to `## Key Decisions`. Per the validation rules, sections 1–5 must all be present with non-empty content — this is a required section regardless of whether the design doc contains visual artifacts. The correct response when no visual artifacts exist is to include the section with a note (e.g., "None — this feature adds no new UI surfaces") rather than omitting it.

**Impact:** context.md fails the five-required-sections check, which triggers ISSUES_FOUND status.

### GAP-2: `--source-tier` flag coverage gap (FR#6 partial)

FR#6 states: "Commands that return collections support filtering by app key, instance, time window, **and result count limit** via flags." The design doc also defines `--source-tier` as a shared filter used by listener, job, and log commands. While FR#6 does not name `--source-tier` explicitly, it is one of the shared filtering flags in the Shared Flags section and is referenced in T07 and T08's Verify items.

More specifically: no task's `implements` list or Verify section addresses the `--source-tier` flag for the `log` command's relationship to FR#6. T08 covers `--app`, `--since`, `--limit`, `--source-tier` in its Verify criterion, so this is addressed — **GAP-2 is informational only**.

Reconsidering: all filtering flags (including `--source-tier`) are covered in T07 and T08 Verify items that cite FR#6. No actual gap.

*Retracting GAP-2 — not a coverage gap.*

### GAP-2 (revised): `hassette dashboard` command not covered by AC#1 smoke test

AC#1 states: "All GET endpoints in the REST API are queryable via CLI subcommands." T09's smoke test cases cover: status, config, service, telemetry, event (system), app/app health/app activity (app commands), listener/listener --app/listener --instance/job (listener/job), log/log --app (log commands). The `hassette dashboard` command (maps to `GET /api/telemetry/dashboard/app-grid`) is not listed among T09's test cases.

T09's Verify criterion reads: "Every GET endpoint is exercised by at least one smoke test." This criterion claims full AC#1 coverage, but the Prompt section's test cases do not include `dashboard`. This is a contradiction between the Verify claim and the Prompt content within T09.

**Impact:** Medium — the omission may be intentional (dashboard exercised by unit tests in T05, and T09 is smoke tests) but the Verify claim says "every GET endpoint." Flag for human review.

### GAP-3: `hassette execution` command not covered by T09 smoke test

Similarly, `hassette execution <uuid>` (maps to `GET /api/executions/{execution_id}`) is not in T09's Prompt test cases. The command requires a valid execution UUID that would exist in the demo setup, which may be why it was omitted. However, T09's Verify item still claims "Every GET endpoint is exercised by at least one smoke test" without excluding it.

**Impact:** Medium — same as GAP-2. The `execution` command would need a live execution UUID to test end-to-end. This may be intentionally deferred, but the Verify item does not acknowledge the exclusion.
