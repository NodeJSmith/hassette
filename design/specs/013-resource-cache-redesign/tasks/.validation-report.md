## Coverage Gaps

- **AC#3** (FR#3) — "All async interface methods (`get`, `set`, `delete`, `get_or_set`, `clear`, `invalidate`) exist and are callable" does not appear in any task's `implements` field or Verify section. Substantively this is exercised by T01's `FR#3` Verify item, but the identifier itself is never traced, so a downstream auditor grepping for `AC#3` finds nothing.
- **AC#4** (FR#4) — "`cache.sync` accessor returns a `SyncCache` with matching method signatures. Calling `.sync.get()` from inside a running event loop raises `RuntimeError`" is not claimed anywhere. Substantively covered by T01's `FR#4` Verify item, but untagged.
- **AC#5** (FR#5, FR#6) — "`set(key, value, ttl=None)` falls back to the instance's pre-resolved `default_ttl`; `set(key, value, ttl=0)` deletes any existing entry and does not store" is not claimed anywhere. Substantively covered by T01's `FR#5`/`FR#6` Verify items, but untagged.
- **AC#6** (FR#9, FR#10) — "`DummyCache` injected via test infrastructure makes `app.cache` return it — `set` stores in-memory with TTL, `get` returns `None` for expired entries, `.sync` accessor works" is not claimed anywhere. The `FR#9` half is covered by T01, the `FR#10` half by T03's `AC#10` item, but the combined end-to-end criterion as stated (`app.cache` returning the injected DummyCache *and* exhibiting correct in-memory TTL behavior together) is never tested as a single criterion in either task.
- **AC#7** (FR#12) — "An app with a corrupt cache directory initializes successfully — integrity check detects corruption, file is recreated, warning logged" is not claimed anywhere. Substantively covered by T01's `FR#12` Verify item (which tests corruption recovery at the `AsyncCache` level, not through a full `App`), but untagged and not tested at the App-integration level in T03.
- **AC#17** (FR#7) — Design doc criterion: *"An App subclass with `default_cache_ttl = 60` uses that value as the default TTL when no per-call `ttl` is specified."* This requires end-to-end behavior: an App subclass sets the class attribute, and a subsequent `cache.set()` without `ttl=` actually uses 60 seconds. T02 claims AC#17 in `implements`, but its Verify text reads "`HassetteConfig` accepts `default_cache_ttl` as an integer" — that tests schema acceptance on `HassetteConfig`, not the App-class-attribute behavior AC#17 actually describes (which doesn't even involve `HassetteConfig`). The real wiring (`cls.default_cache_ttl` → resolved `default_ttl` → `AsyncCache` constructor) is implemented in T03 per its Prompt ("default_ttl resolved from: cls.default_cache_ttl if set on the class → hassette.config.default_cache_ttl → None"), but T03 does not list FR#7 or AC#17 in `implements`, has no Verify item for either, and its "New tests for this task" list does not include a test for the TTL resolution chain. **No task in this plan verifies the actual AC#17 behavior.**
- **AC#18** (FR#8) — Design doc criterion: *"`HassetteConfig.default_cache_ttl = 120` is used as the fallback when the App subclass does not declare `default_cache_ttl`."* This is also an end-to-end behavior requiring the App/cache wiring done in T03. T02's Verify text — "`HassetteConfig.default_cache_ttl = 120` is accepted as valid" — again only tests schema-level acceptance, not the fallback-resolution behavior. Same gap as AC#17: **no task verifies the actual fallback behavior end-to-end.** (Note: the design doc's own `## Test Strategy` section also omits an explicit test for this resolution chain — the gap traces back to the design doc's test strategy, not just the task authoring.)

## Warnings

- **T01 `__init__.py` scope addition beyond design doc:** T01's Prompt says `__init__.py — Export AsyncCache, SyncCache, DummyCache, DummySyncCache, CacheProtocol`, but the design doc's Architecture section for `__init__.py` only lists `Exports AsyncCache, SyncCache, DummyCache, CacheProtocol` (no `DummySyncCache`). This is a minor, harmless addition (not a new FR/AC, just an extra export of an already-specified class) but is technically scope beyond the literal design doc text.
- **T02's FR#7 Verify text is internally inconsistent with FR#7 itself:** the criterion states "An App subclass can declare `default_cache_ttl` as a class attribute" — a statement entirely about `App`, not `HassetteConfig` — then parenthetically claims it is "verified by HassetteConfig accepting the field," which verifies a different requirement (FR#8) instead. This mislabeling is the root cause of the AC#17/AC#18 substantive gap noted above and should be corrected in task authoring: FR#7's actual verification belongs in T03 (where the class attribute and its resolution into `AsyncCache`'s `default_ttl` are actually wired).
- **AC#3, AC#4, AC#5, AC#6, AC#7 labeling gap:** these are not scope violations (the underlying behavior is designed and largely exercised by T01/T03's FR-tagged Verify items) — they are a pure traceability/labeling omission. Recommend adding the AC# tags to the relevant tasks' `implements` fields and Verify sections so the plan's own traceability is self-auditing, and recommend T03 add an App-level Verify item for AC#7 (corruption recovery through the full App lifecycle, not just at the `AsyncCache` unit level) since T01's corruption test only exercises `AsyncCache` directly.
