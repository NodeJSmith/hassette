## Objectives & Success Criteria

`HassetteHarness` defaults to `strict_lifecycle = True`. All ~688 existing tests pass with strict lifecycle enforcement. Every transition violation discovered during the audit is resolved — either by updating the transition table (if the transition is legitimate) or by fixing the code/test (if the transition is actually invalid). The `create_hassette_stub()` mock is verified to not break under strict mode.

## Subtasks

1. Update `HassetteHarness.__init__()` in `src/hassette/test_utils/harness.py` to set `strict_lifecycle = True` on the config by default. If the caller passes an explicit `HassetteConfig` with `strict_lifecycle` already set, respect that override.
2. Verify `create_hassette_stub()` in `src/hassette/test_utils/web_mocks.py` — the MagicMock's `config.strict_lifecycle` will return a truthy Mock by default. Either set it explicitly to `False` (stubs don't go through real lifecycle) or verify the `hasattr` guard from WP01 handles it correctly.
3. Run the full test suite with `strict_lifecycle = True`: `timeout 600 uv run nox -s dev -- -n 2`
4. For each failure, classify and fix:
   - **Transition table gap** — a legitimate code path triggers a transition not in `VALID_TRANSITIONS`. Update the table in `mixins.py` and add a test for the new transition in `test_lifecycle_transitions.py`.
   - **Actually-invalid transition in production code** — fix the production code to use the correct lifecycle methods (e.g., go through STOPPING before STOPPED).
   - **Test-only state manipulation** — test sets `.status` directly to set up a fixture state. Change to `._status = ...` to bypass validation, since it's deliberate test setup, not a lifecycle operation.
   - **WebSocket connection state violation** — same classification: fix table, fix code, or fix test.
5. After all failures are resolved, run the full suite again and verify all tests pass.
6. Run `uv run pyright` to verify no type errors were introduced across all WPs.

## Test Strategy

No new test files. This WP's deliverable is the passing test suite itself.

- The full test suite (`uv run nox -s dev -- -n 2`) is the acceptance test
- If any transition table updates are made in step 4, add a corresponding test case in `test_lifecycle_transitions.py` documenting the transition
- If any production code fixes are made in step 4, add a regression test verifying the corrected behavior

## Review Guidance

- Every transition table update must be justified — the reviewer should verify that the added transition is genuinely valid, not just "this makes the test pass." Each new entry should correspond to a real code path with a clear reason.
- Test fixtures that use `._status = ...` bypass must be limited to test setup — no production code should use the bypass except `_force_terminal()`.
- The final test run must be clean — zero failures, zero warnings from the transition guard. A WARNING in the test output means a non-strict code path is being exercised, which shouldn't happen since the harness defaults to strict.
- Check that `create_hassette_stub()` tests don't accidentally trigger real lifecycle validation — the mock should isolate cleanly.
