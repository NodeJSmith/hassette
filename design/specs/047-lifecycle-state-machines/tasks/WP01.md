## Objectives & Success Criteria

The `status` setter on `LifecycleMixin` validates every transition against a static table. Invalid transitions raise `InvalidLifecycleTransitionError` in strict mode and log a WARNING in non-strict mode. `_force_terminal()` bypasses validation via direct attribute assignment. The `strict_lifecycle` config field and both new exception classes exist and are usable by subsequent WPs.

## Subtasks

1. Add `strict_lifecycle: bool = False` field to `HassetteConfig` in `src/hassette/config/config.py`
2. Add `InvalidLifecycleTransitionError` and `RegistryValidationError` to `src/hassette/exceptions.py` — both inherit from `HassetteError`. `InvalidLifecycleTransitionError` takes `from_status`, `to_status`, and `resource_name` in its constructor.
3. Define `VALID_TRANSITIONS: dict[ResourceStatus, frozenset[ResourceStatus]]` as a module-level constant in `src/hassette/resources/mixins.py`. Include all transitions from the design doc (including STOPPING and EXHAUSTED states that WP02 will use). The table is correct from day one — WP02 adds the code paths that exercise the new transitions.
4. Update the `status` setter in `LifecycleMixin` to validate against `VALID_TRANSITIONS`. Guard with `hasattr(self, 'hassette')` — when the guard fails (object not fully constructed), fall through without validation. In strict mode (`self.hassette.config.strict_lifecycle`), raise `InvalidLifecycleTransitionError`. In non-strict mode, log WARNING with transition pair, `unique_name`, and a 1-2 frame stack summary. Log every valid transition at DEBUG as `"{unique_name}: {old} → {new}"`.
5. Change `_force_terminal()` in `src/hassette/resources/base.py` (Resource) to use `self._status = ResourceStatus.STOPPED` instead of `self.status = ResourceStatus.STOPPED`. Same for the `Service._force_terminal()` override — call `super()._force_terminal()` which handles it.
6. Add `ConnectionState` enum to `src/hassette/types/enums.py` with three values: `DISCONNECTED`, `CONNECTING`, `CONNECTED`. Export from `src/hassette/types/__init__.py`.

## Test Strategy

Tests in `tests/unit/resources/test_lifecycle_transitions.py` (new file):

- `test_valid_transition_sequence` — walk `NOT_STARTED → STARTING → RUNNING → STOPPING → STOPPED` via the setter, assert no error and DEBUG logs emitted
- `test_invalid_transition_raises_strict` — set up strict mode, attempt `NOT_STARTED → RUNNING`, assert `InvalidLifecycleTransitionError` raised with correct fields
- `test_invalid_transition_warns_nonstrict` — non-strict mode, same invalid transition, assert WARNING logged and transition proceeds
- `test_force_terminal_bypasses_validation` — strict mode, resource in RUNNING, call `_force_terminal()`, assert status is STOPPED with no error
- `test_restart_transitions_valid` — `FAILED → STARTING` and `CRASHED → STARTING` are accepted
- `test_exhausted_transitions_valid` — `FAILED → EXHAUSTED_COOLING`, `EXHAUSTED_COOLING → STARTING`, `EXHAUSTED_COOLING → EXHAUSTED_DEAD` are accepted
- `test_terminal_state_rejects_transitions` — `EXHAUSTED_DEAD → STARTING` raises in strict mode
- `test_hasattr_guard_no_hassette` — create a LifecycleMixin instance without hassette attribute, set status, assert no error (guard falls through)
- `test_same_state_no_transition` — verify that `handle_running()` when already RUNNING returns early without hitting the setter (idempotency preserved)

## Review Guidance

- The `VALID_TRANSITIONS` table must match the design doc exactly — verify every entry. Missing transitions will break WP05 (test audit).
- The `hasattr` guard must use `hasattr(self, 'hassette')` not a try/except — the guard is for construction-time, not runtime errors.
- `_force_terminal` must use `self._status = ...` not `self.status = ...` — this is the entire point of the bypass.
- `ConnectionState` enum is defined here but not wired into `WebsocketService` — that's WP03.
