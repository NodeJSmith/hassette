## Objectives & Success Criteria

`WebsocketService` tracks connection state via a three-value `ConnectionState` enum with validated transitions. The `connection_state` read-only property returns the current state. The existing `connected` property returns `True` only when `connection_state == CONNECTED`. Connection state transitions are logged at DEBUG and validated with the same strict/non-strict behavior as lifecycle guards.

## Subtasks

1. Define `WS_VALID_TRANSITIONS: dict[ConnectionState, frozenset[ConnectionState]]` as a module-level constant in `src/hassette/core/websocket_service.py`. Transitions per design: DISCONNECTED → {CONNECTING}, CONNECTING → {CONNECTED, DISCONNECTED}, CONNECTED → {CONNECTING, DISCONNECTED}.
2. Add `_connection_state: ConnectionState` attribute to `WebsocketService.__init__()`, initialized to `ConnectionState.DISCONNECTED`.
3. Add `connection_state` read-only property returning `self._connection_state`.
4. Add `_set_connection_state(self, new: ConnectionState)` private method. Validates against `WS_VALID_TRANSITIONS`. In strict mode, raises `InvalidLifecycleTransitionError`. In non-strict mode, logs WARNING. Logs every valid transition at DEBUG including previous state for reconnect distinguishability: `"WebSocket: {old} → {new}"`.
5. Update `serve()`: set CONNECTING at entry (before the while loop, since this is the first connection attempt). Inside the while loop, on early-drop retry path (after `_send_connection_lost_event` and `mark_not_ready`), set CONNECTING before continuing to `_make_connection`.
6. Update `_connect_ws()`: no state change needed — CONNECTING was set by the caller (serve or retry loop).
7. Update `_start_recv_and_subscribe()`: set CONNECTED after `mark_ready()` (connection is authenticated and subscribed — this is the point where we're truly connected).
8. Update `cleanup()`: set DISCONNECTED at the start (clean shutdown path).
9. Update `before_shutdown()`: no change — it calls `_send_connection_lost_event` which is idempotent and doesn't change connection state.
10. `_partial_cleanup()`: do NOT change connection state here. It's resource cleanup, not a state transition. The caller is responsible for setting state.
11. Update `connected` property to return `self._connection_state == ConnectionState.CONNECTED`.
12. Handle early-drop serve loop: after exception is caught and `is_early` is true, set CONNECTING before `continue` (the loop will call `_make_connection`). When `is_early` is false (genuine failure), set DISCONNECTED before raising.

## Test Strategy

Tests in `tests/unit/core/test_ws_connection_state.py` (new file):

- `test_initial_state_disconnected` — new WebsocketService has `connection_state == DISCONNECTED`
- `test_connected_property_mirrors_state` — set `_connection_state` directly (test setup), verify `connected` returns `True` only for CONNECTED
- `test_valid_connect_sequence` — mock the connection flow, verify DISCONNECTED → CONNECTING → CONNECTED transitions
- `test_reconnect_sequence` — simulate CONNECTED → CONNECTING → CONNECTED (connection lost, retried)
- `test_auth_failure_disconnects` — simulate CONNECTING → DISCONNECTED on InvalidAuthError
- `test_max_retries_disconnects` — simulate CONNECTING → DISCONNECTED on max retries
- `test_clean_shutdown_disconnects` — simulate CONNECTED → DISCONNECTED via cleanup()
- `test_partial_cleanup_no_state_change` — call `_partial_cleanup()`, verify `connection_state` unchanged
- `test_invalid_transition_strict` — attempt DISCONNECTED → CONNECTED (skipping CONNECTING), assert `InvalidLifecycleTransitionError`
- `test_invalid_transition_nonstrict` — same transition, non-strict mode, assert WARNING logged

Integration with existing tests in `tests/integration/test_websocket_service.py`:

- Verify existing reconnection, early-drop, and auth-failure tests still pass
- Add assertions on `connection_state` at key points in existing tests where `connected` is already asserted

## Review Guidance

- `_partial_cleanup()` must NOT call `_set_connection_state`. This is explicitly called out in the design — verify it.
- The CONNECTING state must be set BEFORE `_make_connection()` is called, not inside it. The caller owns the state transition.
- The CONNECTED state must be set AFTER `mark_ready()` in `_start_recv_and_subscribe()`, not before — connection isn't truly established until subscribed.
- Verify the early-drop retry path sets CONNECTING before `continue`, not DISCONNECTED then CONNECTING — the serve loop goes directly back to connecting.
- The `connected` property change must be backwards-compatible — test that all existing callers of `self.connected` still work.
