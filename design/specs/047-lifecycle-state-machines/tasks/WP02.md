## Objectives & Success Criteria

`shutdown()` sets `ResourceStatus.STOPPING` before hooks run — the shutdown phase is observable via status. `ServiceWatcher` sets the service instance's `.status` to `EXHAUSTED_COOLING` or `EXHAUSTED_DEAD` when it transitions a service to those states — the resource's status and broadcast events agree. Direct status assignments in `app_lifecycle_service.py`, `core.py`, `harness.py`, and `web_mocks.py` are valid transitions under the new table.

## Subtasks

1. In `Resource.shutdown()` (`src/hassette/resources/base.py`), add `self.status = ResourceStatus.STOPPING` after the re-entry guards (`if self._shutdown_completed: return`, `if self._shutting_down: return`, `self._shutting_down = True`) and before `request_shutdown()`. The `_shutting_down` flag is retained for re-entry guarding.
2. In `Service.shutdown()` (same file), add the same `self.status = ResourceStatus.STOPPING` at the equivalent position — after re-entry guards, before `request_shutdown()`.
3. In `ServiceWatcher._handle_exhaustion()` (`src/hassette/core/service_watcher.py`), after emitting the `EXHAUSTED_COOLING` or `EXHAUSTED_DEAD` event, look up the service instance via `self._get_service(name)` and set `service.status = ResourceStatus.EXHAUSTED_COOLING` (or `EXHAUSTED_DEAD`). If `_get_service` returns `None` (service not found), log a warning and skip the status set.
4. In `ServiceWatcher._cooldown_and_retry()`, when transitioning from EXHAUSTED_COOLING to EXHAUSTED_DEAD (cooldown cycles exceeded), look up the service and set `service.status = ResourceStatus.EXHAUSTED_DEAD`.
5. Audit direct `.status =` assignments outside mixins.py/base.py:
   - `app_lifecycle_service.py:158,168` — `inst.status = STOPPED`. Verify these are valid transitions (the app is in STARTING state when startup times out → STARTING → STOPPED is in the table). If valid, no change needed. If not, fix.
   - `core.py:586` — `self.status = ResourceStatus.STOPPED` on Hassette shutdown. Verify the transition. Hassette is in RUNNING state at this point → need to check if RUNNING → STOPPED is valid (it's not — RUNNING → STOPPING → STOPPED is the correct path). This may need to be changed to go through shutdown() or set STOPPING first.
   - `harness.py:710` and `web_mocks.py:102` — `status = ResourceStatus.RUNNING`. These are test utilities setting up mock state. Change to `self._status = ResourceStatus.RUNNING` to bypass validation, since they're constructing test fixtures, not going through real lifecycle.
6. Verify `_get_service()` in ServiceWatcher returns the service instance (it already does — line 113-115). Add a `None` check with warning log.

## Review Guidance

- The STOPPING assignment must happen BEFORE `request_shutdown()` — `request_shutdown` calls `mark_not_ready()` which clears the ready event. Code that checks both `status` and `is_ready()` needs a consistent view.
- ServiceWatcher's `_get_service()` lookup is by class name — verify it matches the `name` field in the event payload. If the event uses `unique_name` but `_get_service` searches by `class_name`, the lookup will fail.
- The `core.py:586` assignment (`self.status = STOPPED`) is the most likely to be an invalid transition. The reviewer must trace what state Hassette is in at that point and verify the transition is in the table.
