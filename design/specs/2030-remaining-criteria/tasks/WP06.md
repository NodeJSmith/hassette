## Objectives & Success Criteria

- `db_max_size_mb` config field exists (default 500, 0 disables).
- `db_migration_timeout_seconds` config field exists (default 120).
- Alembic migration `005_enable_incremental_auto_vacuum.py` converts existing databases to `auto_vacuum = INCREMENTAL` via `VACUUM`.
- `_run_migrations()` is wrapped in `asyncio.wait_for()` with `db_migration_timeout_seconds`.
- `_set_pragmas()` has `auto_vacuum = INCREMENTAL` with a comment noting it is documentation-only.
- `_check_size_failsafe()` runs at startup (before write queue creation) and via `_run_size_failsafe()` hourly in `serve()`.
- Size check sums main DB + WAL + SHM files.
- Loop is capped at 10 iterations per invocation.
- Each iteration deletes 1000 oldest records, runs `incremental_vacuum(100)`, then `wal_checkpoint(TRUNCATE)`.
- Sessions table is exempt from deletion.
- Consecutive trigger warnings are logged.

## Subtasks

1. Add `db_max_size_mb: int = Field(default=500, ge=0)` to `HassetteConfig` in `src/hassette/config/config.py` after `db_retention_days`. Add docstring.
2. Add `db_migration_timeout_seconds: int = Field(default=120, ge=10)` to `HassetteConfig`. Add docstring.
3. Create Alembic migration `src/hassette/migrations/versions/005_enable_incremental_auto_vacuum.py`. Use `op.execute("PRAGMA auto_vacuum")` to check current mode; if not 2 (INCREMENTAL), set `PRAGMA auto_vacuum = INCREMENTAL` and run `VACUUM`. Log the conversion. The downgrade should be a no-op (auto_vacuum conversion is not reversible without a full VACUUM back).
4. In `src/hassette/core/database_service.py`, wrap `asyncio.to_thread(self._run_migrations)` in `asyncio.wait_for()` with `self.hassette.config.db_migration_timeout_seconds`.
5. In `_set_pragmas()`, add `await db.execute("PRAGMA auto_vacuum = INCREMENTAL")` with an inline comment: "Intentionally a no-op — auto_vacuum is set via the Alembic migration before table creation. This line documents intent only."
6. Add helper method `_get_db_size_mb()` that sums `self._db_path.stat().st_size` + WAL (`-wal`) + SHM (`-shm`) sizes, returning MB as float.
7. Add `_check_size_failsafe()` method: check `db_max_size_mb == 0` (return early), call `_get_db_size_mb()`, loop up to 10 iterations deleting oldest 1000 from `handler_invocations` and `job_executions` (by `execution_start_ts ASC LIMIT 1000`), run `PRAGMA incremental_vacuum(100)`, run `PRAGMA wal_checkpoint(TRUNCATE)`, re-check size. Log totals at end. Track and log consecutive hourly triggers.
8. Add `_run_size_failsafe()` that wraps `_check_size_failsafe()` via `enqueue()` pattern (matching `_run_retention_cleanup`).
9. Call `await self._check_size_failsafe()` in `on_initialize()` between `_set_pragmas()` and write queue creation.
10. Call `await self._run_size_failsafe()` in `serve()` loop after `_run_retention_cleanup()` (same hourly interval).
11. Add unit test `tests/unit/core/test_database_service.py::test_db_max_size_mb_zero_disables_failsafe`.
12. Add integration test `tests/integration/test_database_service.py::test_size_failsafe_deletes_oldest_records` — insert enough records to exceed a 1 MB threshold, run failsafe, verify oldest records deleted and newer ones preserved.
13. Add integration test `test_size_failsafe_loop_capped_at_10_iterations` — verify loop terminates after 10 iterations even if still over limit.
14. Add integration test `test_startup_size_check_runs` — verify `_check_size_failsafe()` is called during `on_initialize()`.
15. Add migration test `test_auto_vacuum_migration` — create a pre-existing DB without incremental auto_vacuum, run migration, verify `PRAGMA auto_vacuum` returns 2.

## Review Guidance

- Verify `_check_size_failsafe()` is called BEFORE `self._db_write_queue = asyncio.Queue()` in `on_initialize()` — this is the serialization guarantee.
- Verify the migration handles both new databases (no tables yet → PRAGMA is effective) and existing databases (tables exist → needs VACUUM).
- Verify `incremental_vacuum(100)` uses a page count argument, not bare `incremental_vacuum`.
- Verify `wal_checkpoint(TRUNCATE)` is called after vacuum to ensure `st_size` reflects actual size.
- Verify the loop cap is exactly 10 iterations and logs a WARNING when reached.
- Verify sessions table is NOT touched by the failsafe DELETEs.
