## Objectives & Success Criteria

- A "This Session" / "All Time" segmented control appears in the status bar on all pages.
- Toggling scope changes the data shown on the dashboard and app detail pages.
- The scope preference persists across page reloads via `localStorage`.
- When `sessionId` is null and scope is "current", telemetry components show a loading state (not all-time data).
- All telemetry consumers use `useScopedApi` instead of `useApi` directly.
- E2E tests verify toggle behavior and persistence.

## Subtasks

1. Add `sessionScope: signal<"current" | "all">` to `frontend/src/state/create-app-state.ts`, initialized from `localStorage` key `"sessionScope"` (default `"current"`). Use existing `getStoredValue` pattern from the theme signal.
2. Add `sessionId` param to all telemetry fetcher functions in `frontend/src/api/endpoints.ts`: `getAppHealth`, `getAppListeners`, `getAppJobs`, `getHandlerInvocations`, `getJobExecutions`, `getDashboardKpis`, `getDashboardAppGrid`, `getDashboardErrors`. Each gains `sessionId?: number | null` and appends `&session_id=<N>` or `?session_id=<N>` when provided.
3. Create `frontend/src/hooks/use-scoped-api.ts` — a wrapper around `useApi` that reads `state.sessionScope` and `state.sessionId` from `useAppState()`. When scope is `"current"` and `sessionId` is non-null, pass `sessionId` to the fetcher. When scope is `"all"`, pass `null`. When scope is `"current"` and `sessionId` is null, return loading state without firing the fetch. Include `sessionScope.value` and `sessionId.value` in the internal deps so refetch triggers on change.
4. Create `frontend/src/components/layout/session-scope-toggle.tsx` — two-button segmented control. Reads/writes `state.sessionScope`. Writes to `localStorage` on change via `setStoredValue`.
5. Embed `<SessionScopeToggle />` in `frontend/src/components/layout/status-bar.tsx`, between the connection indicator and theme toggle.
6. Convert `frontend/src/pages/dashboard.tsx` to use `useScopedApi` for `getDashboardKpis`, `getDashboardAppGrid`, `getDashboardErrors`.
7. Convert `frontend/src/pages/app-detail.tsx` to use `useScopedApi` for `getAppHealth`, `getAppListeners`, `getAppJobs`.
8. Convert `frontend/src/components/app-detail/handler-row.tsx` to use `useScopedApi` for `getHandlerInvocations`.
9. Convert `frontend/src/components/app-detail/job-row.tsx` to use `useScopedApi` for `getJobExecutions`.
10. Add unit tests for `useScopedApi` in `frontend/src/hooks/use-scoped-api.test.ts` — test scope resolution logic, null sessionId loading state, refetch on scope change.
11. Add E2E test `tests/e2e/test_session_toggle.py::test_scope_toggle_switches_data` — toggle to "All Time", verify KPI numbers change; toggle back to "This Session".
12. Add E2E test `test_scope_toggle_persists_across_reload` — set to "All Time", reload page, verify toggle is still on "All Time".

## Test Strategy

- **Unit tests** in `frontend/src/hooks/use-scoped-api.test.ts`:
  - `test_resolves_session_id_for_current_scope` — scope="current" + sessionId=5 → fetcher receives 5
  - `test_resolves_null_for_all_scope` — scope="all" → fetcher receives null
  - `test_loading_when_current_scope_and_null_session_id` — scope="current" + sessionId=null → returns loading, no fetch fired
  - `test_refetches_on_scope_change` — toggle scope, verify refetch triggered
- **E2E tests** in `tests/e2e/test_session_toggle.py` (new file):
  - `test_scope_toggle_switches_data` — toggle to "All Time", verify dashboard KPIs change
  - `test_scope_toggle_persists_across_reload` — set "All Time", reload, verify still "All Time"
  - `test_scope_toggle_visible_on_all_pages` — verify toggle present on dashboard and app detail

## Review Guidance

- Verify `useScopedApi` reads signals inside a `useSignalEffect` or equivalent so changes trigger refetch — not just in the deps array.
- Verify the sessions page (`sessions.tsx` from WP03) does NOT use `useScopedApi` — session list shows all sessions regardless of scope.
- Verify `handler-row.tsx` and `job-row.tsx` lazy-load behavior (the `lazy: true` option) is preserved when switching to `useScopedApi`.
- Verify no telemetry consumer still uses `useApi` directly for session-scoped data.
