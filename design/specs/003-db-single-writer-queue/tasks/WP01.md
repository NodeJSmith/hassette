## Objectives & Success Criteria

`DatabaseService` has a functional single-writer worker with two public submission methods. The worker serializes all submitted coroutines, propagates exceptions back to callers via Futures, and drains the queue fully before shutdown. No callsites are migrated yet — this WP delivers the plumbing only.

Observable outcomes:
- `service.submit(coro)` returns an awaitable that resolves to the coroutine's return value.
- `service.submit(coro)` re-raises the coroutine's exception at the `await` site when the coroutine fails.
- `service.enqueue(coro)` returns immediately; the coroutine runs asynchronously via the worker.
- An exception inside an enqueued coroutine is logged; the worker continues processing subsequent items.
- `on_shutdown()` blocks until all queued items have been processed, then closes the connection.
- `on_initialize()` starts the worker after opening the connection and setting pragmas.
- Existing tests in `tests/unit/core/test_database_service.py` and `tests/integration/test_database_service.py` still pass.

## Subtasks

1. Add `_db_write_queue: asyncio.Queue[tuple[Coroutine[Any, Any, Any], asyncio.Future[Any] | None]]` and `_db_worker_task: asyncio.Task[None] | None` fields to `DatabaseService.__init__` in `src/hassette/core/database_service.py`. Initialize both to sentinel values (`None`). Add required imports: `asyncio`, `typing.Any`, `collections.abc.Coroutine`.

2. Implement `async def _db_write_worker(self) -> None` in `DatabaseService`. The loop body: `await queue.get()` → try `result = await coro` → `future.set_result(result)` if future is not None → except `Exception as exc` → `future.set_exception(exc)` if future else log the exception → finally `queue.task_done()`. The loop runs until cancelled.

3. Implement `async def submit(self, coro: Coroutine[Any, Any, Any]) -> Any` in `DatabaseService`. Creates a `Future` via `asyncio.get_event_loop().create_future()`, puts `(coro, future)` on `_db_write_queue`, returns `await future`.

4. Implement `def enqueue(self, coro: Coroutine[Any, Any, Any]) -> None` in `DatabaseService`. Calls `self._db_write_queue.put_nowait((coro, None))`. Synchronous (no await needed).

5. Update `on_initialize()` in `DatabaseService`: after `await self._set_pragmas()`, add `self._db_write_queue = asyncio.Queue()` and `self._db_worker_task = asyncio.create_task(self._db_write_worker())`.

6. Update `on_shutdown()` in `DatabaseService`: before `await self._db.close()`, add `await self._db_write_queue.join()` followed by `self._db_worker_task.cancel()` and `await asyncio.gather(self._db_worker_task, return_exceptions=True)`. Guard with `if self._db_worker_task is not None`.

7. Write new unit tests in `tests/unit/core/test_database_service.py`:
   - `test_submit_returns_coroutine_result`: verify `await service.submit(coro)` returns the value the coro returns.
   - `test_submit_propagates_coroutine_exception`: verify `await service.submit(failing_coro)` raises the coro's exception.
   - `test_enqueue_is_fire_and_forget`: verify `service.enqueue(coro)` returns synchronously; coroutine completes after `await queue.join()`.
   - `test_worker_continues_after_enqueue_error`: enqueue a failing coro, then a succeeding coro; verify second coro completes after the first fails.
   - `test_worker_not_started_before_initialize`: before `on_initialize`, `_db_worker_task` is None; after, it is a running Task.

8. Write new integration test in `tests/integration/test_database_service.py`:
   - `test_drain_on_shutdown`: enqueue several slow coroutines, call `on_shutdown()`, verify all coroutines completed before shutdown returned.
   - Update `initialized_service` fixture cleanup to call `await service.on_shutdown()` instead of manually calling `await service._db.close()` — this ensures the worker is cancelled cleanly in all tests that use this fixture.

## Test Strategy

TDD — write each test before implementing the corresponding subtask where possible.

All new unit tests go in `tests/unit/core/test_database_service.py`. New integration tests go in `tests/integration/test_database_service.py`.

For unit tests: use an `initialized_service_with_worker` helper or inline fixture that calls `on_initialize()` and cancels the worker task in cleanup. Do not call `on_shutdown()` in unit tests that need to verify internals directly.

For `test_drain_on_shutdown`: use `asyncio.Event` gates to delay coroutines inside the queue, verify they complete, then verify the connection is closed after `on_shutdown()` returns.

Key test patterns from the existing codebase:
- `AsyncMock(side_effect=Exception(...))` for forcing errors
- `patch("hassette.core.database_service._HEARTBEAT_INTERVAL_SECONDS", 0.01)` pattern for time control

## Review Guidance

Verify:
- `_db_write_worker` calls `queue.task_done()` in a `finally` block — not just in the happy path. `queue.join()` deadlocks if `task_done()` is skipped on error.
- `on_shutdown()` sequence is: `queue.join()` → cancel worker → gather worker → close connection. Any other order risks processing items after the connection is closed.
- `submit()` creates the Future before putting on the queue, not after. Creating it after introduces a race where the worker could resolve the future before the caller awaits it (Python Futures are fine with this, but the pattern must be consistent).
- `enqueue()` uses `put_nowait()`, not `await put()`. It is a synchronous method that must not block the caller.
- Worker loop handles cancellation cleanly — `asyncio.CancelledError` propagates out of `await queue.get()` and exits the loop.
- No direct `db.execute()` + `db.commit()` calls added in this WP — this WP is infrastructure only.

FAIL indicators:
- `queue.task_done()` is only called on success — drain test will deadlock.
- `on_shutdown()` closes connection before `queue.join()` — queued items run against a closed connection.
- `submit()` is not type-annotated — Pyright will flag this.
