## Objectives & Success Criteria

The two internal write methods in `DatabaseService` ŌĆö `_update_heartbeat` and `_run_retention_cleanup` ŌĆö no longer call `db.execute()` + `db.commit()` directly. Their DB logic is extracted to `_do_*` private coroutines and submitted through `enqueue()`. All existing tests for these methods continue to pass.

Observable outcomes:
- `_update_heartbeat()` calls `self.enqueue(self._do_update_heartbeat())` and returns without awaiting the DB write.
- `_do_update_heartbeat()` contains the execute + commit logic that was previously inline in `_update_heartbeat()`, including the failure counter logic.
- `_run_retention_cleanup()` calls `self.enqueue(self._do_run_retention_cleanup())` and returns.
- `_do_run_retention_cleanup()` contains the execute ├Ś 2 + commit logic that was previously inline.
- All existing integration tests (`test_heartbeat_update`, `test_retention_cleanup`, `test_heartbeat_failure_counter_tracks_failures`, `test_heartbeat_recovery_resets_counter`) pass.
- No direct `db.execute()` + `db.commit()` calls remain in `_update_heartbeat()` or `_run_retention_cleanup()`.

## Subtasks

1. Extract the DB write body of `_update_heartbeat()` in `src/hassette/core/database_service.py` into a new private coroutine `_do_update_heartbeat(self) -> None`. This coroutine does: get `session_id`, get current time, `await self._db.execute(UPDATE sessions ...)`, `await self._db.commit()`, update `_consecutive_heartbeat_failures`. The early-return guards (`if self._db is None`, `except RuntimeError for session_id`) stay in `_update_heartbeat()` before the `enqueue()` call.

2. Update `_update_heartbeat()` to call `self.enqueue(self._do_update_heartbeat())` after passing the early-return guards. Remove the `try/except Exception` block from `_update_heartbeat()` ŌĆö the worker's error logging handles it. The `_consecutive_heartbeat_failures` increment on error stays ŌĆö but it now lives inside `_do_update_heartbeat()`, wrapped in `try/except`.

3. Extract the DB write body of `_run_retention_cleanup()` into `_do_run_retention_cleanup(self) -> None`. This coroutine does: get `retention_days`, compute `cutoff`, `await db.execute(DELETE ... handler_invocations)`, `await db.execute(DELETE ... job_executions)`, `await db.commit()`, log deleted counts. The `if self._db is None` guard stays in `_run_retention_cleanup()`.

4. Update `_run_retention_cleanup()` to call `self.enqueue(self._do_run_retention_cleanup())`. Remove the outer `try/except Exception` ŌĆö the worker handles error logging.

5. Update `tests/integration/test_database_service.py`: since `_update_heartbeat()` and `_run_retention_cleanup()` are now fire-and-forget, tests that verify DB state after calling these methods must drain the queue before asserting. Add `await service._db_write_queue.join()` after each call in:
   - `test_heartbeat_update`
   - `test_retention_cleanup`
   - `test_heartbeat_failure_counter_tracks_failures`
   - `test_heartbeat_recovery_resets_counter`
   - `test_serve_runs_heartbeat_and_retention`

   Also update the `initialized_service` fixture teardown to call `await service.on_shutdown()` instead of `await service._db.close()` directly ŌĆö this ensures the worker task is cancelled cleanly.

6. Verify `test_serve_raises_after_max_heartbeat_failures` still passes ŌĆö `serve()` still checks `_consecutive_heartbeat_failures` and raises after threshold. The enqueue pattern preserves this because the failure counter update is inside `_do_update_heartbeat()`.

## Test Strategy

TDD: write the queue-drain assertions first (they will fail since the method isn't async yet), then migrate the implementation.

Focus tests: `test_heartbeat_update` ŌĆö after migration, the pattern becomes:
```python
await initialized_service._update_heartbeat()
await initialized_service._db_write_queue.join()
# assert DB state changed
```

For `test_heartbeat_failure_counter_tracks_failures`: this test closes the DB to force failures. After migration, the failure increments happen inside the worker. The test must drain the queue after each `_update_heartbeat()` call to observe updated counter state:
```python
await initialized_service._update_heartbeat()
await initialized_service._db_write_queue.join()
assert initialized_service._consecutive_heartbeat_failures == 1
```

## Review Guidance

Verify:
- `_consecutive_heartbeat_failures` increment/reset logic is fully inside `_do_update_heartbeat()`, not split between `_update_heartbeat()` and the helper. If it's split, the counter won't update correctly for fire-and-forget calls.
- `_update_heartbeat()` still has the `if self._db is None: return` guard ŌĆö this guard must remain in the outer method because `enqueue()` requires the queue to exist (which requires the worker to have started).
- `_do_update_heartbeat()` is annotated as `async def ... -> None`.
- The `try/except Exception` block in `_do_update_heartbeat()` uses `self.logger.exception(...)` with the same message format as before.
- No new `db.execute()` + `db.commit()` calls are added anywhere in this WP.
- `_set_pragmas()` is not touched ŌĆö it remains exempt from the queue.

FAIL indicators:
- `_consecutive_heartbeat_failures` increment removed or moved outside the `_do_*` helper ŌĆö heartbeat failure tests will fail.
- `queue.join()` not added to integration tests ŌĆö tests will check DB state before the worker has run, producing intermittent failures.
- Fixture teardown still manually closes `_db` without cancelling worker ŌĆö worker leak in test teardown.
