## Objectives & Success Criteria

The three write methods in `CommandExecutor` â€” `register_listener`, `register_job`, and `_persist_batch` â€” no longer call `db.execute()` + `db.commit()` directly. Their DB logic is extracted to `_do_*` private coroutines submitted through `database_service.submit()`. Public method signatures are unchanged. The key startup race condition is covered by a regression test.

Observable outcomes:
- `register_listener(registration)` returns an `int` (the listener row ID) via `await self.hassette.database_service.submit(self._do_register_listener(registration))`.
- `register_job(registration)` returns an `int` (the job row ID) the same way.
- `_persist_batch(invocations, job_executions)` routes its executemany+commit via `await self.hassette.database_service.submit(self._do_persist_batch(invocations, job_executions))`.
- `_write_queue`, `_drain_and_persist`, `_flush_queue`, and `serve()` are unchanged.
- A new test `test_concurrent_registrations_do_not_raise` verifies that N concurrent `register_listener` calls complete without `OperationalError`.
- All existing `CommandExecutor` tests pass.

## Subtasks

1. Extract the DB body of `register_listener()` in `src/hassette/core/command_executor.py` into `_do_register_listener(self, registration: ListenerRegistration) -> int`. This coroutine executes the `INSERT OR CONFLICT ... RETURNING id` SQL, fetches the row ID via `cursor.fetchone()`, calls `db.commit()`, and returns the integer ID. Obtain `db` via `self.hassette.database_service.db` inside the coroutine.

2. Update `register_listener()` to call `return await self.hassette.database_service.submit(self._do_register_listener(registration))`. Remove the direct `db.execute()` / `fetchone()` / `commit()` calls from the method body.

3. Extract the DB body of `register_job()` into `_do_register_job(self, registration: ScheduledJobRegistration) -> int`. Same pattern: INSERT OR CONFLICT ... RETURNING id, fetchone, commit, return int.

4. Update `register_job()` to call `return await self.hassette.database_service.submit(self._do_register_job(registration))`.

5. Extract the DB body of `_persist_batch()` into `_do_persist_batch(self, invocations: list[HandlerInvocationRecord], job_executions: list[JobExecutionRecord]) -> None`. This coroutine does: `await db.executemany(INSERT INTO handler_invocations ...)`, `await db.executemany(INSERT INTO job_executions ...)`, `await db.commit()`.

6. Update `_persist_batch()` to call `await self.hassette.database_service.submit(self._do_persist_batch(invocations, job_executions))`.

7. Update `tests/integration/test_command_executor.py`: the `executor` fixture uses `initialized_db` which calls `on_initialize()` â€” after WP01, this starts the real DB worker. The `register_listener` and `register_job` tests now go through the worker automatically. Verify that:
   - The `mock_hassette.wait_for_ready = AsyncMock(return_value=True)` line in the `executor` fixture is sufficient for `on_initialize()` to complete.
   - No test mocks `database_service.db.execute` directly for `register_listener`/`register_job` â€” these tests already use a real DB via `initialized_db`. Confirm they work as-is.

8. Write regression test `test_concurrent_registrations_do_not_raise` in `tests/integration/test_command_executor.py`:
   - Create a real `DatabaseService` + `CommandExecutor` wired together.
   - Concurrently `await asyncio.gather(*[executor.register_listener(reg) for reg in regs])` for N=10 distinct registrations.
   - Assert all return valid positive integer IDs with no exception raised.
   - This test directly validates the fix for the startup race.

## Review Guidance

Verify:
- `_do_register_listener` and `_do_register_job` obtain `db` via `self.hassette.database_service.db` **inside the coroutine body**, not outside. The coroutine runs inside the worker; if `db` is captured before submission, it's fine, but the `db` reference must point to the real connection.
- `_do_persist_batch` takes the invocations and job_executions as parameters â€” these are plain Python objects, not coroutines or futures. The coroutine is closed over their values at creation time.
- `_write_queue`, `_drain_and_persist`, `_flush_queue`, `serve()` are completely unchanged. Open these methods and confirm no edits.
- `register_listener` and `register_job` return types are still `int` (or `int | None` if that was the original) â€” check the original return type annotation and preserve it.
- `test_concurrent_registrations_do_not_raise` uses a real `DatabaseService` (not mocked) â€” the whole point is to verify real SQLite serialization.

FAIL indicators:
- `_do_register_listener` captures `db = self.hassette.database_service.db` outside the coroutine body â€” this is fine but note it.
- `_persist_batch` still has direct `db.execute()` calls â€” test isolation issues will appear at runtime.
- `_write_queue` or `_flush_queue` is modified â€” out of scope; revert.
