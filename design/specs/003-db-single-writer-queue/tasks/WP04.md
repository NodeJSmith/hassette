## Objectives & Success Criteria

The four session write methods in `Hassette` (`core.py`) — `_mark_orphaned_sessions`, `_create_session`, `_finalize_session`, and `_on_service_crashed` — no longer call `db.execute()` + `db.commit()` directly. Their DB logic is extracted to `_do_*` private coroutines submitted through `database_service.submit()` or `database_service.enqueue()`. Session lifecycle tests continue to pass.

Observable outcomes:
- `_mark_orphaned_sessions()` calls `await self._database_service.submit(self._do_mark_orphaned_sessions())`.
- `_create_session()` calls `self._session_id = await self._database_service.submit(self._do_create_session())`. The session ID is assigned in the calling context (`run_forever`), not inside the method.
- `_finalize_session()` calls `await self._database_service.submit(self._do_finalize_session())`.
- `_on_service_crashed(event)` calls `self._database_service.enqueue(self._do_on_service_crashed(event))` — fire-and-forget.
- All existing tests in `tests/integration/test_session_lifecycle.py` pass.
- No direct `self._database_service.db.execute()` + `commit()` calls remain in these four methods.

## Subtasks

1. In `src/hassette/core/core.py`, extract the DB body of `_mark_orphaned_sessions()` into `_do_mark_orphaned_sessions(self) -> None`. This coroutine executes `UPDATE sessions SET status = 'orphaned' WHERE status = 'running'` (or equivalent) and commits.

2. Update `_mark_orphaned_sessions()` to call `await self._database_service.submit(self._do_mark_orphaned_sessions())`. Remove direct `db.execute()` + `db.commit()` calls.

3. Extract the DB body of `_create_session()` into `_do_create_session(self) -> int`. This coroutine executes the `INSERT INTO sessions` row, calls `db.commit()`, and returns `cursor.lastrowid`. The return type is `int`.

4. Update `_create_session()` to: `self._session_id = await self._database_service.submit(self._do_create_session())`. The session ID assignment moves here (or stays in `run_forever` if `_create_session` was a void method that set `self._session_id` inline — check and adapt). Remove direct `db.execute()` + `db.commit()` from `_create_session()`.

5. Extract the DB body of `_finalize_session()` into `_do_finalize_session(self) -> None`. This coroutine executes the `UPDATE sessions SET status = 'finished', ended_at = ?` (or equivalent) and commits.

6. Update `_finalize_session()` to call `await self._database_service.submit(self._do_finalize_session())`.

7. Extract the DB body of `_on_service_crashed()` into `_do_on_service_crashed(self, event: HassetteServiceEvent) -> None`. This coroutine executes the `UPDATE sessions SET status = 'crashed'` (or equivalent) and commits.

8. Update `_on_service_crashed()` to call `self._database_service.enqueue(self._do_on_service_crashed(event))`. This is fire-and-forget — the crash handler does not await the write.

9. Update `tests/integration/test_session_lifecycle.py`: the existing tests bind `Hassette._create_session` (and other methods) to `mock_hassette` via unbound method calls. After migration, these tests need:
   - A real `DatabaseService` with a running worker (already provided by the `db_service` fixture which calls `on_initialize()`).
   - The mock hassette's `_database_service` to point to this real `db_service`.
   - Where tests drain state via `await db_service.db.execute(SELECT ...)` after calling the session methods, add `await db_service._db_write_queue.join()` to ensure writes have completed before assertions.
   - Update `db_service` fixture teardown to call `await service.on_shutdown()` instead of manually closing `service._db`.

## Test Strategy

TDD — read `tests/integration/test_session_lifecycle.py` fully before implementing. For each session method test:
1. Add `await db_service._db_write_queue.join()` after calling the method (needed for `enqueue()` paths; harmless for `submit()` paths since submit already awaits).
2. The `_create_session` test pattern: `mock_hassette._database_service = db_service` must be set before calling the method. After migration, `_create_session` sets `self._session_id` (on mock_hassette) from the Future result — verify the test's assertion on `mock_hassette._session_id` still holds.

For `_on_service_crashed`: this uses `enqueue()` so the write is fire-and-forget. Tests must drain the queue before asserting DB state:
```python
Hassette._on_service_crashed(mock_hassette, event)
await db_service._db_write_queue.join()
# assert session status changed
```

## Review Guidance

Verify:
- `_create_session` return chain: `_do_create_session` returns `int` → `submit()` resolves the Future to that int → `_create_session` assigns it to `self._session_id`. Confirm `self._session_id` is set exactly once, by `_create_session` (or wherever it was set before — preserve the existing assignment site).
- `_on_service_crashed` uses `enqueue()`, not `submit()`. The caller does not await the DB write — this is correct per design decision.
- No other methods in `core.py` call `self._database_service.db` directly. After this WP, there should be zero `.db.execute()` + `.commit()` calls in `core.py`.
- The `db_service` fixture in `test_session_lifecycle.py` must call `on_shutdown()` in teardown, not manually close `_db`. After WP01, `on_shutdown()` cancels the worker; bypassing it leaks the worker task.
- `_do_*` helpers in `core.py` obtain the connection via `self._database_service.db` — same as before, just inside the submitted coroutine.

FAIL indicators:
- `_on_service_crashed` uses `submit()` instead of `enqueue()` — minor but diverges from design.
- `_create_session` no longer sets `self._session_id` (or sets it in the wrong place) — downstream uses of `self.session_id` will raise `RuntimeError`.
- `db_service` fixture teardown doesn't cancel the worker task — test suite will emit "Task was destroyed but it is pending" warnings.
- Any `_do_*` method in `core.py` directly accesses `self._database_service.db` before the worker has started — guard with `RuntimeError` same as the existing `.db` property.
