# Design: Preserve Listener/Job Registrations Across Sessions

**Date:** 2026-04-06
**Status:** archived
**Research:** design/research/2026-04-06-stable-identity-upsert/research.md, design/research/2026-04-06-predicate-description-stability/research.md
**Issue:** #466

## Problem

When an app restarts, `clear_registrations(app_key)` deletes all listener and scheduled_job rows. Due to `ON DELETE SET NULL` foreign keys, historical `handler_invocations` and `job_executions` rows lose their parent references (set to NULL). This causes "All Time" scope queries to return identical data as "This Session" because INNER JOINs exclude NULL parent references.

The root cause: registration row IDs are ephemeral — destroyed and recreated on every restart — so FK references from execution history cannot survive across sessions.

## Non-Goals

- **Active/soft-delete flag**: Prior art (Brandur Leach, Cultured Systems, Dingwall) strongly criticizes this pattern. Every query must remember `WHERE active = 1`, and forgotten filters produce incorrect results. The upsert approach avoids this entirely.
- **Event-sourced registration history**: Overkill for this use case. Registration config rarely changes between restarts.
- **Migrating existing NULL FK references**: Existing orphaned `handler_invocations`/`job_executions` rows with `listener_id = NULL` or `job_id = NULL` cannot be retroactively linked. This fix prevents future orphaning only.
- **Batch-collect registration model**: The existing architecture registers listeners incrementally during `on_initialize()`/`on_ready()`. A batch-collect model would require a new lifecycle phase and buffering infrastructure. The incremental upsert model matches the existing architecture.

## Architecture

### Natural Key and Upsert Mechanism

Use `INSERT ... ON CONFLICT DO UPDATE` with a unique index on the registration's natural key. Rows are updated in place rather than deleted and re-inserted, preserving the integer PK that child tables reference via FK.

Each `register_listener()` / `register_job()` call individually upserts — same call sites as today, but with `ON CONFLICT DO UPDATE` semantics instead of plain `INSERT`. No batch collection phase needed.

**Listener natural key**: `(app_key, instance_index, handler_method, topic, COALESCE(name, human_description, ''))`

- `human_description` is generated by `predicate.summarize()` (`bus_service.py:98-100`), which produces deterministic strings from data fields — no memory addresses, no function references.
- `predicate_description` (from `repr()`) is **NOT** part of the natural key — it includes memory addresses for `ValueIs`, `Guard`, `DidChange`, `IsPresent`, `IsMissing` predicates and is unstable across restarts.

**Scheduled job natural key**: `(app_key, instance_index, job_name)`

**Stability contract**: `summarize()` output is identity-stable after deployment. Changes to its format are breaking changes that require either a migration updating existing `human_description` values or explicit acceptance of history orphaning.

### Unique Index (not UNIQUE constraint)

SQLite does not allow expressions in `UNIQUE(...)` table constraints. Use `CREATE UNIQUE INDEX` instead:

```sql
CREATE UNIQUE INDEX idx_listeners_natural
    ON listeners(app_key, instance_index, handler_method, topic, COALESCE(name, human_description, ''))
    WHERE once = 0;

CREATE UNIQUE INDEX idx_scheduled_jobs_natural
    ON scheduled_jobs(app_key, instance_index, job_name);
```

The listener index is a **partial index** (`WHERE once = 0`) — `once=True` listeners are excluded from uniqueness enforcement entirely, allowing unlimited fresh inserts for ephemeral registrations.

### ValueIs.summarize() Fix (Pre-existing Bug)

`ValueIs.summarize()` currently returns `"custom condition"` for ALL conditions — including literal values like `"on"` or `ResourceStatus.RUNNING`. This is a pre-existing bug that makes collisions broader than just callables.

**Fix**: `ValueIs.summarize()` returns `f"value is {self.condition} from {callable_name(self.source)}"` when the condition is non-callable. For callable conditions with non-callable sources, include the source accessor name: `f"custom condition from {callable_name(self.source)}"`. For fully-callable predicates (both source and condition are callables), return `"custom condition"` — the `name=` escape hatch covers this case.

This addresses the source-callable omission: two `ValueIs` predicates with the same condition but different sources (e.g., `get_state_value_new` vs `get_attr_new("brightness")`) now produce distinct `human_description` values.

**Known limitation**: `ServiceDataWhere.summarize()` formats from raw spec key-value pairs. If callable conditions appear in a `ServiceDataWhere` spec, they will stringify unstably. Verify during implementation whether this path is exercised; add a companion fix if so.

### Collision Detection at Registration Time

Collision detection lives in `Bus.on()` (and `on_state_change()`, `on_attribute_change()`, `on_call_service()`) — the **synchronous** call site where `ValueError` propagates directly to user code in `on_initialize()`. This is critical: detection in the async `BusService._register_then_add_route()` would be swallowed by the task runner and logged rather than raised.

**Data structure**: Each `Bus` instance maintains a `_registered_keys: set[tuple[str, int, str, str, str]]` — a per-app-instance set of natural key tuples `(app_key, instance_index, handler_method, topic, coalesce(name, human_description, ''))`. Populated synchronously when `add_listener()` is called. **Cleared at the start of initialization** (not only on shutdown) to prevent false-positive collisions after partial initialization failures where `on_shutdown` may not fire.

**Validation rules**:
- If the natural key tuple is already in `_registered_keys`, raise `ValueError` with a clear message directing the user to add `name=` to disambiguate.
- If two listeners provide the same `name=` for the same `(handler_method, topic)`, raise `ValueError("Duplicate listener name '...' for handler '...' on topic '...'")`.

### Optional `name=` Escape Hatch

Add `name: str | None = None` as an **explicit named parameter** on `Bus.on()`, `Bus.on_state_change()`, `Bus.on_attribute_change()`, `Bus.on_call_service()`. The `name` must be extracted explicitly in `on()` (like `once`, `debounce`, `throttle`) — not passed through `**opts` — because `Bus.on()` uses explicit named args, not `**opts` passthrough. The `Options` TypedDict is updated to include `name` for type checking, but extraction is explicit.

When provided, `name` replaces `human_description` in the natural key for that registration.

Propagation chain: `Bus.on()` → `Listener.create(name=name)` → `Listener.name` field → `BusService._register_then_add_route()` → `ListenerRegistration(name=name)` → `TelemetryRepository.register_listener()`.

The `name=` parameter is only for listeners — scheduled jobs use `job_name` as the stable identifier.

### `once=True` Handling

Listeners registered with `once=True` always use plain `INSERT` — they never participate in upsert. The partial unique index (`WHERE once = 0`) excludes them from uniqueness enforcement.

**Lifecycle**: `once=True` listeners are visible in the current session (they have a DB row with invocations). On restart, reconciliation deletes `once=True` rows from previous sessions, guarded by a NOT EXISTS check to preserve rows whose invocations haven't been flushed yet:

```sql
DELETE FROM listeners
WHERE app_key = ? AND once = 1
  AND id NOT IN (<live_once_ids_this_session>)
  AND NOT EXISTS (
    SELECT 1 FROM handler_invocations
    WHERE listener_id = listeners.id AND session_id = ?
  )
```

FK cascades set `listener_id = NULL` in `handler_invocations` for deleted rows, preserving invocation records in anonymous form. This bounds growth to at most one row per `once=True` listener per live session.

**Note**: `once` is NOT a mutable upsert field. A `once=True` registration is a semantically different object from a `once=False` registration — changing `once` between restarts produces a new row (new identity), not an update.

### Incremental Upsert + Post-Ready Reconciliation

Registration and reconciliation are two independent operations:

**1. Incremental upsert (during on_initialize/on_ready):**

Each `register_listener()` / `register_job()` call uses `INSERT ... ON CONFLICT DO UPDATE SET <mutable fields> RETURNING id`. Same call sites as today. Returns the row ID (existing or new). The upsert also resets `retired_at = NULL` on reactivated rows.

**2. Post-ready reconciliation (inside initialize_instances):**

Reconciliation runs inside `initialize_instances()` after the `for idx, inst in enumerate(instances)` loop completes. Before reconciling, an **explicit await barrier** ensures all pending registration tasks have flushed:

1. `BusService` tracks pending registration tasks per `app_key` and exposes `await_registrations_complete(app_key)`. If a registration task errors (DB unavailable), it completes with `db_id = None` — these listeners are excluded from `live_listener_ids` but are not actively retired (they'll be re-registered on the next restart). The barrier has a timeout (config-driven, default 30s) to prevent indefinite hangs if the DB write queue stalls.
2. `initialize_instances()` calls `await bus_service.await_registrations_complete(app_key)` and the scheduler equivalent.
3. `live_listener_ids` are collected by iterating all instances for the app via the registry, accessing each instance's `Bus.get_listeners()`, and extracting `{listener.db_id for listener in listeners if listener.db_id is not None}`. This is `app_key`-scoped (via registry iteration) and instance-aware — the Router's `get_listeners_by_owner(owner_id)` uses `owner_id` (which is `{app_key}:{instance_index}`), not `app_key` directly, so the collection iterates per-instance.
4. `live_job_ids` are collected similarly from each instance's `Scheduler` — iterating registered jobs and extracting `db_id` values.
5. `reconcile_registrations(app_key, live_listener_ids, live_job_ids)` is called.

**Reconciliation SQL:**

- Delete stale non-once rows without history:
  ```sql
  DELETE FROM listeners WHERE app_key = ? AND once = 0
    AND id NOT IN (<live_ids>)
    AND NOT EXISTS (SELECT 1 FROM handler_invocations WHERE listener_id = listeners.id)
  ```
- Set `retired_at = now()` on stale non-once rows WITH history.
- Delete `once=True` rows from previous sessions (with NOT EXISTS guard — see once=True section).
- Same pattern for `scheduled_jobs` / `job_executions`.

**Safety guard**: The reconciliation DELETE also excludes listener IDs from the in-memory Router (`{listener.db_id for listener in router.get_listeners_by_owner(app_key)}`) to prevent deleting rows that currently-executing handlers depend on. The Router guard is wrapped in try/except — if it raises, reconciliation proceeds with `live_listener_ids` only (conservative fallback).

**Empty ID list guard**: When `live_listener_ids` is empty (scheduler-only app), the reconciliation step omits the `NOT IN (...)` clause and uses only the `NOT EXISTS` subquery to avoid SQLite's `NOT IN ()` syntax error.

**Error handling**: Reconciliation failure (disk full, WAL contention) logs a warning and allows the app to continue running. Stale rows are a display artifact, not a correctness issue. No retry is scheduled — the next restart reconciles again.

This replaces the current `clear_registrations()` call in `app_lifecycle_service.py:306`.

### Retired Registration Tracking

Add `retired_at REAL` (nullable timestamp) to `listeners` and `scheduled_jobs`. Set during reconciliation for stale-with-history rows. Reset to `NULL` when a retired row is re-upserted (handler re-added to code).

**Enforcement via database views**: To avoid the "forgotten filter" problem that made `active = 1` untenable, registration count queries use database views rather than raw table access:

```sql
CREATE VIEW active_listeners AS SELECT * FROM listeners WHERE retired_at IS NULL;
CREATE VIEW active_scheduled_jobs AS SELECT * FROM scheduled_jobs WHERE retired_at IS NULL;
```

- Registration count queries (`get_global_summary`, `get_all_app_summaries`) use `active_listeners` / `active_scheduled_jobs`.
- History queries (`get_listener_summary`, `get_recent_errors`, `get_slow_handlers`) use the base tables directly — they intentionally include retired registrations.
- Forgetting the filter is now a schema error (wrong table name), not a logic error (missing WHERE clause).

**Retired row cleanup**: Extend `_do_run_retention_cleanup()` in `DatabaseService` to delete retired registration rows: `DELETE FROM listeners WHERE retired_at IS NOT NULL AND retired_at < (now - retention_days * 86400)`. Same for `scheduled_jobs`. Uses the same retention window as invocation records.

### Mutable Fields (Updated on Upsert)

Listener: `debounce`, `throttle`, `priority`, `predicate_description`, `source_location`, `registration_source`, `retired_at` (reset to NULL)

`once` is NOT mutable — it is part of the listener's behavioral identity. The partial index (`WHERE once = 0`) means `once=True` rows are invisible to the upsert conflict target.

Scheduled job: `handler_method`, `trigger_type`, `trigger_value`, `repeat`, `args_json`, `kwargs_json`, `source_location`, `registration_source`, `retired_at` (reset to NULL)

`human_description` is part of the unique key and is NOT updated (it's the identity). `once` is NOT mutable — changing it produces a new row.

### Schema Changes

Since no other users exist, the migration can be simple — drop and recreate the registration tables with the new schema. Existing data is expendable.

**listeners table changes:**
- Add `name TEXT` column (nullable, for the optional escape hatch)
- Add `retired_at REAL` column (nullable)
- Add unique expression index: `idx_listeners_natural` (partial, `WHERE once = 0`)

**scheduled_jobs table changes:**
- Add `retired_at REAL` column (nullable)
- Add unique index: `idx_scheduled_jobs_natural`

**Database views:**
- `active_listeners` — `SELECT * FROM listeners WHERE retired_at IS NULL`
- `active_scheduled_jobs` — `SELECT * FROM scheduled_jobs WHERE retired_at IS NULL`

**Child table changes:**
- Add single-column index: `idx_hi_listener_id ON handler_invocations(listener_id)` (covering index for reconciliation NOT EXISTS subquery)

**FK behavior**: Keep `ON DELETE SET NULL` — needed for reconciliation of `once=True` rows and stale-with-history rows cleaned by retention.

### Query Service Impact

- **`get_recent_errors()` and `get_slow_handlers()`**: INNER JOINs are now correct for "All Time" scope — persistent rows are never deleted during normal operation. Update stale comments that reference `clear_registrations()`.
- **`get_listener_summary()` and `get_job_summary()`**: Query base tables (not views). Return both active and retired registrations for the app. Retired rows have invocation history to show in "All Time" view.
- **`get_global_summary()` and `get_all_app_summaries()`**: Registration counts use `active_listeners` / `active_scheduled_jobs` views. Activity counts (invocations, errors, executions) aggregate across all registrations including retired.

### CommandExecutor Delegation

```python
# CommandExecutor — individual upsert (same pattern as current register_listener)
async def register_listener(self, registration: ListenerRegistration) -> int:
    await self.hassette.wait_for_ready([self.hassette.database_service])
    return await self.hassette.database_service.submit(
        self.repository.register_listener(registration)
    )

# CommandExecutor — post-ready reconciliation (new method)
async def reconcile_registrations(
    self,
    app_key: str,
    live_listener_ids: list[int],
    live_job_ids: list[int],
) -> None:
    await self.hassette.wait_for_ready([self.hassette.database_service])
    await self.hassette.database_service.submit(
        self.repository.reconcile_registrations(app_key, live_listener_ids, live_job_ids)
    )
```

### `mark_registered()` Guard

The `Listener.mark_registered()` and `ScheduledJob.mark_registered()` idempotency guards are unaffected. With upsert, `register_listener()` may return an existing row ID, but the in-memory `Listener` object is freshly created on each startup with `db_id = None`, so the guard won't fire. No change needed.

## Alternatives Considered

### Soft-delete with `active` flag (rejected)

The original plan. Criticized by multiple authoritative sources (Brandur Leach, Cultured Systems, Richard Dingwall) as creating more problems than it solves. Every query must remember `WHERE active = 1`. The challenge found 6+ query sites that would need the filter, with no enforcement mechanism. The upsert approach avoids this entirely — rows are simply updated in place.

### Session-scoped registration table (rejected)

Append new registration rows per session with a `session_id` FK. "All Time" queries group by natural key across sessions. Rejected because it re-derives "registration identity" at query time — the same problem upsert solves at write time, but with more complex queries.

### Natural key as primary key (rejected for now)

Use a string PK like `"myapp.on_light_change.light.kitchen"` instead of integer autoincrement. Eliminates the surrogate-vs-natural key mismatch. Rejected because it requires migrating all FK references in `handler_invocations` and `job_executions` from integer to string, which is a larger change than needed.

### Batch-collect model with sync_registrations() (rejected)

Collect all registrations before any DB writes, then upsert+reconcile in one transaction. Rejected because the existing architecture registers incrementally during `on_initialize()`/`on_ready()` — there is no "all registrations done" lifecycle signal. Adding one would require buffering, a new lifecycle phase, and changing when `db_id` becomes available. The incremental model (individual upserts + post-ready reconciliation) matches the existing architecture and is how APScheduler handles the same pattern.

## Test Strategy

- **Unit tests** (`tests/unit/core/test_telemetry_repository.py`):
  - Upsert: same natural key returns same ID, different key returns new ID
  - Upsert resets `retired_at` to NULL on reactivated rows
  - `once=True` always inserts (new ID each time)
  - Reconciliation deletes stale rows without history
  - Reconciliation sets `retired_at` on stale rows with history
  - Reconciliation deletes `once=True` rows from previous sessions (guarded by NOT EXISTS)
  - Reconciliation preserves `once=True` rows with current-session invocations
  - Empty `live_ids` list doesn't crash reconciliation
  - **Delete** existing tests that encode the reversed "always-new-ID" contract: `test_register_listener_two_calls_return_distinct_ids`, `test_register_job_two_calls_return_distinct_ids`

  - `name=` parameter propagates through `on()` → `Listener.create()` → `Listener.name`
- **Integration tests** (`tests/integration/test_command_executor.py`):
  - FK preserved across simulated restart — register listener, create invocation, upsert with same key, assert invocation still references same listener_id
  - Reconciliation ordering test (replaces `clear_registrations` assertion)
  - **Delete** existing "always-new-ID" tests: `test_register_listener_creates_separate_rows`, `test_register_job_creates_separate_rows`

**Implementation note**: SQLite `SELECT *` views don't auto-update when base table columns change. Future migrations adding columns to `listeners`/`scheduled_jobs` must also drop/recreate the `active_listeners`/`active_scheduled_jobs` views.

## Open Questions

- **Guard/ValueIs relationship**: Investigated in WP01. **Not redundant — do not consolidate.**
  - `Guard(fn)` wraps an arbitrary `Predicate[EventT]` — a single callable that takes a full event and returns bool. It has no internal structure (no source/condition split).
  - `ValueIs(source, condition)` composes a value extractor (`source`) with a condition test (`condition`). It carries structured fields that participate in `callable_name()` summarization.
  - The two serve orthogonal purposes: `Guard` is an escape hatch for any predicate callable; `ValueIs` is structured source+condition composition. Consolidating would require giving `Guard` a source/condition split, which is not how it's used. Both retain `summarize()` implementations appropriate to their structure.
