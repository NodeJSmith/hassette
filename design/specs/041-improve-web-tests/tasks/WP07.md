## Objectives & Success Criteria

At least one E2E test exercises the WebSocket session path (scope='current' + session ID) end-to-end. MSW `onUnhandledRequest` is switched from `'warn'` to `'error'` now that all component tests and handlers are in place. Both `npm test` and `uv run nox -s e2e` pass.

## Subtasks

1. Add a new E2E test fixture in `tests/e2e/conftest.py` that enables WebSocket (does not pass `ws='none'`) and does not force `sessionScope='all'`. This may require a separate mock server instance or a conditional fixture
2. Add `test_websocket_session_scoped_fetch` to `tests/e2e/test_websocket.py`: connect via WebSocket, receive `connected` message with `session_id`, verify that dashboard data fetches include `?session_id=N` in the scope='current' mode
3. Switch `onUnhandledRequest` from `'warn'` to `'error'` in `frontend/src/test-setup.ts`
4. Run `npm test` and verify all frontend tests pass with strict MSW enforcement — any missing handler is now a test failure
5. Document the E2E WebSocket gap closure in `tests/e2e/conftest.py` with a brief comment explaining the two server configurations (ws=none for most tests, ws=enabled for session path test)

## Test Strategy

- `tests/e2e/test_websocket.py::test_websocket_session_scoped_fetch` — new E2E test verifying the default user flow: WS connects → session ID received → API calls include session_id parameter
- Frontend: all existing tests must pass with `onUnhandledRequest: 'error'` — this is the enforcement step that validates handler completeness from WP01

## Review Guidance

- The E2E WebSocket test must exercise the real path — not mock the session ID, but receive it from the WebSocket connection
- Verify `onUnhandledRequest` is now `'error'` not `'warn'` in `test-setup.ts`
- If any frontend test fails after the switch, the fix is adding the missing handler to `handlers.ts`, not weakening enforcement
