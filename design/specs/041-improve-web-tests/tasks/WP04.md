## Objectives & Success Criteria

All untested app-detail components (`handler-row.tsx`, `handler-list.tsx`, `handler-invocations.tsx`, `health-strip.tsx`, `error-display.tsx`, `error-cell.tsx`, `job-executions.tsx`) and apps components (`manifest-row.tsx`, `status-filter.tsx`) have thorough test files. Lazy-loaded components (handler-row, job-executions, handler-invocations) test both collapsed and expanded states. `npm test` passes.

## Subtasks

1. Create `frontend/src/components/app-detail/handler-row.test.tsx` — uses `renderWithAppState()` helper since it calls `useScopedApi`. Test: collapsed state shows handler name and invocation count, expand click triggers API fetch via MSW, expanded state shows invocation detail, error indicator when `failed > 0`
2. Create `frontend/src/components/app-detail/handler-list.test.tsx` — renders list of handler rows, empty list state, sorting if applicable
3. Create `frontend/src/components/app-detail/handler-invocations.test.tsx` — renders invocation table with status, duration, timestamp columns. Test: success and error rows styled differently, error rows show error type/message
4. Create `frontend/src/components/app-detail/health-strip.test.tsx` — renders health metrics (error rate, handler avg, job avg, last activity). Test: each metric rendered, classify colors (good/warn/bad), zero values handled
5. Create `frontend/src/components/app-detail/error-display.test.tsx` — renders error message and traceback. Test: with traceback (expandable), without traceback (message only), null/empty error
6. Create `frontend/src/components/app-detail/error-cell.test.tsx` — renders inline error indicator. Test: with error (shows icon/text), without error (hidden or empty)
7. Create `frontend/src/components/app-detail/job-executions.test.tsx` — renders execution table, success/error status styling, duration formatting
8. Create `frontend/src/components/apps/manifest-row.test.tsx` — renders manifest info in row format: app key, class name, status, instance count. Test: each status variant, action buttons presence
9. Create `frontend/src/components/apps/status-filter.test.tsx` — renders filter dropdown, fires onChange with selected status, "All" option clears filter

## Test Strategy

- `handler-row.test.tsx` — uses `renderWithAppState()` + MSW `server.use()` for lazy-loaded invocations. Tests: `test_collapsed_shows_handler_name`, `test_expand_fetches_invocations`, `test_expanded_shows_invocation_table`, `test_error_indicator_shown_when_failed`, `test_collapsed_hides_invocations`
- `handler-invocations.test.tsx` — pure prop test with factory data. Tests: `test_success_row_styling`, `test_error_row_shows_type_and_message`, `test_empty_invocations`
- `health-strip.test.tsx` — mock hook (render-with-known-data). Tests: `test_all_metrics_rendered`, `test_error_rate_color_classes`, `test_zero_values`
- `error-display.test.tsx` — pure prop test. Tests: `test_with_traceback_expandable`, `test_without_traceback`, `test_null_error`
- `status-filter.test.tsx` — render + user interaction. Tests: `test_renders_filter_options`, `test_fires_onchange`, `test_all_clears_filter`

## Review Guidance

- `handler-row` must use `renderWithAppState()` not `vi.mock(useScopedApi)` — it tests data-fetching behavior (expand → fetch)
- `health-strip` and `handler-list` should use `vi.mock(hook)` — they test rendering given known data
- Verify all lazy-loaded components test both collapsed AND expanded states
