## Objectives & Success Criteria

Backend web API has integration tests for all input validation constraints, thin endpoint coverage gaps (invocations/executions/sessions limit and ordering), and WebSocket protocol edge cases. `uv run nox -s dev -- -n 2` passes.

## Subtasks

1. Add `TestAppKeyValidation` class to `tests/integration/test_web_api.py`:
   - Invalid `app_key` format on POST start/stop/reload (e.g., `"!!invalid"`, `""`, key longer than 128 chars) — verify 422 response
   - Non-existent `app_key` on start/stop/reload — verify appropriate error response
2. Add `TestLimitParameterValidation` class to `tests/integration/test_web_api.py`:
   - Out-of-range `limit` on `/api/events/recent` (0, -1, 501) — verify 422
   - Out-of-range `limit` on `/api/logs/recent` (0, 2001) — verify 422
   - Out-of-range `limit` on `/api/telemetry/sessions` (0, 201) — verify 422 (check if already covered by existing `TestTelemetrySessionsEndpoint`)
   - Out-of-range `limit` on `/api/telemetry/handler/{id}/invocations` (0, 501) — verify 422
   - Out-of-range `limit` on `/api/telemetry/job/{id}/executions` (0, 501) — verify 422
3. Add `TestHandlerInvocationsExpanded` class — extend existing single-test coverage:
   - `test_limit_param_respected` — verify only N results returned when limit=N
   - `test_empty_result` — verify empty list returned for listener with no invocations
   - `test_ordering` — verify results ordered by `execution_start_ts` descending
4. Add `TestJobExecutionsExpanded` class — mirror of handler invocations:
   - `test_limit_param_respected`, `test_empty_result`, `test_ordering`
5. Add `TestSessionsOrdering` — verify sessions returned newest-first
6. Add WebSocket edge case tests to `tests/integration/test_ws_endpoint.py`:
   - `test_malformed_json_message` — send non-JSON string via `ws.send_text("{bad")`, verify the connection closes gracefully without crashing the server. If Starlette TestClient absorbs the error silently, add a unit test on the `_read_client` path instead
   - `test_unknown_message_type` — send `{"type": "unknown_type"}`, verify connection stays open and message is silently ignored (or logged)
   - `test_subscribe_with_invalid_level` — send subscribe with `min_log_level: "INVALID"`, verify fallback to INFO (check existing `test_subscribe_invalid_level_defaults_to_info` first — skip if already covered)
   - `test_subscribe_with_missing_fields` — send `{"type": "subscribe", "data": {}}`, verify reasonable default behavior

## Test Strategy

- All tests in `tests/integration/test_web_api.py` or `tests/integration/test_ws_endpoint.py`
- Use existing fixture chain: `mock_hassette` → `runtime_query_service` → `app` → `client`
- Test functions named per existing convention: `test_<behavior>` inside test classes
- Run: `timeout 300 uv run pytest tests/integration/test_web_api.py tests/integration/test_ws_endpoint.py -v -n 2`

## Review Guidance

- Before writing limit validation tests, check existing coverage — `TestTelemetrySessionsEndpoint` may already cover session limit validation
- WebSocket tests should verify the connection survives invalid messages — the server must not crash
- Verify invalid `app_key` tests use the actual regex from the route definition, not a guess
