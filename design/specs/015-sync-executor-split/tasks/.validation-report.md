## Findings

### Finding 1 — AC#1's grep is unsatisfiable as literally written (HIGH)

AC#1: `grep -rn 'contextlib.suppress' src/hassette/task_bucket/task_bucket.py` should
return no hits. Confirmed against the actual file, there are **two** occurrences today:

```
task_bucket.py:122:        with contextlib.suppress(ValueError):
task_bucket.py:384:    with contextlib.suppress(RuntimeError):
```

Line 122 is in `uninstall_exception_recorder()` — completely unrelated to the sync
executor split, and no task (T01/T02/T03) touches it. Only line 384 (the factory's
`RuntimeError` suppression) is in scope for removal. After a fully correct T02
implementation, line 122's `contextlib.suppress(ValueError)` still remains, so the
literal grep command in AC#1 (and reproduced verbatim in T02's Verify checklist)
**will always find a hit and can never return "no hits."**

Fix: scope the check, e.g. `grep -n 'contextlib.suppress(RuntimeError)' src/hassette/task_bucket/task_bucket.py` or grep near `_create_task_bucket` specifically. Update both the design doc's AC#1 and T02's Verify line together.

### Finding 2 — AC#2's grep is unsatisfiable given T02's own prescribed fix (HIGH)

AC#2: `grep -n 'task_bucket._sync' src/hassette/core/core.py` should return no hits.
Today this matches the two post-hoc patch lines in `wire_services()` (lines 192-193):

```
core.py:192:        self.task_bucket._sync_service = self._sync_executor_service
core.py:193:        self._sync_executor_service.task_bucket._sync_service = self._sync_executor_service
```

T02 removes those two lines — but T02 also *adds* a new line in `Hassette.__init__`:

```
self.task_bucket._sync_executor = self._sync_executor
```

This line contains the substring `task_bucket._sync`, so it **will still match** the
AC#2 grep pattern after the "fix" is applied. T02's own Verify checklist reproduces
AC#2 verbatim and appends a note ("the one explicit line in `__init__` ... is not a
`wire_services()` patch") — but a plain-text grep cannot distinguish which function a
matching line lives in. As written, the check will still report a hit even in a
fully correct implementation, and the note doesn't change that outcome.

Fix: scope the grep to the `wire_services()` function body only (e.g. `sed -n '/def wire_services/,/^    def /p' core.py | grep 'task_bucket._sync'`), or rewrite AC#2 as "the only occurrence of `task_bucket._sync` in `core.py` is the one line in `__init__`" with a check that verifies count == 1 and its location. Update both the design doc and T02's Verify line together.

### Finding 3 — T01 breaks `test_sync_executor_service_wiring.py`'s imports before T02 fixes them (HIGH)

T01 is described as "purely additive... the current `SyncExecutorService`, which
still works as-is" and its own Verify section only greps for the new class — it does
not run tests. But T01's prompt also directs modifying `tests/unit/conftest.py` to
**replace** `make_sync_executor_config`, `make_sync_executor_hassette`, and
`make_service` (i.e., remove/rename these names).

Confirmed: `tests/unit/test_sync_executor_service_wiring.py:33` (a file not in T01's
target list, not touched until T02) does:

```python
from tests.unit.conftest import make_service, make_sync_executor_config, make_sync_executor_hassette
```

If T01 lands as an independently-verified unit (per this repo's own
`sequence-verifiable-units` convention — each task should leave tests passing), the
test suite will fail on collection with an `ImportError` in
`test_sync_executor_service_wiring.py` immediately after T01, before T02 runs. T01's
"purely additive" framing is not accurate for this specific change — it's additive
for production code but destructive for a test helper that a *different*, not-yet-
touched task file still depends on.

Fix: either (a) have T01 leave the three conftest helpers in place (marked
deprecated) and let T03 do the actual removal/rename after T02 has migrated the
wiring test, or (b) move the `test_sync_executor_service_wiring.py` import fix into
T01 alongside the conftest change, or (c) reorder so conftest helper removal happens
in T02 (which already touches the wiring test file) rather than T01.

### Finding 4 — `on_shutdown()`'s `hasattr(self, "executor")` guard silently breaks (MEDIUM-HIGH)

Current `SyncExecutorService.on_shutdown()`:

```python
async def on_shutdown(self) -> None:
    ...
    if not hasattr(self, "executor"):
        return
    ...
    await asyncio.to_thread(self.executor.shutdown, timeout=budget)
```

T02's prompt for `on_shutdown()` says only: "Replace `self.executor.shutdown(timeout=budget)`
with `self.sync_executor.shutdown_pool(budget)`." It does not mention the `hasattr(self, "executor")`
guard above it. After T02's changes, `SyncExecutorService` never sets a `self.executor`
attribute on itself anywhere (the executor lives on `self.sync_executor.executor`
instead). If the guard is left untouched, `hasattr(self, "executor")` is **always
False**, so `on_shutdown()` returns early unconditionally — the thread pool would
never be shut down via the normal Service lifecycle path, a silent functional
regression.

This guard exists to handle the case where `on_initialize()` never ran (so `executor`
was never set) — after the split, this case can't happen the same way, since
`self.sync_executor` is always present (set in `__init__` from `hassette.sync_executor`,
which is always built in `Hassette.__init__`) and its `.executor` pool is always
built (in `SyncExecutor.__init__`). The guard is now unnecessary and should be
deleted, not silently left in place referencing a now-nonexistent attribute.

Mitigating factor: `test_restart_rebuilds_executor` in `test_sync_executor_service_wiring.py`
does assert that the pre-shutdown executor rejects new submissions after `on_shutdown()`
runs — if that test is correctly migrated to check `svc.sync_executor.executor` (per
Finding 5), it would catch this regression at `ptest dev` time (AC#5). But the task
files don't name this dependency explicitly, so it relies on downstream test failure
rather than being flagged as a known risk during T02.

Fix: add an explicit line item to T02: "Remove the now-stale `hasattr(self, 'executor')`
guard at the top of `on_shutdown()` — it referenced the pre-split `self.executor`
attribute, which no longer exists on `SyncExecutorService`."

### Finding 5 — Pervasive `svc.executor` → `svc.sync_executor.executor` rename in the wiring test isn't called out explicitly (MEDIUM)

`tests/unit/test_sync_executor_service_wiring.py` references `svc.executor` directly
at roughly 15 call sites (lines 72, 82-84, 94-95, 101-105, 108-133, 192-193, 266-286).
T02's prompt says only "adapt" for this file's sections, without naming the specific
attribute-path change. One assertion in particular, line 72
(`assert not hasattr(svc, "executor")`), currently tests "executor doesn't exist
until `on_initialize()` runs" — after the split, `svc.executor` never exists at any
point (it's `svc.sync_executor.executor` now), so this assertion becomes vacuously
and permanently true, silently losing the invariant it used to check. T02 should
explicitly call out (a) the `svc.executor` → `svc.sync_executor.executor` rename
across this file, and (b) that line 72's assertion needs rethinking (e.g., assert
something about `svc.sync_executor` identity instead) rather than being left as a
no-longer-meaningful pass.

### Finding 6 (LOW) — Stale docstring/error-message text in `task_bucket.py` not flagged for update

`TaskBucket.run_in_thread()`'s docstring ("Delegates to :meth:`SyncExecutorService.submit`")
and its `RuntimeError` message ("no SyncExecutorService is wired... inject a
SyncExecutorService or mock via `task_bucket._sync_service`") both reference the
old class name and old attribute name. T02's prompt covers the code path (attribute
rename, `submit()` call) but doesn't mention updating this docstring/error text.
Low severity — doesn't break tests or type-checking — but leaves misleading text
pointing at names that will no longer exist. Worth a one-line addition to T02's
prompt.
