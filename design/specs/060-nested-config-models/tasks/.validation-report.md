## Warnings

- T01 Verify FR#2: "resolve correctly on a constructed HassetteConfig" is not fully binary-verifiable. "Resolve correctly" needs a concrete observable outcome — e.g., "return the expected default value without raising AttributeError". The companion AC#2 Verify in T03 is more concrete ("used consistently across all source files"), but for FR#2 in T01 a sharper criterion would help.

- T01 Verify FR#3 and T01 Verify AC#3: Minor TOML section notation inconsistency with AC#3 in the design doc. The design doc's AC#3 text says `[database]` / `[websocket]` / `[logging]` (no parent prefix), while the design doc's Architecture section and the T01 Verify criterion both use `[hassette.database]` / `[hassette.websocket]` / `[hassette.logging]` (with parent prefix). The Architecture section is authoritative for this. The AC#3 text in the design doc appears to be informal shorthand. No change needed in the task, but T01 should verify the full `[hassette.database]` form as written.

- T01 Verify FR#3: "loads correctly" is vague — lacks a concrete observable outcome such as "loads without error and the loaded field values match those set in the TOML file". The intent is clear from context but the criterion as written cannot be checked without reading the code.

- T01 Verify FR#4: "address nested fields" is vague — lacks a concrete observable outcome such as "the env var `HASSETTE__DATABASE__RETENTION_DAYS=14` causes `config.database.retention_days` to equal `14` after construction". This is substantially covered by the more concrete AC#4 Verify criterion in the same task, so the duplication may be intentional, but FR#4's own criterion adds no independent verifiability beyond FR#5/AC#4.

- T04 does not include a Verify item for AC#10 (Pyright passes). AC#10 is exclusively owned by T05, which is appropriate since T05 is the integration/verification task. This is not a gap — just noting the split is intentional by design.
