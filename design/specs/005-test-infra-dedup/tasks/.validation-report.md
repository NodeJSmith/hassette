## Coverage Gaps

### AC#7 is claimed and verified by the wrong task (blocking)

- design.md itself ties AC#7 to FR#11 explicitly: `AC#7 ... — all three moved to web_api/conftest.py (FR#11)` (design.md:113).
- FR#11 (the fixture move) is implemented by **T03**, not T02. T03's `implements` list includes `FR#11`, and T03's Verify section already contains the identical check under the FR#11 item: `grep -rn "def app\b|def client\b|def runtime_query_service\b" tests/integration/conftest.py` returns zero results.
- Yet **T02's frontmatter `implements` list also includes `AC#7`**, and T02's Verify section repeats the same grep check.
- T02's Target Files list does **not** include `tests/integration/conftest.py` or `tests/integration/web_api/conftest.py` — T02 never touches the files this AC checks. T02 has no way to make this AC true.
- **This is a real ordering hazard, not just a labeling slip.** T02's `depends_on` is `["T01"]` only — it does not depend on T03. T03's `depends_on` is `[]`. Nothing in the plan guarantees T03 runs before T02. If an orchestrator runs T02 before T03 (both are eligible to run as soon as T01 finishes, since T03 has no dependencies at all), T02's own Verify checklist for AC#7 will fail because the fixture move hasn't happened yet — even though T02 did everything it was actually supposed to do.

**Fix:** Remove `AC#7` from T02's `implements` and Verify list; add it to T03's `implements` list (T03 already has the equivalent check under FR#11, just not the `AC#7` label).

No other FR/AC has this problem — verified by cross-checking every task's Target Files against the files each of its Verify-section grep commands targets.

## Warnings

1. **T02 Prompt miscounts "module-level" `noop` reimplementations.** T02's Prompt says *"Migrate 4 module-level reimplementations: `test_scheduler_resource.py:76`, `test_scheduled_job.py:31`"* but only two files are named, and design.md's FR#12 explicitly labels these same two as `(module-level)` — i.e., there are 2 module-level reimplementations, not 4. The overall migration counts still sum to 21 correctly (2 module-level + 1 + 1 + 17 nested), and the Verify checklist correctly says "21 reimplementations," so this looks like a copy-paste/typo in the Prompt prose rather than a scope error — but it could confuse an implementer into hunting for 2 phantom extra module-level files. Recommend fixing "4" → "2" before execution.

2. **AC#10's scope ("all FRs") is claimed only by T02, which is not the last task in the dependency chain.** design.md states AC#10 explicitly as `uv run nox -s dev` passing "with zero regressions from the baseline (all FRs)". T04 (depends on T01+T02) adds a new test file (`tests/unit/tools/test_check_test_factories.py`) and a new pre-commit hook after T02 completes; T05 (depends on T01+T02+T03) adds new doc/rule files. Neither T04 nor T05 re-verifies AC#10 after their own changes land, even though T04 in particular adds new test surface that becomes part of `nox -s dev`. This isn't a hard gap (T04/T05's own Verify items are adequate for their own scope, and nothing suggests docs or the linter would break the suite), but if the intent of AC#10 is a final full-suite gate covering "all FRs," it would be safer for T05 (the last task, depending on everything) to also carry AC#10 as a final check.

3. **Overlapping edit to the same TESTING.md line across T03 and T05.** T03's Prompt (FR#9, point 6) and T05's Prompt (FR#19, point 2) both instruct updating the `_HARNESS_FIXTURES` fixture count from "8 module-scoped" to "7 module-scoped" in `tests/TESTING.md`. T05's instruction is guarded with "(if not already done in T03)" and T05 correctly `depends_on: ["T01","T02","T03"]`, so this resolves safely in practice — flagging only because it's the one place two tasks touch identical content in the same file, worth a quick sanity check during T05 execution that the edit wasn't already applied verbatim by T03.
