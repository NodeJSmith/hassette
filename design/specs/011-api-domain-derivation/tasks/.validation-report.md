## 4. Coverage Gaps / Contradictions

### Finding 1 (CRITICAL — contradiction, unaddressed regression)

**The design's domain-derivation change (FR#1, Architecture section) will break `turn_on`/`turn_off`/`toggle` for every entity domain that has no matching HA service — a strictly worse outcome than the current behavior — and this is neither acknowledged nor tested by any task.**

- Confirmed by reading `src/hassette/models/entities/base.py`: `BaseEntity.turn_off()` calls `self.api.turn_off(self.entity_id)`, `BaseEntity.toggle()` calls `self.api.toggle_service(self.entity_id)` — **domain is never passed explicitly**. The design's own Architecture section confirms this is intentional: "They do not pass `domain` — the `None` default triggers automatic derivation from `self.entity_id` in the Api layer" (design.md line 121).
- 15 entity classes have no `turn_on`/`turn_off`/`toggle` override and therefore inherit this exact code path: `alarm_control_panel.py`, `button.py`, `date.py`, `datetime.py`, `image.py`, `lawn_mower.py`, `lock.py`, `number.py`, `select.py`, `text.py`, `time.py`, `timer.py`, `todo.py`, `update.py`, `weather.py` (verified via `grep -rL "def turn_on" src/hassette/models/entities/*.py`).
- **Today**, calling e.g. `lock_entity.turn_off()` dispatches `homeassistant.turn_off` (the current `"homeassistant"` default) — a real, generic HA service that HA maps appropriately across domains.
- **After this design ships**, the same call derives `domain="lock"` from `entity_id="lock.front_door"` and dispatches `lock.turn_off` — **a service that does not exist in Home Assistant** (locks use `lock.lock`/`lock.unlock`). The same failure mode applies to `number.turn_on`, `weather.toggle`, `date.turn_off`, etc. — none of these exist as HA services.
- This is not speculative: `tests/unit/test_sync_entity_facade.py` has an existing module docstring documenting exactly this concern: *"LockEntity (serviceless domain) inherits base turn_on/turn_off/toggle which dispatch via `homeassistant.*` generic services, **not `lock.turn_on` (nonexistent)**."* (lines 12–13), plus dedicated tests `test_lock_sync_uses_homeassistant_domain` and `test_lock_async_uses_homeassistant_domain` (lines ~250–280) whose names and docstrings assert the current `homeassistant.*`-domain routing is the correct, intended behavior for serviceless-domain entities.
- These two Lock tests will **not catch the regression** because they mock `api`/`api.sync` entirely as a `MagicMock` and only assert the entity-level method forwards to the right *method name* with the right `entity_id` — they never exercise the real (unmocked) domain-derivation logic inside `Api`. T02's prompt (section A, `test_sync_entity_facade.py` lines 253/272) only instructs renaming `"toggle_service"` → `"toggle"` in these tests' parametrize data — it does not flag that the tests' own docstrings and names will become false statements about production behavior, and it adds no new test that would catch a real Lock/Number/Weather/etc. dispatch going to a nonexistent domain.
- The design's Non-Goals section says "No resolution of serviceless-domain gap (lock, button, etc. inheriting turn_on/turn_off/toggle) — tracked in #1320," which reads as "we're not fixing this pre-existing gap" — but the change as specified doesn't leave the gap alone, it actively converts deprecated-but-functional dispatch into calls that will fail against real Home Assistant. The Edge Cases section does not mention this interaction at all.

**Recommendation before this plan proceeds:** either (a) have `BaseEntity`/`BaseEntitySyncFacade`'s generic `turn_on`/`turn_off`/`toggle` continue to pass `domain="homeassistant"` explicitly (preserving current behavior for serviceless-domain entities while still fixing the boilerplate problem for `Api` callers who pass entity_id directly), or (b) explicitly scope FR#1 to the `Api`-level methods only when called directly by app authors and re-verify the entity-model cascade doesn't inherit auto-derivation for domains without matching services, or (c) explicitly accept the regression as a documented, deliberate tradeoff and add regression tests proving the new (broken) behavior, updating the misleading test docstrings and names accordingly. As written, none of T01/T02/T03 make this choice — the design and tasks simply don't see the interaction.

### Finding 2 (Minor — incomplete task instruction, will produce a failing test)

`tests/unit/test_recording_api.py` line 537 has a live assertion, not just a comment:
```python
api.assert_called_exact("turn_off", entity_id="light.x", domain="homeassistant")
```

T02's Prompt (section B) only calls out "Line 535: Comment referencing `"domain": "homeassistant"` → update to derived domain" — it does not mention line 537, the actual assertion, which must also change to `domain="light"` or the test will fail after T01's change ships. AC#9 ("all existing tests pass") would force whoever executes T02 to notice this when the test suite is run, so it is not a silent gap, but the Prompt as written under-specifies the fix and could cause an extra debugging round-trip.
