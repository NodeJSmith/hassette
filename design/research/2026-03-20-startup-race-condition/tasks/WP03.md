## Objectives & Success Criteria

- `_safe_session_id()` raises `RuntimeError` if the session does not exist (no more fallback to 0).
- `_do_persist_batch()` drop filter is converted from WARNING to ERROR level ŌĆö it should never fire after WP01+WP02 but serves as defense-in-depth.
- All existing tests pass with zero "Dropping" warnings in test output.
- No code path produces `listener_id=0`, `job_id=0`, or `session_id=0` records.

## Subtasks

1. In `src/hassette/core/command_executor.py`, change `_safe_session_id()`:
   - Remove the `try/except RuntimeError: return 0` fallback
   - Simply return `self.hassette.session_id` ŌĆö let the `RuntimeError` propagate if no session exists
   - Update the docstring to reflect the new contract: session must exist, absence is a bug
2. In `src/hassette/core/command_executor.py`, change `_do_persist_batch()`:
   - Change `self.logger.warning(...)` to `self.logger.error(...)` for both the handler invocation and job execution drop messages
   - Update the comment to say "regression guard" instead of "startup race"
   - Keep the filter logic (log + drop) ŌĆö do not raise, as this is defense-in-depth
3. Verify no remaining `or 0` sentinel patterns:
   - Grep `src/hassette/core/bus_service.py` for `or 0` ŌĆö should be gone (WP02)
   - Grep `src/hassette/core/scheduler_service.py` for `or 0` ŌĆö should be gone (WP02)
   - Grep `src/hassette/core/command_executor.py` for `return 0` ŌĆö should be gone (this WP)
4. Run the full test suite (`uv run pytest tests/ -m "not e2e" --dist loadscope`) and verify:
   - Zero "Dropping" warnings in output
   - All tests pass

## Test Strategy

No new tests ŌĆö this WP tightens existing contracts. The full test suite serves as the verification. If any test produces a `session_id=0` or `listener_id=0` record after WP01+WP02, the ERROR log from the regression guard will surface it.

Test command: `uv run pytest tests/ -m "not e2e" --dist loadscope -W ignore::pytest.PytestWarning`

## Review Guidance

- **Ordering dependency**: This WP MUST only be applied after WP01 and WP02 are complete. If `_safe_session_id()` raises before phased startup is in place, handler execution will crash.
- **Defense-in-depth**: The persist filter should log ERROR + drop, NOT raise. Raising in the persist layer would crash the write queue worker and lose all subsequent records.
- **Grep verification**: The reviewer should independently grep for `or 0`, `return 0`, and `session_id=0` sentinel patterns to confirm they're all eliminated.
